# Task 11: Deploy script (forge + TS post-step) + register CLI + README

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Final shipping artifacts that take this feature from "tests pass" to "a real developer can onboard". Four deliverables, all wired together because they share the deployed addresses that flow `Deploy.s.sol` → `broadcast/run-latest.json` → `post-deploy.ts` → `networks.ts` → `register.ts` → README.

Per iteration-4 addendum §4 T11, the on-chain deploy is now a **two-step pipeline**: a Foundry script in Solidity broadcasts the deploy, and a small TS post-step reads forge's broadcast artifact and patches the middleware's `networks.ts`. Hardhat is dropped entirely.

1. **Forge deploy script** (`contracts/script/Deploy.s.sol`) — Solidity forge script. Reads env vars (`DEPLOYER_KEY`, `PLATFORM_TREASURY_ADDRESS`, `INITIAL_FEE_BPS` defaulting to `50`; `ARC_RPC_URL` is consumed by the `--rpc-url` flag and surfaced to the script via `foundry.toml`'s `[rpc_endpoints]` `arc_testnet = "${ARC_RPC_URL}"` per iter-4 §1). USDC address comes from a script-local constant (the canonical Arc Testnet value verified by T3) or from env `USDC_ADDRESS` — the script chooses env override if present. Calls `new PaymentSplitterFactory(usdc, treasury, uint16(feeBps))` (3-arg constructor per iter-3 addendum §1; `vaultImpl` is auto-deployed inside the factory constructor — no separate deploy). Wrapped in `vm.startBroadcast(deployerKey) / vm.stopBroadcast()`. Logs the factory address via `console2.log`. Forge writes `contracts/broadcast/Deploy.s.sol/5042002/run-latest.json` with both the factory deploy and the nested `vaultImpl` creation visible in the `transactions[].additionalContracts` array.

2. **TS post-deploy step** (`contracts/scripts/post-deploy.ts`) — reads `broadcast/Deploy.s.sol/<chainId>/run-latest.json`, extracts the factory `contractAddress` (top-level deploy tx) and the `vaultImpl` address (from the same tx's `additionalContracts[]` entry — the inner CREATE deployed during the factory constructor). Patches `packages/middleware/src/networks.ts` via sed-style sentinel replacement (per systemic-fix §13 — T6 owns the sentinel comments; this task only does the substitution). No cross-package TS imports — reads as UTF-8 string, regex-replaces against `/* deploy-script:factoryAddress */` and `/* deploy-script:vaultImplAddress */`, writes back. Idempotent: re-running against the same broadcast artifact reproduces the same edit. Chain-id is configurable via `--chain-id` flag (default `5042002` for Arc Testnet); for local-anvil smoke the script accepts `--chain-id 31337` and reads the corresponding broadcast directory.

3. **Register CLI** (`scripts/register.ts`) — unchanged from iter-3 design. Viem-based tsx-executable CLI invoked by a developer to call `factory.register()` from their EOA. Reads `REGISTER_KEY` from env, wraps it in `OpaqueRelayerKey` per D13 so the key never reaches stdout/stderr through any serialization path. Accepts `--network` flag (default `arc-testnet`). Resolves `factoryAddress` from `NETWORKS`. Idempotent: if the EOA is already registered (pre-flight `factory.vaults(msg.sender)` check), the CLI prints an informational message with the existing vault address and exits with code **0**. Otherwise prints the freshly deployed vault address on success; on failure exits non-zero with a typed error that does NOT echo the input key shape. Imports `getRelayerKeySecret` from the internal middleware module path (iter-3 §12) — not from the public entry point.

4. **README onboarding section** — end-to-end developer flow: claim USDC from the faucet, run `tsx scripts/register.ts --network arc-testnet`, install `@universal-paywall/middleware`, run a Node server with `PAYWALL_RELAYER_KEY` env. Documented in the order a new developer follows it. No Hardhat commands anywhere in the README.

This task is Wave 8. It depends on Task 4 (the factory contract must compile under `forge build` so the deploy script can `new PaymentSplitterFactory(...)`) and Task 8 (the middleware's `index.ts` must export `OpaqueRelayerKey` and `NETWORKS` for the register CLI to consume). It produces the artifacts Task 16 (deploy + publish) consumes; the forge script + post-deploy TS written here are what Task 16 actually runs against Arc Testnet.

  - `// SPDX-License-Identifier: MIT` and `pragma solidity ^0.8.20;`.
  - Imports: `Script, console2` from `forge-std/Script.sol`; `PaymentSplitterFactory` from `../src/PaymentSplitterFactory.sol`; `IERC20` from `@openzeppelin/contracts/token/ERC20/IERC20.sol` (via the remappings T2 sets up).
  - `contract Deploy is Script` with a `run()` external function.
  - Read env via forge cheats:
    - `uint256 deployerKey = vm.envUint("DEPLOYER_KEY");`
    - `address treasury = vm.envAddress("PLATFORM_TREASURY_ADDRESS");`
    - `uint256 feeBpsRaw = vm.envOr("INITIAL_FEE_BPS", uint256(50));` → cast to `uint16` after a bounds check (`feeBpsRaw <= 1000` per D10).
    - `address usdc = vm.envOr("USDC_ADDRESS", DEFAULT_ARC_TESTNET_USDC);` where `DEFAULT_ARC_TESTNET_USDC` is a hardcoded constant matching T3's verified value (document the source in a NatSpec comment).
  - Assert preconditions before broadcast: `require(treasury != address(0), "treasury_zero");`, `require(usdc != address(0), "usdc_zero");`, `require(feeBpsRaw <= 1000, "feeBps_exceeds_max");`.
  - `vm.startBroadcast(deployerKey);` → `PaymentSplitterFactory factory = new PaymentSplitterFactory(IERC20(usdc), treasury, uint16(feeBpsRaw));` → `vm.stopBroadcast();`.
  - Log both addresses via `console2.log("FACTORY_ADDRESS", address(factory));` and `console2.log("VAULT_IMPL_ADDRESS", factory.vaultImpl());`. These appear in forge's stdout and the broadcast log; the TS post-step reads them from the JSON artifact, not from stdout, but the log lines are valuable for the deployer watching the run.
  - Do NOT add a `verify()` call or a separate verify step inside the script — verification is handled by `--verify` flag on the `forge script` invocation (uses `verifier_url` from foundry.toml `[etherscan]` block for arcscan).
  - The script is run with: `cd contracts && forge script script/Deploy.s.sol:Deploy --rpc-url $ARC_RPC_URL --broadcast --verify`. For local smoke, replace `$ARC_RPC_URL` with `http://127.0.0.1:8545` and drop `--verify`.
  - Compilation: the script must `forge build` clean against T2's `foundry.toml` + remappings.

- Write `contracts/scripts/post-deploy.ts`:
  - tsx-executable. Shebang `#!/usr/bin/env tsx`.
  - CLI flags: `--chain-id <id>` (default `5042002`), `--broadcast-dir <path>` (default `contracts/broadcast/Deploy.s.sol/<chain-id>`), `--networks-file <path>` (default `../packages/middleware/src/networks.ts` resolved relative to `contracts/`), `--network-alias <alias>` (default `arc-testnet` — used to scope the sentinel replacement so that a local-anvil run with chainId 31337 patches a `localhost` entry instead, if present; if `arc-testnet` is the target, only that entry is touched).
  - Read `<broadcast-dir>/run-latest.json` via `fs.readFileSync` + `JSON.parse`. Forge's schema (as of forge 0.2.x): top-level `transactions: Array<{ hash, transactionType, contractName, contractAddress, additionalContracts: Array<{ transactionType, address, initCode }> }>`. The deploy tx has `contractName === "PaymentSplitterFactory"` and `transactionType === "CREATE"` (or `"CREATE2"` if T4 uses one; assume `CREATE` since the factory is deployed normally and only the vault clones use CREATE2 via `Clones`). Extract `factoryAddress` from this entry's `contractAddress`. Extract `vaultImplAddress` from the same entry's `additionalContracts[0].address` (the `PaymentVaultImpl` deployed inside the factory's constructor surfaces as an inner CREATE).
  - Validate: both addresses must be 20-byte hex strings (`/^0x[0-9a-fA-F]{40}$/`). Fail loudly with an actionable error if either is missing or malformed (point to the broadcast JSON path).
  - Read `<networks-file>` as UTF-8 string. Apply two regex replacements anchored on T6's sentinel comments:
    - `/("0x[0-9a-fA-F]{40}") *\/\* deploy-script:factoryAddress \*\//` → `"${factoryAddress}" /* deploy-script:factoryAddress */`
    - `/("0x[0-9a-fA-F]{40}") *\/\* deploy-script:vaultImplAddress \*\//` → `"${vaultImplAddress}" /* deploy-script:vaultImplAddress */`
  - If a sentinel is missing, fail loudly with `networks.ts is missing deploy-script sentinel comments — see Task 6` (do NOT silently leave placeholder addresses).
  - Write the file back via `fs.writeFileSync`. Do not run prettier; substitution preserves formatting because only the address literal changes.
  - Idempotency: re-running with the same broadcast artifact and the same networks.ts produces the same output bytes. Re-running after an external prettier pass also still finds the sentinels (regex tolerates intra-line whitespace).
  - Print `FACTORY_ADDRESS=0x...` and `VAULT_IMPL_ADDRESS=0x...` to stdout in machine-parseable form (for CI capture in T16). Exit 0 on success, non-zero on parse or write failure.
  - The script does NOT import anything from `packages/middleware/` (cross-workspace TS resolution issues). It treats `networks.ts` as a text file.
  - Live-network safety: if both sentinel-anchored address literals in `networks.ts` are already non-zero (i.e., a prior deploy already patched them) AND `--chain-id` is a non-local id (≠ 31337), exit with code 4 and the message `networks.ts already has non-zero addresses for arc-testnet — pass --force to overwrite`. Add a `--force` flag to bypass.

**Edge cases:**
- Arc Testnet RPC down at deploy time → `forge script ... --rpc-url $ARC_RPC_URL` will fail before broadcasting. Forge surfaces the error; the deployer re-runs after verifying `ARC_RPC_URL`. No special handling needed in the script.
- `DEPLOYER_KEY` missing → `vm.envUint` reverts the script with a clear forge error. Document in the README maintainer section: `export DEPLOYER_KEY=0x...` is mandatory.
- `--verify` flag's arcscan call fails with "already verified" or "still indexing" — forge handles this gracefully (logs the failure, does not abort the broadcast). Documented in the README maintainer callout: re-run `forge verify-contract ...` if needed.
- `networks.ts` patch race: two concurrent deploy runs could clobber each other's edits. Mitigation: this script is run once per network per release; the `--force` flag in `post-deploy.ts` is the explicit operator confirmation knob.
- Developer EOA has zero USDC balance when `register.ts` runs → tx will fail at gas estimation (Arc gas paid in USDC). Surface as `register_failed: insufficient_usdc_for_gas` with a faucet link instead of a raw viem error.
- `REGISTER_KEY` contains whitespace or trailing newline from shell `cat key.txt` → trim before validation; the malformed-key regex should anchor with `^` and `$` after `.trim()`.
- Idempotent already-registered: pre-flight `factory.vaults(msg.sender)` check before sending the tx is the canonical path. Race condition between the read and the send is rare (single developer EOA) and surfaces as a normal revert in the typed error message; the test suite does not assert on it.
- README onboarding section in a repo that has no `npm` workspace set up yet (developer cloned a release) → `npm install @universal-paywall/middleware` is correct for that case; do NOT recommend `npm link` or workspace-relative paths.
- The `OpaqueRelayerKey` wrapper's `[util.inspect.custom]` only protects against `util.inspect` and downstream formatters that call it (`console.log`, `pino`, `winston`). Direct string interpolation (`` `key=${key}` ``) bypasses it — the wrapper's `toString()` redaction covers that case. The test must exercise both paths.
- Sentinel missing in `networks.ts`: `post-deploy.ts` must fail loudly with `networks.ts is missing deploy-script sentinel comments — see Task 6` rather than silently leaving placeholder addresses.
- Broadcast JSON schema drift (forge version upgrade changes field names): `post-deploy.ts` should validate the shape (presence of `transactions[]`, `contractAddress`, `additionalContracts[]`) and produce an actionable error pointing the operator to update the parser against the current forge release. Pin the forge version in T2's setup notes for stability.
- Anvil unavailable in CI → the test should detect `anvil` not on PATH and skip with a clear `[anvil not installed; install foundry to run register-cli.test]` rather than failing opaquely.
