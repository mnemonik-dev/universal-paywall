# Task 11: Deploy script + register CLI + README

## Description

Final shipping artifacts that take this feature from "tests pass" to "a real developer can onboard". Three deliverables, all live in this single task because they share the same mental model (deploy → register → consume) and share the deployed addresses that flow `01_deploy_factory.ts` → `networks.ts` → `register.ts` → README.

1. **Hardhat deploy script** (`contracts/deploy/01_deploy_factory.ts`) — deploys `PaymentSplitterFactory(usdcAddress, platformTreasuryAddress, 50)` (3 args per iteration-3 addendum §1: `IERC20 _usdc`, `address _platformTreasury`, `uint16 _initialFeeBps`). `PaymentVaultImpl` is deployed **inside the factory constructor** (`vaultImpl = address(new PaymentVaultImpl())`), NOT externally — the script does a single `factory.deploy(...)` call and then reads `vaultImplAddress = await factory.vaultImpl()` post-deploy. Prints both addresses, runs `hardhat-verify` against arcscan, and patches `packages/middleware/src/networks.ts` with the deployed `factoryAddress` + `vaultImplAddress` via **sed-style sentinel replacement** (per systemic-fix §13 — Task 6 owns the sentinel comments; this task only does the substitution). The script CANNOT `import` TypeScript across packages, so the patch reads the file as text, regex-replaces against the sentinel anchors, and writes back. The script is idempotent and dry-runnable on a local Hardhat node — the smoke verification depends on this.
2. **Register CLI** (`scripts/register.ts`) — tsx-executable CLI invoked by a developer to call `factory.register()` from their EOA. Reads `REGISTER_KEY` from env, wraps it in `OpaqueRelayerKey` per D13 so the key never reaches stdout/stderr through any serialization path (the wrapper survives `console.log`, `JSON.stringify`, `util.inspect`, error stack capture). Accepts `--network` flag (default `arc-testnet`). Resolves `factoryAddress` from `NETWORKS`. Idempotent: if the EOA is already registered, the CLI prints an informational message with the existing vault address and exits with code **0** (not an error — re-running the same command is a legitimate operation). Otherwise prints the freshly deployed vault address on success; on failure exits non-zero with a typed error that does NOT echo the input key shape.
3. **README onboarding section** — end-to-end developer flow: claim USDC from the Circle faucet (`faucet.circle.com`), run `tsx scripts/register.ts --network arc-testnet`, install `@universal-paywall/middleware`, run a Node server with `PAYWALL_RELAYER_KEY` env. Documented in the order a new developer follows it.

This task is Wave 8 (parallel with Task 9). It depends on Task 4 (the factory must compile so the deploy script can load its artifact) and Task 8 (the middleware's `index.ts` must export `OpaqueRelayerKey` and `NETWORKS` for the register CLI to consume). It produces the artifacts Task 16 (deploy + publish) consumes; the deploy script written here is what Task 16 actually runs against Arc Testnet.
