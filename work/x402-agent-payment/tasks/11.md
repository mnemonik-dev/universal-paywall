## Description

Final shipping artifacts that take this feature from "tests pass" to "a real developer can onboard". Three deliverables, all live in this single task because they share the same mental model (deploy → register → consume) and share the deployed addresses that flow `01_deploy_factory.ts` → `networks.ts` → `register.ts` → README.

1. **Hardhat deploy script** (`contracts/deploy/01_deploy_factory.ts`) — deploys `PaymentSplitterFactory(usdc, treasury, 50 bps)` and the `PaymentVaultImpl` (the OZ Clones implementation contract). Prints both addresses, runs `hardhat-verify` against arcscan, and patches `packages/middleware/src/networks.ts` with the deployed `factoryAddress` + `vaultImplAddress` (the addresses that are stubbed during Wave 4 and only become real after Wave 2 contracts deploy). The script is idempotent and dry-runnable on a local Hardhat node — the smoke verification depends on this.
2. **Register CLI** (`scripts/register.ts`) — tsx-executable CLI invoked by a developer to call `factory.register()` from their EOA. Reads `REGISTER_KEY` from env, wraps it in `OpaqueRelayerKey` per D13 so the key never reaches stdout/stderr through any serialization path (the wrapper survives `console.log`, `JSON.stringify`, `util.inspect`, error stack capture). Accepts `--network` flag (default `arc-testnet`). Resolves `factoryAddress` from `NETWORKS`. Prints the deployed vault address on success; on failure exits non-zero with a typed error that does NOT echo the input key shape.
3. **README onboarding section** — end-to-end developer flow: claim USDC from the Circle faucet (`faucet.circle.com`), run `tsx scripts/register.ts --network arc-testnet`, install `@universal-paywall/middleware`, run a Node server with `PAYWALL_RELAYER_KEY` env. Documented in the order a new developer follows it.

This task is Wave 6 because it consumes Task 4 (the factory must compile) and Task 8 (NETWORKS must be the canonical place to patch addresses; the middleware's `core/index` must export the types `register.ts` reads). It produces the artifacts Task 16 (deploy + publish) consumes; the deploy script written here is what Task 16 actually runs against Arc Testnet.

## What to do

- Write `contracts/deploy/01_deploy_factory.ts`:
  - Read deploy params from env: `USDC_ADDRESS` (defaults to `NETWORKS["arc-testnet"].usdcAddress` from the middleware module — import it directly so the source of truth stays in one place), `PLATFORM_TREASURY_ADDRESS`, `DEPLOYER_KEY` (Hardhat already wires this through `accounts` in `hardhat.config.ts`; the script asserts it is set).
  - Deploy `PaymentVaultImpl` first (no constructor args except `_disableInitializers()` runs internally per D15). Capture `vaultImpl.address`.
  - Deploy `PaymentSplitterFactory(usdcAddress, treasuryAddress, vaultImpl.address, 50)` — fee default 50 bps per D10. (Cross-check the actual constructor signature against `contracts/contracts/PaymentSplitterFactory.sol` from Task 4 — if Task 4's factory deploys the impl internally instead of taking it as an arg, drop the separate impl deploy and read `vaultImpl()` from the factory after deploy. Do not invent a signature.)
  - Print both addresses in human-readable form AND in machine-parseable form (e.g. trailing `FACTORY_ADDRESS=0x...` lines) so the README copy-paste step works and CI can capture them.
  - Run `hardhat-verify` against arcscan: `hre.run("verify:verify", { address, constructorArguments })`. Wrap in try/catch — verification can race with arcscan indexing; failure here must not abort the script (print a warning + the manual `npx hardhat verify` command for retry).
  - Patch `packages/middleware/src/networks.ts`: replace the `factoryAddress` and `vaultImplAddress` placeholders for the `arc-testnet` entry with the deployed addresses. Implementation: read the file, replace the matching lines via a deterministic AST-or-regex edit anchored to clearly-marked sentinels in `networks.ts` (e.g. `/* deploy-script:factoryAddress */`). Do NOT regenerate the entire file. After patching, run `prettier --write` on the file so the diff is minimal.
  - Idempotency: re-running the script against an already-deployed factory should detect the existing addresses in `networks.ts` and prompt (or skip via `--force` flag) rather than blindly deploying again on a live network. On `localhost` (chainId from Hardhat is 31337), always proceed without prompt.
  - Exit codes: 0 on success, non-zero on deploy failure or env-var miss.

- Write `scripts/register.ts`:
  - Shebang `#!/usr/bin/env tsx` so it is executable via `tsx scripts/register.ts ...` or directly after `chmod +x`.
  - Argument parsing: minimal hand-rolled or via a tiny dep already in the workspace (avoid adding a new dep if possible). Flags: `--network <alias>` (default `arc-testnet`), `--help`.
  - Read `REGISTER_KEY` from env. If absent OR doesn't match a 32-byte hex shape (`0x` + 64 hex chars), exit with code 2 and the message `REGISTER_KEY env var missing or malformed` — explicitly do NOT include any substring of the input in the error message.
  - Wrap the key immediately in `OpaqueRelayerKey` (the wrapper from `packages/middleware/src/relayer-key.ts`, Task 6). All subsequent code paths handle the wrapper, never the raw string. The only extraction point is when constructing the viem `WalletClient` (use the same symbol-based extract pattern as `settle.ts`).
  - Resolve network: look up `NETWORKS[args.network]`. If missing or `enabled: false`, exit with code 3 and a clear message.
  - Construct viem `PublicClient` + `WalletClient` for the chosen network using `network.rpcUrl`. Call `factory.register()` against `network.factoryAddress`. Await receipt.
  - On success: read `factory.vaults(developerEoa)` (or capture the `VaultDeployed` event from the receipt) to get the deployed vault address. Print:
    ```
    Registered. Vault: 0x...
    Tx: 0x...
    ```
    Never print the EOA's private key, the raw env var, any `Authorization` header, or the wallet client object directly (`console.log(walletClient)` must redact via the wrapper).
  - On failure (RPC error, revert, nonce issue): exit non-zero, print a typed message `register_failed: <reason>`. If the revert decodes to `AlreadyRegistered`, print a friendly "this EOA already has a vault at 0x..." instead.
  - `--help` prints usage + the four supported env vars (`REGISTER_KEY`, `RPC_URL` override, etc.) but does NOT print any current env-var values.

  - Asserts: with `REGISTER_KEY` missing entirely, CLI exits code 2 with the canonical message.

**Edge cases:**
- Arc Testnet RPC down at deploy time → `hardhat run --network arcTestnet` will fail before reaching `verify:verify`. Catch at the deploy step, print actionable error.
- Deploy script run with no `DEPLOYER_KEY` env var → Hardhat falls back to default signer (which is `0x0` on a live network and a funded test account on `localhost`). The smoke verification uses `localhost`, so this works. On Arc Testnet, the script asserts `DEPLOYER_KEY` is set and exits with a clear error if not.
- `hardhat-verify` against arcscan may fail with "already verified" or "still indexing" — both are non-fatal and must not abort the script. Print the manual retry command.
- `networks.ts` patch race: two concurrent deploy runs could clobber each other's edits. Mitigation: this script is only run once per network per release; no concurrency expected. If `--force` is needed in CI, document in code comment that the operator confirms no other deploy is in flight.
- Developer EOA has zero USDC balance when `register.ts` runs → tx will fail at gas estimation (Arc gas paid in USDC). Surface as `register_failed: insufficient_usdc_for_gas` with a faucet link instead of a raw viem error.
- `REGISTER_KEY` contains whitespace or trailing newline from shell `cat key.txt` → trim before validation; the malformed-key regex should anchor with `^` and `$` after `.trim()`.
- README onboarding section in a repo that has no `npm` workspace set up yet (developer cloned a release) → `npm install @universal-paywall/middleware` is correct for that case; do NOT recommend `npm link` or workspace-relative paths.
- The `OpaqueRelayerKey` wrapper's `[util.inspect.custom]` only protects against `util.inspect` and downstream formatters that call it (`console.log`, `pino`, `winston`). Direct string interpolation (`` `key=${key}` ``) bypasses it — the wrapper's `toString()` redaction covers that case. The test must exercise both paths.

**Implementation hints:**
- The `01_deploy_factory.ts` numeric prefix matters: Hardhat's `hardhat-deploy` plugin (if Task 2 added it) runs `deploy/*.ts` in numeric order. If the toolbox doesn't include `hardhat-deploy`, the script is just a normal `hardhat run` target — both invocations work for this task. Stick with `hardhat run` per the tech-spec smoke command.
- The address-patch logic in `01_deploy_factory.ts`: prefer regex against the sentinel comments over an AST transform — TypeScript AST modification is overkill for a two-line edit, and `prettier --write` afterwards keeps the diff stable. Pattern: `/(\/\* deploy-script:factoryAddress \*\/\s*)"0x[0-9a-fA-F]{40}"/` → `$1"${factoryAddr}"`.
- For the register CLI, `viem`'s `privateKeyToAccount(extract(opaque))` is the canonical pattern. The `extract` symbol is the same one `settle.ts` uses — import it from `relayer-key.ts` (it's not in the public type, but it IS in the source per D13).
- Stdout/stderr capture in the test: spawn `tsx scripts/register.ts ...` as a child process via `node:child_process.spawn`, collect both streams into buffers, then assert. Do not use `execSync` (it buffers and interleaves).
- Cross-platform `chmod +x` on `register.ts` is not necessary — `tsx scripts/register.ts` works without it. The shebang is only for users who `chmod +x` themselves.
- README onboarding section in Markdown: keep the example server snippet under 30 lines so it fits in a single screen. Use a fenced code block with explicit language tag (`ts` or `js`) so syntax highlighting works on npmjs.com and GitHub.
