# Task 2: Foundry setup

## Description

Initialize the `contracts/` Foundry project that will host `PaymentSplitterFactory.sol` + `PaymentVaultImpl.sol` (Wave 3, Task 4), Solidity unit/fuzz/invariant tests (Task 5), the `forge script` deploy pipeline (Task 11/16), and the off-chain TS helpers (Arc Testnet USDC spike in Task 3 and the post-deploy networks.ts patcher in Task 11/16). This task only sets up tooling, layout, and configuration ‚Äî no Solidity sources, tests, or deploy scripts are written here. After this task lands, any subsequent contract task can run `cd contracts && forge build` against an empty `contracts/src/` directory and have all required OpenZeppelin 5.x primitives available, anvil ready for local testing, and a vendored-ABI hook ready to expose the compiled JSON ABIs to the middleware (T7/T10) on every build.

Tech-spec **D9** mandates Foundry (forge + anvil + cast + slither) as the canonical contract toolchain ‚Äî Hardhat is dropped entirely from this project. Iteration-4-addendum ¬ß1 and ¬ß4 T2 set the binding paths, configs, and commands; this task implements them.

Concrete deliverables:
- Foundry toolchain installed locally via `foundryup` (forge, anvil, cast available on PATH).
- `contracts/foundry.toml` per iteration-4-addendum ¬ß1 (Solidity `0.8.20`, optimizer 200 runs, `[profile.ci]` with fuzz=1000 / invariant runs=256 depth=64, `arc_testnet` rpc endpoint stub).
- OpenZeppelin contracts installed via `forge install OpenZeppelin/openzeppelin-contracts --no-commit`, **pinned to tag `v5.0.2`**, tracked as a git submodule under `contracts/lib/openzeppelin-contracts`.
- `contracts/remappings.txt` mapping `@openzeppelin/contracts/=lib/openzeppelin-contracts/contracts/` so source files use the canonical `@openzeppelin/contracts/...` import paths (matches what T4 will write and what the tech-spec Dependencies section specifies).
- `.gitmodules` at the repo root recording the OpenZeppelin submodule.
- `contracts/slither.config.json` with `filter_paths: "test/,lib/"` and `exclude_low: false` (per iteration-4-addendum ¬ß2) so T13's `slither contracts/src/ --config-file contracts/slither.config.json` audit excludes test mocks and the OZ library from production-security severity but still surfaces low-severity findings in production sources.
- ABI export hook: a small post-build script (`contracts/scripts/export-abi.ts`, invoked via the `build` npm script of `contracts/`) that, after `forge build`, copies the JSON ABIs from `contracts/out/<Contract>.sol/<Contract>.json` to `packages/middleware/src/abi/` for viem consumption in T7 and T10. The destination directory is created if missing; existing files are overwritten so re-runs are idempotent.
- `contracts/package.json` declares a minimal TS-only dev surface (no hardhat, no @nomicfoundation/* anywhere ‚Äî they are explicitly dropped). It keeps `viem` and `tsx` in `devDependencies` because T3 (TS spike via viem) and the post-deploy/post-build TS helpers (this task and Task 11/16) need them. It exposes `build` (= `forge build && tsx scripts/export-abi.ts`), `test` (= `forge test`), `coverage`, and `lint` scripts.
- `contracts/tsconfig.json` for the TS helper scripts (ESM-only, matches the rest of the monorepo).
- Empty placeholder dirs (`contracts/src/`, `contracts/test/`, `contracts/script/`, `contracts/scripts/`) with `.gitkeep` files so the layout is tracked and `forge build` does not error on missing folders. Note plural `scripts/` is the off-chain TS scripts dir (Task 3 spike, Task 11/16 post-deploy); singular `script/` is forge's Solidity scripts dir (Task 11 `Deploy.s.sol`). Both must exist ‚Äî they are different things per iteration-4-addendum ¬ß1.
- Root `package.json` `workspaces` array MUST list `"contracts"` (Task 1 establishes the workspaces; if it is already present from T1 nothing to do here, otherwise append it).
- `contracts/.env.example` lists the canonical env-var names per `deployment.md` and iteration-2 ¬ß3: `ARC_RPC_URL`, `DEPLOYER_KEY`, `ARC_VERIFY_API_KEY`. The legacy names `ARC_TESTNET_RPC_URL` and `ARC_TESTNET_PRIVATE_KEY` are forbidden and MUST NOT appear anywhere.

Driven by tech-spec **D8** (Solidity pragma + OZ 5.x choice), **D9** (Foundry over Hardhat), tech-spec Dependencies (lines 407‚Äì409: `@openzeppelin/contracts ^5.0.2 via forge install OpenZeppelin/openzeppelin-contracts@v5.0.2`, `foundry-rs/foundry` via `foundryup`, `slither-analyzer`), architecture.md "Smart contracts" row, deployment.md canonical env-var names, iteration-2-systemic-fixes ¬ß3 (env-var canonicalization), iteration-3-addendum (constructor signature for T4/T11 consumers of this setup), iteration-4-addendum ¬ß1, ¬ß2, ¬ß3, ¬ß4 T2.

- Create the `contracts/` directory layout with empty placeholder folders, each tracked via `.gitkeep`:
  - `contracts/src/.gitkeep` (production Solidity sources ‚Äî Task 4 writes here)
  - `contracts/test/.gitkeep` (Solidity tests ‚Äî Task 5 writes here; mocks live under `contracts/test/mocks/`)
  - `contracts/script/.gitkeep` (forge Solidity scripts ‚Äî Task 11 writes `Deploy.s.sol` here)
  - `contracts/scripts/.gitkeep` (off-chain TS helpers ‚Äî Task 3 writes the USDC spike here, Task 11 writes post-deploy.ts here; this task writes `export-abi.ts` here)
  - `contracts/lib/.gitkeep` is NOT needed ‚Äî `forge install` populates it and it tracks via the git submodule.
- Write `contracts/foundry.toml` exactly as specified in iteration-4-addendum ¬ß1:
  ```toml

  src = "src"
  test = "test"
  script = "script"
  out = "out"
  libs = ["lib"]
  solc = "0.8.20"
  optimizer = true
  optimizer_runs = 200
  via_ir = false

  [profile.ci]
  fuzz = { runs = 1000 }
  invariant = { runs = 256, depth = 64 }

  [rpc_endpoints]
  arc_testnet = "${ARC_RPC_URL}"
  ```
  Prepend a comment block documenting: (a) the `foundryup` install command, (b) the canonical env-var name `ARC_RPC_URL` (not `ARC_TESTNET_RPC_URL`), (c) that mocks under `test/mocks/` are auto-compiled by the `test` path and are NOT pollution of `src/` (per iteration-4-addendum ¬ß2). Do NOT add additional profiles, sources, or remappings inline ‚Äî `remappings.txt` is the canonical location for OZ remap (forge auto-discovers it).

The `--no-commit` flag is required because this task batches the submodule add with the rest of the layout into a single commit at the end. After install, confirm:

  ```

  ```
  Trailing newline. No other remaps.
- Confirm `.gitmodules` (at the repo root, NOT inside `contracts/`) exists and tracks the OZ submodule. `forge install` writes this automatically; verify the resulting block matches:
  ```
  [submodule "contracts/lib/openzeppelin-contracts"]
      path = contracts/lib/openzeppelin-contracts
      url = https://github.com/OpenZeppelin/openzeppelin-contracts
  ```
  If `forge install` produced a slightly different layout (e.g. missing branch pin), add `branch = v5.0.2` so the submodule is unambiguously pinned. The commit SHA in the submodule pointer is what actually pins the dep ‚Äî the `branch =` line is documentation.
- Write `contracts/slither.config.json` per iteration-4-addendum ¬ß2:
  ```json
  { "filter_paths": "test/,lib/", "exclude_low": false }
  ```
  Trailing newline. T13's command `slither contracts/src/ --config-file contracts/slither.config.json` will pick this up and skip test mocks and the OZ library while still surfacing low-severity findings in production sources.
- Write `contracts/scripts/export-abi.ts` ‚Äî the ABI export hook. This is a small TS script (tsx-runnable) that:
  1. Reads `contracts/out/` recursively.
  2. For each `<Contract>.sol/<Contract>.json` produced by `forge build`, extracts the `abi` field.
  3. Writes the ABI as a typed JSON file to `packages/middleware/src/abi/<Contract>.json` (parent dir created via `fs.mkdirSync(..., { recursive: true })`).
  4. Hard-codes the list of contracts to export: `PaymentSplitterFactory`, `PaymentVaultImpl` (per Data Models in tech-spec). Logs a clear warning and continues if the source JSON does not exist yet (T2 commits with no Solidity sources yet ‚Äî `forge build` produces no output, so the script must be a no-op in that case). This makes the script safe to run from T2 onward and not blocked by the absence of source contracts.
  5. Exits 0 always (no source = no-op; missing destination dir = create; existing destination = overwrite). Errors propagate only on actual filesystem failure.
  Use `fileURLToPath(import.meta.url)` + `path.dirname(...)` for ESM-correct `__dirname` resolution (this is a TS ESM script, NOT CJS ‚Äî `contracts/package.json` is `"type": "module"`).
  Top-of-file JSDoc must document: the script runs as a post-step of `forge build`, source path (`contracts/out/<Contract>.sol/<Contract>.json`), destination path (`packages/middleware/src/abi/<Contract>.json`), and the list of contracts exported. Reference Task 7 (settle.ts) and Task 10 (forked integration test) as the consumers.

**Edge cases:**

- `forge install` requires network access on first run to fetch the OZ submodule from GitHub. CI environments must allow `https://github.com/`. Document this in the foundry.toml leading comment.
- `forge install` without a tag fetches `main` of OZ. The `@v5.0.2` qualifier is REQUIRED ‚Äî never run `forge install OpenZeppelin/openzeppelin-contracts --no-commit` without it. The tech-spec Dependencies section (line 407) explicitly pins `^5.0.2`; using a different version risks API drift in `Ownable2Step` / `Clones` / `Initializable`.
- `forge build` on an empty `src/` directory exits 0 (forge prints "Nothing to compile" or similar but does not error). The smoke check above relies on this. If forge's behavior changes in a future release and it errors on empty `src/`, add a single `Placeholder.sol` with one empty contract ‚Äî but as of forge 0.2.x this is not needed.
- `export-abi.ts` must be a no-op when `contracts/out/` is missing or empty. Use `fs.existsSync(outDir)` guard, log a one-line warning ("skip ABI export: no forge output yet"), and exit 0. T7/T10 consumers run after T4+T5+forge build, so the ABI files will exist by then; T2's `npm run build` smoke check runs against an empty src and must pass.
- `forge install` writes to `.gitmodules` at the repo root, NOT inside `contracts/`. This is git's convention. If `.gitmodules` does not exist yet, forge creates it. If it already exists with other submodules (none expected at T2 time), forge appends.
- `tsx` is needed both for this task's `export-abi.ts` AND for T3's USDC spike (`verify-usdc-eip3009.ts`) AND for T11's `post-deploy.ts`. Adding it once in `contracts/devDependencies` is the canonical place ‚Äî do NOT duplicate it in T3's task spec (T3 reads it from here).
- ESM-vs-CJS: `contracts/package.json` is `"type": "module"`. tsx supports ESM natively. The TS scripts (`export-abi.ts`, T3's spike, T11's post-deploy) MUST use ESM syntax ‚Äî top-level `await` is OK, `import.meta.url` is OK, `require()` is forbidden. This is a clean break from the legacy Hardhat-CJS world.
- `foundryup` is interactive on some terminals. For CI, the recommended bootstrap is to download the prebuilt binaries from `https://github.com/foundry-rs/foundry/releases` or use the GitHub Action `foundry-rs/foundry-toolchain@v1`. This is a CI concern (Task 12); T2 documents the local-dev install (`foundryup`) and trusts that the local developer has run it.
- The OZ submodule pinning relies on the commit SHA stored in the git index (not the `branch =` line in `.gitmodules`, which is informational). Anyone cloning the repo must run `git submodule update --init --recursive` to materialize `contracts/lib/openzeppelin-contracts/` before running `forge build`. Document this in the foundry.toml leading comment so the next contributor knows.
- `slither.config.json` `filter_paths` uses a comma-separated regex-like list; the exact slither syntax accepts comma-separated path prefixes. The form `"test/,lib/"` (with trailing slashes) is what iteration-4-addendum ¬ß2 specifies verbatim ‚Äî do not normalize away the trailing slashes.

**Implementation hints:**

- Reference Foundry book chapters: "Project Layout" (foundry.toml fields), "Forge ‚Üí Tests" (profile.ci semantics), "Forge ‚Üí Dependencies" (`forge install` + `remappings.txt`).
- The `[profile.ci]` block in `foundry.toml` is activated when `forge test` is invoked with `FOUNDRY_PROFILE=ci` env or `--profile ci` flag. T5 (tests) and T15 (pre-deploy QA) decide when to activate it; T2 just declares it. The `[profile.default]` settings apply to local dev runs.
- The `arc_testnet` rpc endpoint uses `${ARC_RPC_URL}` env-var interpolation ‚Äî forge resolves this at command invocation time. If `ARC_RPC_URL` is unset, forge errors when the endpoint is referenced; that is correct behavior (deploy commands MUST set the env var).
- `forge install OpenZeppelin/openzeppelin-contracts@v5.0.2 --no-commit` ‚Äî the `@v5.0.2` qualifier resolves to a git tag, not a branch. Verify with `cd contracts/lib/openzeppelin-contracts && git tag --points-at HEAD` ‚Äî should include `v5.0.2`.
- Also run `forge install foundry-rs/forge-std --no-commit` here (no version pin needed; forge-std is forge's own stdlib and tracks forge versions). This makes T5 tests work out of the box without an extra install step. Add the resulting submodule to `.gitmodules` (forge writes it).
- The `export-abi.ts` script is intentionally tiny ‚Äî under 50 lines of TS. It's a build-time hook, not a library. Resist the urge to add CLI args, watch mode, or AST parsing ‚Äî that would scope-creep into Task 1's tsx setup.
- Do NOT add `@openzeppelin/contracts` as an npm dep in `contracts/package.json`. The Solidity import resolution comes from `lib/openzeppelin-contracts/` via the git submodule + `remappings.txt`. The npm package would be redundant and confusing (and could drift in version from the submodule).
