# Task 15: Pre-deploy QA (requires user approval of tech-spec)

## Required Skills

Before starting, load:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](~/.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Final gate before deploy and npm publish (Task 16). This task validates that the entire x402-agent-payment feature is production-ready by running the full automated test suite and methodically walking every acceptance criterion in `user-spec.md` and `tech-spec.md`. The output is a single checklist report (`qa-report.md`) that either greenlights deploy or blocks it with a concrete list of unmet criteria.

Pre-conditions:
- **`tech-spec.md` frontmatter must be `status: approved`.** If it is `status: draft`, abort immediately and surface the block to the user — deploy cannot proceed without explicit user approval of the tech-spec.
- All prior waves (Tasks 1–14) must be marked `status: done`.

This is a verification-only task — it does NOT modify code or tests. If any AC fails, it stops, reports, and hands back to the relevant implementation task.

## What to do

1. **Approval gate:** Read `work/x402-agent-payment/tech-spec.md` frontmatter. If `status != approved`, write a one-line blocker to `qa-report.md` (`BLOCKED: tech-spec status is "<value>", awaiting user approval`) and stop. Do not run tests.

2. **Run full test suites** from repo root and collect exit codes + summaries:
   - `npm test --workspace=packages/middleware` — vitest unit tests + forked-e2e (Hardhat fork runs in-process, no env flag). Must exit 0. Capture coverage and verify ≥85% line coverage on `src/`.
   - `cd contracts && npx hardhat test` — Hardhat contract tests with mock USDC. Must exit 0.
   - `cd contracts && npx hardhat coverage` — verify ≥95% branch coverage on both `PaymentSplitterFactory` and `PaymentVaultImpl`.
   - `ARC_TESTNET_E2E=1 npm run test:e2e --workspace=packages/middleware` — live Arc Testnet e2e (requires test funds + relayer key in env). Must exit 0.
   - `npm install && npm run build --workspace=packages/middleware` — confirm ESM-only build succeeds; inspect output to confirm no CJS artifacts and bundle <30 KB minified+gzip (excluding viem).
   - `cd contracts && npx hardhat compile` — confirm clean compile under pragma `^0.8.20`, OZ 5.x resolves.
   - `gitleaks detect --no-banner` (or repo pre-commit) — confirm no secret-shaped patterns in tree.

3. **Walk every acceptance criterion** from both specs and mark pass/fail with evidence (test name, file:line, log excerpt, or on-chain tx hash). Group the report into the same sections as user-spec to make gaps obvious:
   - **user-spec → Middleware** (lines 148–176): 22 ACs covering `withPaywall`/`fastifyPaywall` exports, 402 body shape (ajv-valid against vendored x402 v1 schema), valid `X-PAYMENT` flow (EIP-712 ecrecover, `to == computeVaultAddress`, value/time bounds, NonceStore, network match, paused/vault-not-deployed gates, `transferWithAuthorization` settle, receipt wait, 200 + `X-PAYMENT-RESPONSE` header), every error code (`invalid_signature`, `insufficient_amount`, `authorization_expired`, `authorization_not_yet_valid`, `nonce_already_used`, `settlement_failed` with full reason taxonomy, `network_mismatch`, `header_too_large`, `malformed_payment_header`), `NETWORKS` dual-keyed map, chain-id mismatch throw, `OpaqueRelayerKey` redaction (non-enumerable, no leak in JSON.stringify / error stacks), npm package shape (ESM-only), price-to-amount integer math.
   - **user-spec → Contracts: PaymentSplitterFactory** (lines 178–196): 11 ACs — constructor validation, `Ownable2Step`, `Pausable`, `vaultImpl` deployed in constructor, `register()` semantics (`whenNotPaused`, no double-register, `Clones.cloneDeterministic` with salt = developer EOA, `initialize`, `vaults` mapping, `VaultDeployed` event), `computeVaultAddress` matches `predictDeterministicAddress`, `setFeeBps` (owner, cap 1000, event), `setPlatformTreasury` (owner, non-zero, event), `pause`/`unpause`, all view getters, arcscan verification.
   - **user-spec → Contracts: PaymentVaultImpl** (lines 198–205): 5 ACs — `Initializable` + storage-based `ReentrancyGuard`, `initialize` semantics, `withdraw()` math + `SafeERC20` transfers + event, withdraw-while-paused works, no setters for `developer`/`factory`, no `receive()` payable.
   - **user-spec → Деплой и тестирование** (lines 207–214): 6 ACs — `npm test --workspace=packages/middleware` ≥85% line, `npx hardhat test` ≥95% branch, `forked-e2e.test.ts` runs in CI without env flag, `arc-testnet-e2e.test.ts` gated on `ARC_TESTNET_E2E=1`, deploy script runs, README onboarding flow.
   - **tech-spec → Acceptance Criteria (technical complement)** (lines 524–537): 10 ACs — build/compile success, coverage thresholds, forked + live e2e, deploy script + arcscan, no committed secrets / gitleaks blocks, bundle size, `engines.node: ">=20"`, ESM exports.

4. **Cross-check artifacts** produced by the audit tasks: `audit-code.md` (Task 12), `audit-security.md` (Task 13), `audit-tests.md` (Task 14). Every blocker-level finding in those reports must be resolved (linked commit or fix-task). If any blocker is open, mark the corresponding AC as failed.

5. **Write `work/x402-agent-payment/qa-report.md`** with:
   - Header: feature name, commit SHA, tech-spec status, run date, runner identity.
   - Summary block: total ACs / passed / failed / blocked, decision (`READY FOR DEPLOY` / `BLOCKED`).
   - Suite results table: command, exit code, duration, coverage %, log path.
   - Per-section AC checklist (the 4 groups above) with evidence for each item.
   - Open blockers list (if any) with concrete next step per item.
   - Sign-off line — only present if all ACs pass.

6. **If any AC fails:** do NOT pass the gate. Report stays as `BLOCKED`, Task 16 must not start. Hand back to the agent owning the failing area.

## Details

**Files:**
- `work/x402-agent-payment/qa-report.md` — the deliverable. Created fresh. Structure per the "Write qa-report.md" step above. Anchors `READY FOR DEPLOY` or `BLOCKED` in the summary line so Task 16 orchestration can grep for the verdict.

**Files read for context:**
- `work/x402-agent-payment/user-spec.md` — source of truth for all functional ACs (sections: Middleware, Contracts: PaymentSplitterFactory, Contracts: PaymentVaultImpl, Деплой и тестирование).
- `work/x402-agent-payment/tech-spec.md` — source of truth for technical ACs (section: "Acceptance Criteria (technical complement)"). Must also be checked for `status: approved`.
- `packages/middleware/` — the package under test. Map each Middleware AC to the test file or source location that demonstrates it (e.g., `src/__tests__/x402.test.ts::"402 body validates against ajv schema"`).
- `contracts/` — Solidity tree under test. Map each contract AC to the Hardhat test (e.g., `contracts/test/PaymentSplitterFactory.test.ts::"register reverts on double-register"`).
- Audit reports from Tasks 12/13/14 — used to confirm no open blockers carry over.

**Dependencies:** Tasks 12 (code audit), 13 (security audit), 14 (test audit) must be `status: done` with their reports written. Tasks 1–11 must be `status: done`.

**Edge cases / things to watch:**
- `ARC_TESTNET_E2E=1` requires a funded relayer key + test USDC. If env is missing, fail loudly — do NOT mark the AC as pass-by-skip.
- Hardhat coverage thresholds: branch coverage on Solidity must be ≥95%; line coverage on TS must be ≥85%. Distinguish the two metrics; do not conflate them.
- Bundle-size check excludes `viem` (peer or external dep). Verify the bundler config matches this assumption before failing on size.
- The forked-e2e test runs in the standard `npm test` invocation (no env flag) — confirm it actually ran in the captured vitest output, not just the unit tests.
- If tech-spec was modified between approval and this run, treat as `status: draft` until re-approved.
- Audit findings of severity "info" / "low" are not blockers; only "high" / "critical" / "blocker" gate deploy.

**Implementation hints:**
- Use the `pre-deploy-qa` skill's checklist methodology — walk specs section-by-section, do not shortcut to "looks good".
- When evidence is a test, cite `path/to/test.ts::"test name"`. When evidence is a code property (e.g., ESM-only), cite `path/to/file.json` + the relevant key/value.
- Prefer mechanical evidence (test passes, on-chain reads, file inspection) over judgment calls. Where judgment is required (e.g., README clarity), note it explicitly.
- The deliverable is meant to be readable by the user as a deploy go/no-go artifact — keep it concise per AC but evidence-backed.
