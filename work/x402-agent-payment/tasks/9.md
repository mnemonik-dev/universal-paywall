# Task 9: Middleware unit tests (incl. adapter unit tests)

## Required Skills

Before starting, load:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Implement the complete unit test suite for `@universal-paywall/middleware` per tech-spec "Testing Strategy → Unit tests" (≥85% line coverage). The suite must validate every pure-module and orchestration invariant defined in Waves 5-7 (Tasks 6, 7, 8): the x402 codec and 4 KB header cap, network-id normalization (CAIP-2 ↔ alias), EIP-712 verify with full tamper matrix, USD price parsing edge cases (parseUsdPrice lives in `x402.ts` per systemic-fix §6), the synchronous NonceStore semantics and TTL/cap eviction, the 7-reason settlement failure taxonomy with replay-store retention, OpaqueRelayerKey defense-in-depth across every serialization surface, the canonical error-response bodies and reason strings (§4), the core orchestrator's full happy/sad paths, the D14 startup chainId pin, the D18 SecurityLogger event catalog with redaction, the factory-state cache with TTL/RPC-error semantics (cache lives inside `core.ts` per §5), and the framework-agnostic adapter contracts (Node http exception propagation + Fastify `preHandler` lifecycle).

This task is the safety net for the entire middleware. Every Decision that has a runtime invariant (D1, D5, D13, D14, D18) and every Risk row with a "test asserts" mitigation MUST be covered by a named test here. The 402 body builder is additionally validated against the vendored x402 v1 JSON Schema using ajv so any future drift from the wire format is caught at unit-test time.

**T9 also owns the x402 v1 schema fixture at `packages/middleware/src/__tests__/fixtures/x402-v1.schema.json`.** T10 (forked + Arc Testnet e2e, Wave 9) consumes it.

**Edge cases:**

- viem `recoverTypedDataAddress` returns a checksummed string; equality compare must be case-insensitive or both sides checksummed.
- `validBefore` boundary tests must use `vi.setSystemTime` so wall-clock skew does not destabilize the suite.
- The 100k cap test should NOT actually insert 100_001 unique payers in real time — use a stub clock and a low cap injected via constructor for the test, then assert the eviction-by-oldest-validBefore policy. Add a separate slower test that pins the canonical 100k production cap (skippable under `CI=fast` if needed).
- Pino's default serializers will turn the wrapper into something — verify both `pino.stdSerializers.err` style and direct `logger.info({relayerKey})` style.
- `structuredClone` in Node 20+ may either throw on non-cloneable getters or yield an empty object — the test must accept either outcome as long as the raw key is absent.
- Mocking viem: prefer constructing a fake `publicClient` / `walletClient` object that exposes only the methods used by `verify.ts` / `settle.ts` / `core.ts`. Avoid module-level mocks of the entire `viem` package — too brittle.
- Fastify integration test: use `fastify.inject({method, url, headers})` rather than spinning up a real socket; faster and deterministic.
- Node http adapter test: use the built-in `node:http` + an ephemeral port OR a `MockReq`/`MockRes` pair — either is fine, but pick one and reuse across the adapter file.
- Factory-state cache test must use fake timers and a MockPublicClient that can be reconfigured mid-test to return a new `paused`/`vaults` value AFTER the cached read, so the TTL boundary is observable.
- `rpc_timeout` vs `mine_timeout`: the T7 classifier distinguishes by call site. `rpc_timeout` = RPC `eth_sendRawTransaction` (or equivalent send call) timed out (transaction may or may not have been broadcast). `mine_timeout` = transaction was broadcast and we never observed a receipt within the 30 s window via `waitForTransactionReceipt`. Mocks must exercise these distinct code paths — do not collapse them into one test.

**Implementation hints:**

- For the verify tamper matrix, write a single helper `signFreshAuth(overrides)` that returns a valid signed payload, then mutate one field per test. This keeps tests under 20 lines each.
- For the SecurityLogger boundary checks, write a `RecordingLogger` helper that pushes every event into an array; assert `expect(recorder.events).toContainEqual({name: 'signature_invalid', payload: expect.objectContaining({...})})`.
- For the 7 settlement reasons, parameterize with `it.each`. Each entry: `(reasonName, mockSetup, expectedReason)`.
- For the factory-state cache, write parameterized tests with `vi.advanceTimersByTime` to step through the 5 s TTL boundary; assert call-count on the mocked RPC method.
- The schema vendoring: copy the x402 v1 JSON Schema verbatim into the fixtures dir; if upstream lacks a JSON Schema, transcribe the TypeScript definitions into JSON Schema manually and add a header comment linking the spec URL + commit SHA so future updates are traceable.
- Coverage threshold should be enforced in CI — add `--coverage.thresholds.lines=85` (or the `vitest.config.ts` equivalent) so CI fails on regressions, not just on developer machines.
- All test files import the canonical reason strings from `src/errors.ts` (or wherever T6 exports them) where practical, instead of hard-coding string literals, to keep tests in sync if a reason is renamed (though §4 codifies the current set as canonical).
