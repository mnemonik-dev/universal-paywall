# Task 7: Verify + Settle

## Description

Implement the two facilitator-side modules that form the core of the x402 payment path: off-chain signature verification (`verify.ts`) and on-chain settlement (`settle.ts`). These modules turn a parsed `PaymentPayload` into either an authoritative "ok, settled on-chain" result or a precisely-classified failure that the core orchestrator can translate into the right 402 reason for the agent.

`verify.ts` is the security gate. It runs `viem.recoverTypedDataAddress` against the EIP-712 domain pulled from the `NETWORKS` registry (so the chainId and verifyingContract are bound to the configured network — defense against EIP-712 chain-replay per D14 + Risk row "EIP-712 chain-replay attack") and then walks every check from Solution step 7c: recovered signer equals `authorization.from`, `to` equals the expected vault address, `value >= maxAmountRequired` (BigInt compare), `validBefore > now + 5_000ms` (5s safety margin against block-time skew), `validAfter <= now`, the payload network normalizes to the same canonical form as the configured network, and finally the synchronous `NonceStore.has(...) → false` followed by `insert(...)` with no `await` between them (TOCTOU-safe per D5). Any failed check produces a typed reason string the orchestrator maps to a 402 body.

`settle.ts` is the on-chain executor. It uses a viem `WalletClient` (constructed from the `OpaqueRelayerKey` via the private symbol extract — D13) to call `USDC.transferWithAuthorization(from, to, value, validAfter, validBefore, nonce, v, r, s)` directly on the configured USDC contract (no wrapper contract — D2). It awaits the receipt with a 30s timeout, and every conceivable failure path collapses into exactly one of the seven settlement reasons enumerated in D5: `rpc_timeout`, `rpc_5xx`, `gas_estimate_revert`, `mine_timeout`, `receipt_reverted`, `relayer_no_balance`, `authorization_already_used_onchain`. Of these, `relayer_no_balance` has two detection paths per the Risks table: a proactive `USDC.balanceOf(relayer) < gasEstimate * 2` check before `writeContract`, and a reactive re-classification when `writeContract` fails on insufficient gas. Both emit `securityEvent("settlement_failed")` (D18 wiring is done in core; this module only returns the reason).

Two cross-cutting concerns are wired here. First, the D14 startup chainId pin: on first use of a network within this module's lifecycle, call `publicClient.getChainId()` and assert it equals `NETWORKS[network].chainId` — mismatch throws `NetworkMismatchError` (which the orchestrator translates to `chain_id_mismatch`). Second, replay-store retention on settle failure: when `settle.ts` returns a non-ok result, the `NonceStore` entry must NOT be deleted by `verify.ts` — the agent must mint a fresh nonce to retry (Risks table row "Settlement failure mid-flight"). This is achieved structurally: `verify.ts` inserts on success and never removes; `settle.ts` never touches the store.

## What to do

1. Implement `packages/middleware/src/verify.ts` exporting `verifyEip3009Authorization(payload, opts) → { ok: true, recoveredFrom } | { ok: false, reason }`.
   - Build the EIP-712 domain from `NETWORKS[opts.network]` (name=`usdcEip712Name`, version=`usdcEip712Version`, chainId, verifyingContract=`usdcAddress`).
   - Run `viem.recoverTypedDataAddress` against the `TransferWithAuthorization` typed-data structure with the supplied `payload.payload.signature` and `payload.payload.authorization` message.
   - Run all Solution-7c checks in order: recovered == from; `to == opts.expectedVaultAddress`; `BigInt(value) >= BigInt(maxAmountRequired)`; `BigInt(validBefore) > BigInt(Math.floor(Date.now()/1000)) + 5n`; `BigInt(validAfter) <= BigInt(Math.floor(Date.now()/1000))`; normalized `payload.network === opts.network` (use `NETWORKS` helpers to compare alias and CAIP-2 forms).
   - Synchronously call `opts.nonceStore.has({ from, nonce })` — if true, return `{ ok: false, reason: 'nonce_already_used' }`. Otherwise immediately call `opts.nonceStore.insert({ from, nonce, validBefore })` (no `await` in between). Return `{ ok: true, recoveredFrom }`.
   - Map each individual check failure to a stable reason string used by `errors.ts`: `signature_invalid`, `recipient_mismatch`, `value_too_low`, `authorization_expired`, `authorization_not_yet_valid`, `network_mismatch`, `nonce_already_used`.

2. Implement `packages/middleware/src/settle.ts` exporting `settleOnChain(payload, recoveredFrom, opts) → { ok: true, transactionHash } | { ok: false, reason }`.
   - Construct (or retrieve a cached) viem `PublicClient` and `WalletClient` for `opts.network`. The `WalletClient` is built from the `OpaqueRelayerKey` extracted via the private symbol exported by `relayer-key.ts`.
   - On first use of this network in this process, call `publicClient.getChainId()` and assert it equals `NETWORKS[opts.network].chainId`. Mismatch throws `NetworkMismatchError`.
   - Proactive `relayer_no_balance` check: read `USDC.balanceOf(relayerAddress)` (use the same USDC contract address as the transfer target). If below a configurable threshold (`gasEstimate * 2` heuristic — for MVP, hardcode a small constant in micro-USDC documented in the source), return `{ ok: false, reason: 'relayer_no_balance' }` without touching the chain.
   - Call `walletClient.writeContract` with the USDC contract address and `transferWithAuthorization(from, to, value, validAfter, validBefore, nonce, v, r, s)` (split the 65-byte signature into v/r/s using viem's `parseSignature` or equivalent).
   - Call `publicClient.waitForTransactionReceipt({ hash, timeout: 30_000 })`.
   - On `receipt.status === 'success'`, return `{ ok: true, transactionHash: hash }`.
   - Wrap all failures in a single classifier function that maps each error class to exactly one of the 7 reasons:
     - viem `TimeoutError` from RPC call → `rpc_timeout`
     - viem `HttpRequestError` with `status >= 500` → `rpc_5xx`
     - viem `ContractFunctionExecutionError` raised during gas estimation (pre-broadcast) → `gas_estimate_revert`
     - `waitForTransactionReceipt` timing out → `mine_timeout`
     - receipt returned but `receipt.status === 'reverted'` → `receipt_reverted`
     - any error whose decoded revert reason includes "FiatTokenV2: authorization is used or canceled" or `0xee0e9b58` selector (or matches the EIP-3009 `authorizationState[from][nonce] == true` check) → `authorization_already_used_onchain`
     - reactive insufficient-gas / insufficient-balance error during `writeContract` → `relayer_no_balance`
   - Return `{ ok: false, reason }` for any classified failure. On settle failure the replay-store entry stays intact (this is enforced by the architectural choice that `verify.ts` already inserted and `settle.ts` does not call any nonce-store API — document this contract in a code comment).

3. Add `NetworkMismatchError` class (or reuse the one from `errors.ts` if Task 6 added it; otherwise define it locally and re-export from `errors.ts` once consolidated). The error must carry `expectedChainId` and `observedChainId` fields so the core orchestrator can emit the D18 `chain_id_mismatch` security event.

4. Add per-network client cache (module-level `Map<network, { publicClient, walletClient, chainIdPinned: boolean }>`) so chainId pin happens once per network per process. Eviction is unnecessary for MVP (network set is bounded by `NETWORKS`).

5. Export both `verifyEip3009Authorization` and `settleOnChain` from the module top so they can be unit-tested in isolation (Task 9) and orchestrated by `core.ts` (Task 8).

## Details

**Files:**
- `packages/middleware/src/verify.ts` — new file. Implements EIP-712 recovery against `NETWORKS`-derived domain and the seven Solution-7c checks. Synchronous nonce-store has+insert is the only state mutation. Exports a single function plus its result/options types.
- `packages/middleware/src/settle.ts` — new file. Implements per-network client cache, chainId pin, proactive balance check, USDC `writeContract` call, receipt await with 30s timeout, and the seven-way settlement reason classifier. Exports a single function plus its result/options types. Pulls the relayer key out of the `OpaqueRelayerKey` wrapper via the private symbol from `relayer-key.ts`.

**Dependencies:**
- Task 4 (Factory + Vault contracts) — provides `contracts/contracts/interfaces/IERC3009.sol` with the canonical `transferWithAuthorization` signature used for ABI typing.
- Task 6 (Middleware pure modules) — provides `x402.ts` (decoded `PaymentPayload` shape), `networks.ts` (`NETWORKS` registry + normalization helpers), `replay-store.ts` (`NonceStore` API), `relayer-key.ts` (`OpaqueRelayerKey` + private extraction symbol), `types.ts` (shared types), `errors.ts` (where `NetworkMismatchError` may already live — coordinate with Task 6 author).
- `viem` 2.x — `recoverTypedDataAddress`, `createPublicClient`, `createWalletClient`, `http`, `parseSignature`, `TimeoutError`, `HttpRequestError`, `ContractFunctionExecutionError`, `waitForTransactionReceipt`.

**Edge cases:**
- Clock skew between agent and facilitator: handled by the 5s safety margin on `validBefore` (and the strict `<=` on `validAfter`).
- The relayer wallet's USDC balance dropping to zero between the proactive check and `writeContract`: reactive classifier catches this and returns `relayer_no_balance`.
- viem's exception hierarchy is wide; ensure the classifier checks the most specific class first (e.g., `TimeoutError` before generic `BaseError`) so no failure falls through unclassified.
- `parseSignature` of a 65-byte signature: confirm the `v` value normalization (some libraries return 0/1, EIP-3009 expects 27/28). Test with both formats from a known signer.
- Race between two requests with the same `(from, nonce)`: NonceStore guarantees synchronous has+insert, so only the first wins; second sees `nonce_already_used` from the store before any on-chain tx is attempted.
- Network ID input may arrive as alias `arc-testnet` or CAIP-2 `eip155:5042002`; always normalize through `networks.ts` helpers before lookup or comparison.

**Implementation hints:**
- Define the `TransferWithAuthorization` EIP-712 types object once at module scope (it is identical across all USDC EIP-3009 deployments).
- The per-network client cache makes chainId pin a single-flight check; on a race during init, accept that both callers may call `getChainId` once each — the assertion is idempotent.
- For the classifier, prefer matching on viem's exported error classes (`error instanceof TimeoutError`) over string-matching error messages, except for the `authorization_already_used_onchain` decode which requires looking at the revert data because USDC's revert reason format is a known string in the FiatTokenV2 source.
- Keep the relayer-key extract path narrow: `import { OPAQUE_RELAYER_KEY_SYMBOL } from './relayer-key.js'` then `const raw = wrapped[OPAQUE_RELAYER_KEY_SYMBOL]` — never let the raw key be passed across a function boundary as a string.
- Mocking strategy for tests: vitest's `vi.mock('viem', ...)` to stub `recoverTypedDataAddress`, `createPublicClient`, `createWalletClient`. Provide controllable `getChainId`, `balanceOf`-equivalent reads, `writeContract`, and `waitForTransactionReceipt` stubs per test.
- Do not delete from `NonceStore` in `settle.ts` — even on failure, the entry remains until natural TTL expiry (per Risks table row "Settlement failure mid-flight"). Add a code comment referencing this row.
- The `gasEstimate * 2` threshold for proactive balance is fuzzy at MVP — pick a concrete micro-USDC value (e.g., `1_000_000n` = 1 USDC) and document the choice in a constant `RELAYER_MIN_BALANCE_MICRO_USDC` so it's easy to revisit after Task 3's gas measurement spike.
- D18 SecurityLogger wiring lives in `core.ts` (Task 8), not here — these modules return typed reasons and the orchestrator emits the events. Do not import the logger into `verify.ts` / `settle.ts`.
