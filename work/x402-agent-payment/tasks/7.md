# Task 7: Verify + Settle

## Required Skills

Перед выполнением задачи загрузи:

## Description

Implement the two facilitator-side modules that form the core of the x402 payment path: off-chain signature verification (`verify.ts`) and on-chain settlement (`settle.ts`). These modules turn a parsed `PaymentPayload` into either an authoritative "ok, settled on-chain" result or a precisely-classified failure that the core orchestrator can translate into the right 402 reason for the agent.

`verify.ts` is the security gate. It runs `viem.recoverTypedDataAddress` against the EIP-712 domain pulled from the `NETWORKS` registry (so the chainId and verifyingContract are bound to the configured network — defense against EIP-712 chain-replay per D14 + Risk row "EIP-712 chain-replay attack") and then walks every check from Solution step 7c: recovered signer equals `authorization.from`, `to` equals the expected vault address, `value >= maxAmountRequired` (BigInt compare), `validBefore > now + 5_000ms` (5s safety margin against block-time skew), `validAfter <= now`, the payload network normalizes to the same canonical form as the configured network, and finally the synchronous `NonceStore.has(...) → false` followed by `insert(...)` with no `await` between them (TOCTOU-safe per D5). Any failed check produces a typed reason string that maps directly to the canonical 402 reason catalog from systemic-fixes §4 (e.g. `invalid_signature`, `to_mismatch`, `insufficient_amount`).

`settle.ts` is the on-chain executor and is the **sole owner** of two cross-cutting concerns per systemic-fixes §5: (1) it constructs and caches the viem `WalletClient` because, per D13, it is the only module allowed to extract the raw key from the `OpaqueRelayerKey` (core.ts holds the wrapper opaque and passes it through unchanged); (2) it performs the D14 startup chainId pin because it issues the request's first `writeContract` call. `settle.ts` reuses the `PublicClient` lazy-created by `core.ts` (passed in via `opts.publicClient`) for receipt-await and balance reads. It calls `USDC.transferWithAuthorization(from, to, value, validAfter, validBefore, nonce, v, r, s)` directly on the configured USDC contract (no wrapper contract — D2). It awaits the receipt with a 30s timeout, and every conceivable failure path collapses into exactly one of the seven settlement reasons enumerated in D5: `rpc_timeout`, `rpc_5xx`, `gas_estimate_revert`, `mine_timeout`, `receipt_reverted`, `relayer_no_balance`, `authorization_already_used_onchain`. Of these, `relayer_no_balance` has two detection paths per the Risks table: a proactive `USDC.balanceOf(relayer) < threshold` check before `writeContract`, and a reactive re-classification when `writeContract` fails on insufficient gas. Both surface as `settlement_failed` with the reason — D18 `securityEvent` emit is wired in core.ts, this module only returns the typed reason.

Replay-store retention on settle failure: when `settle.ts` returns a non-ok result, the `NonceStore` entry inserted by `verify.ts` must NOT be deleted (Risks row "Settlement failure mid-flight" — the agent must mint a fresh nonce to retry). This is enforced structurally: `verify.ts` is the only module that touches the store; `settle.ts` never imports or calls any NonceStore API.

## What to do

1. Implement `packages/middleware/src/verify.ts` exporting `verifyEip3009Authorization(payload, opts) → { ok: true, recoveredFrom } | { ok: false, reason }`.
   - Build the EIP-712 domain from `NETWORKS[opts.network]` (name=`usdcEip712Name`, version=`usdcEip712Version`, chainId, verifyingContract=`usdcAddress`).
   - Run `viem.recoverTypedDataAddress` against the `TransferWithAuthorization` typed-data structure with the supplied `payload.payload.signature` and `payload.payload.authorization` message.
   - Run all Solution-7c checks in order: recovered == from; `to == opts.expectedVaultAddress`; `BigInt(value) >= BigInt(maxAmountRequired)`; `BigInt(validBefore) > BigInt(Math.floor(Date.now()/1000)) + 5n`; `BigInt(validAfter) <= BigInt(Math.floor(Date.now()/1000))`; normalized `payload.network === opts.network` (use `NETWORKS` helpers to compare alias and CAIP-2 forms).
   - Synchronously call `opts.nonceStore.has({ from, nonce })` — if true, return `{ ok: false, reason: 'nonce_already_used' }`. Otherwise immediately call `opts.nonceStore.insert({ from, nonce, validBefore })` (no `await` in between). Return `{ ok: true, recoveredFrom }`.
   - Map each individual check failure to the canonical reason string from systemic-fixes §4: `invalid_signature` (recovered != from OR any domain tamper — chainId, verifyingContract, name, version), `to_mismatch` (NOT `recipient_mismatch`), `insufficient_amount` (NOT `value_too_low`), `authorization_expired`, `authorization_not_yet_valid`, `network_mismatch`, `nonce_already_used`. These are the wire-format codes consumed by `errors.ts` to build the 402 body.

2. Implement `packages/middleware/src/settle.ts` exporting `settleOnChain(payload, recoveredFrom, opts) → { ok: true, transactionHash } | { ok: false, reason }`.
   - **WalletClient ownership (per systemic-fixes §5):** `settle.ts` is the sole owner of the viem `WalletClient`. Maintain a module-level `Map<network, { walletClient, chainIdPinned: boolean }>` cache. On first use of a network: extract the raw key from `opts.relayerKey` (an `OpaqueRelayerKey`) via the private symbol exported by `relayer-key.ts`, build a viem account, construct `walletClient = createWalletClient({ account, chain, transport: http(rpcUrl) })`, and cache. The raw key never escapes this module — no parameters, no return values, no error fields contain it.
   - **PublicClient consumption:** `opts.publicClient` is provided by `core.ts` (which owns the per-network `PublicClient` cache per systemic-fixes §5). Use it for `getChainId()`, `readContract` (USDC balance), and `waitForTransactionReceipt`. Do NOT create a `PublicClient` here.
   - **ChainId pin (per systemic-fixes §5, D14):** because `settle.ts` issues the first `writeContract` per request, it owns the chainId pin check. On first use of a network (`chainIdPinned === false`), call `opts.publicClient.getChainId()` and assert it equals `NETWORKS[opts.network].chainId`. Mismatch throws `NetworkMismatchError` with `expectedChainId` and `observedChainId` fields. On success, set `chainIdPinned = true` so subsequent requests on the same network skip the check.
   - **Proactive `relayer_no_balance` check:** read `USDC.balanceOf(relayerAddress)` (use the same USDC contract address as the transfer target). If below the threshold `RELAYER_MIN_BALANCE_MICRO_USDC` (hardcoded constant, MVP heuristic — document `1_000_000n` ≈ 1 USDC, revisit after Task 3 spike), return `{ ok: false, reason: 'relayer_no_balance' }` without touching the chain.
   - Call `walletClient.writeContract` with the USDC contract address and `transferWithAuthorization(from, to, value, validAfter, validBefore, nonce, v, r, s)` (split the 65-byte signature into v/r/s using viem's `parseSignature` or equivalent — normalize v to 27/28 per EIP-3009).
   - Call `opts.publicClient.waitForTransactionReceipt({ hash, timeout: 30_000 })`.
   - On `receipt.status === 'success'`, return `{ ok: true, transactionHash: hash }`.
   - **Single classifier function** mapping each error to exactly one of the 7 reasons:
     - viem `TimeoutError` (or `RpcRequestError` wrapping a fetch network timeout) raised by the RPC transport during `writeContract` or `balanceOf` → `rpc_timeout`
     - viem `HttpRequestError` with `status >= 500` → `rpc_5xx`
     - viem `ContractFunctionExecutionError` raised during gas estimation (pre-broadcast) → `gas_estimate_revert`
     - viem `WaitForTransactionReceiptTimeoutError` thrown from `waitForTransactionReceipt` after the 30s timeout → `mine_timeout` (note: this is the receipt-mining timeout class, NOT the generic transport `TimeoutError`)
     - receipt returned but `receipt.status === 'reverted'` → `receipt_reverted`
     - any error whose decoded revert reason includes "FiatTokenV2: authorization is used or canceled" or `0xee0e9b58` selector (or matches the EIP-3009 `authorizationState[from][nonce] == true` check) → `authorization_already_used_onchain`
     - reactive insufficient-gas / insufficient-balance error during `writeContract` (viem `InsufficientFundsError` or revert message matching that pattern) → `relayer_no_balance`
   - Return `{ ok: false, reason }` for any classified failure. On settle failure the replay-store entry stays intact (this is enforced by the architectural choice that `verify.ts` already inserted and `settle.ts` does not call any nonce-store API — document this contract in a code comment).

3. Add `NetworkMismatchError` class (or reuse the one from `errors.ts` if Task 6 added it; otherwise define it locally and re-export from `errors.ts` once consolidated). The error must carry `expectedChainId` and `observedChainId` fields so the core orchestrator can emit the D18 `chain_id_mismatch` security event.

4. The per-network cache for `settle.ts` is `Map<network, { walletClient, chainIdPinned: boolean }>` — `PublicClient` is NOT in this cache (it's owned by core.ts). Eviction is unnecessary for MVP (network set is bounded by `NETWORKS`).

5. Export `verifyEip3009Authorization` from `verify.ts` and `settleOnChain` from `settle.ts` so they can be unit-tested in isolation (Task 9) and orchestrated by `core.ts` (Task 8). The settle.ts `opts` type must include `publicClient: PublicClient` and `relayerKey: OpaqueRelayerKey` (passed through opaquely from core).

**Files:**
- `packages/middleware/src/verify.ts` — new file. Implements EIP-712 recovery against `NETWORKS`-derived domain and the seven Solution-7c checks. Synchronous nonce-store has+insert is the only state mutation. Exports a single function plus its result/options types. All failure reasons use the canonical wire-format codes from systemic-fixes §4.

Performs first-call chainId pin via the passed-in `PublicClient` (which is owned by core.ts). Proactive USDC balance check, USDC `writeContract` call, receipt await with 30s timeout, seven-way classifier. Exports a single function plus its result/options types. Pulls the relayer key out of the `OpaqueRelayerKey` wrapper via the private symbol from `relayer-key.ts` — this is the only call site for that extraction.

**Dependencies:**
- Task 4 (Factory + Vault contracts) — provides `contracts/contracts/interfaces/IERC3009.sol` with the canonical `transferWithAuthorization` signature used for ABI typing.
- Task 6 (Middleware pure modules) — provides `x402.ts` (decoded `PaymentPayload` shape), `networks.ts` (`NETWORKS` registry + normalization helpers), `replay-store.ts` (`NonceStore` API), `relayer-key.ts` (`OpaqueRelayerKey` + private extraction symbol), `types.ts` (shared types), `errors.ts` (canonical reason catalog, possibly `NetworkMismatchError` — coordinate with Task 6 author).
- `viem` 2.x — `recoverTypedDataAddress`, `createWalletClient`, `http`, `parseSignature`, `TimeoutError`, `HttpRequestError`, `ContractFunctionExecutionError`, `WaitForTransactionReceiptTimeoutError` (for `mine_timeout` classification — distinct from generic `TimeoutError`).

**Ownership reminders (per systemic-fixes §5):**
- `PublicClient` is owned by `core.ts` (Task 8). It is passed into `settle.ts` via `opts.publicClient`. `settle.ts` MUST NOT call `createPublicClient`.
- `WalletClient` is owned by `settle.ts` (this task). It is created and cached here, never passed to or from any other module.
- `OpaqueRelayerKey` extraction (via the private symbol) happens ONLY in `settle.ts`. `core.ts` holds the wrapper opaque and passes it through.
- `chainId` pin check is performed ONLY in `settle.ts` (it's the first `writeContract` caller per request).
- `NonceStore` is owned by `core.ts` and consumed ONLY by `verify.ts`. `settle.ts` never touches it (structural enforcement of replay-store retention on failure).

**Edge cases:**
- Clock skew between agent and facilitator: handled by the 5s safety margin on `validBefore` (and the strict `<=` on `validAfter`).
- The relayer wallet's USDC balance dropping to zero between the proactive check and `writeContract`: reactive classifier catches this and returns `relayer_no_balance`.
- viem's exception hierarchy is wide; ensure the classifier checks the most specific class first. In particular: `WaitForTransactionReceiptTimeoutError` (mine_timeout) is checked before the generic `TimeoutError` (rpc_timeout) — they are distinct viem classes and must NOT collapse into one branch.
- `parseSignature` of a 65-byte signature: confirm the `v` value normalization (some libraries return 0/1, EIP-3009 expects 27/28). Test with both formats from a known signer.
- Race between two requests with the same `(from, nonce)`: NonceStore guarantees synchronous has+insert in `verify.ts`, so only the first wins; second sees `nonce_already_used` from the store before any on-chain tx is attempted.
- Network ID input may arrive as alias `arc-testnet` or CAIP-2 `eip155:5042002`; always normalize through `networks.ts` helpers before lookup or comparison.

**Implementation hints:**
- Define the `TransferWithAuthorization` EIP-712 types object once at module scope (it is identical across all USDC EIP-3009 deployments).
- The per-network WalletClient cache makes chainId pin a single-flight check; on a race during init, accept that both callers may call `getChainId` once each — the assertion is idempotent.
- For the classifier, prefer matching on viem's exported error classes (`error instanceof WaitForTransactionReceiptTimeoutError`, `error instanceof TimeoutError`, `error instanceof HttpRequestError`) over string-matching error messages, except for the `authorization_already_used_onchain` decode which requires looking at the revert data because USDC's revert reason format is a known string in the FiatTokenV2 source.
- Keep the relayer-key extract path narrow: `import { OPAQUE_RELAYER_KEY_SYMBOL } from './relayer-key.js'` then `const raw = wrapped[OPAQUE_RELAYER_KEY_SYMBOL]` — never let the raw key be passed across a function boundary as a string. This is the ONLY place in the codebase performing the extraction (per D13 + systemic-fixes §5).
- Mocking strategy for tests: vitest's `vi.mock('viem', ...)` to stub `recoverTypedDataAddress`, `createWalletClient`. Provide controllable `getChainId`, `readContract` (for `balanceOf`), `writeContract`, and `waitForTransactionReceipt` stubs per test. Mock the `PublicClient` directly as a plain object passed via `opts.publicClient` — no need to mock `createPublicClient`.
- Do not delete from `NonceStore` in `settle.ts` — even on failure, the entry remains until natural TTL expiry (per Risks table row "Settlement failure mid-flight"). Add a code comment referencing this row.
- The threshold for proactive balance is fuzzy at MVP — use the constant `RELAYER_MIN_BALANCE_MICRO_USDC = 1_000_000n` (= 1 USDC) and document the choice in a top-of-file constant so it's easy to revisit after Task 3's gas measurement spike.
- D18 SecurityLogger wiring lives in `core.ts` (Task 8), not here — these modules return typed reasons and the orchestrator emits the events. Do not import the logger into `verify.ts` / `settle.ts`.
