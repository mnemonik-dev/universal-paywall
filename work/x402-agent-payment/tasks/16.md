## Description

Ship the feature to the world. This is the first externally consumable release of `x402-agent-payment` and has two co-equal artifacts that must land atomically: the `PaymentSplitterFactory` + `PaymentVaultImpl` on Arc Testnet (chainId `5042002`), and the `@universal-paywall/middleware@0.1.0-alpha.0` npm package on the public registry. The middleware package contains the canonical factory address inside `NETWORKS["arc-testnet"]` / `NETWORKS["eip155:5042002"]`, so the contract deploy MUST precede the publish ‚Äî there is no "publish first, patch later" path that preserves x402 routing safety (see tech-spec Risks ‚Üí "Rogue clone deployment": `payTo` is derived off-chain from the canonical factory address baked into `NETWORKS`).

Concrete artifact set landing in this task:

1. **On-chain deploy via `contracts/deploy/01_deploy_factory.ts --network arcTestnet`** (script from Task 11). Outputs: `vaultImpl` address (deployed INSIDE the factory constructor per D3 ‚Äî not a separate top-level deploy step), `factory` address, deploy tx hashes, deployer EOA, and the factory's constructor args (exactly 3: `usdc`, `platformTreasury`, `feeBps=50` as `uint16`). Deploy artifacts are committed under `contracts/deploy/artifacts/arcTestnet/` (or wherever Task 11 standardized) and the factory + vaultImpl addresses are mirrored into `packages/middleware/src/networks.ts` (the `NETWORKS` map flips `arc-testnet`/`eip155:5042002` entries `enabled: true` and fills `factoryAddress` + `vaultImplAddress` with the live values, via sentinel comments `/* deploy-script:factoryAddress */` / `/* deploy-script:vaultImplAddress */` introduced by Task 6 (creates networks.ts with sentinel comments) and substituted by Task 11 (sed substitution)).
2. **Block-explorer verification of source** on `https://testnet.arcscan.app` for both `PaymentSplitterFactory.sol` and `PaymentVaultImpl.sol` using `@nomicfoundation/hardhat-verify`. Verification is required (not optional) because: (a) audit transparency for users following the 402 body ‚Äî they SHOULD be able to read the source of the factory whose vault they're paying into; (b) `_disableInitializers()` / no-setters / no-`receive()` invariants from D15/D16/D17 are only externally verifiable if source is visible on-chain. The verify step uses the custom-chain config wired in Task 2's `hardhat.config.ts` (the `TODO(task-11)` placeholder for arcscan API/browser URLs must be resolved by Task 11; this task surfaces a deviation if not).
3. **Bump + commit + tag + publish (in the correct order)** ‚Äî see "What to do" for the exact ordering. The package version goes to `0.1.0-alpha.0`, `networks.ts` is filled with live addresses, the release commit is created, then `npm publish` runs, and the annotated git tag is created and pushed ONLY after a successful publish. This ordering matters: a failed publish (version conflict, auth, provenance error) must NOT leave a stale tag pointing at a version that doesn't exist on the registry.
4. **npm publish** of `@universal-paywall/middleware@0.1.0-alpha.0` via `npm publish --access=public --tag=alpha --provenance --workspace=@universal-paywall/middleware` (note: `--workspace` takes the **package name**, not the directory path). `--access=public` is required because the package name is namespaced (`@universal-paywall/...`) and npm scoped packages default to private. `--tag=alpha` keeps the package out of `npm install @universal-paywall/middleware` (latest) while still discoverable via `npm install @universal-paywall/middleware@alpha`. `--provenance` enables SLSA build provenance, recording the workflow + commit SHA in the npm registry so consumers can audit the supply chain ‚Äî gated behind a pre-flight check (npm ‚â• 9.5 AND `GITHUB_ACTIONS` env present); if either is missing, omit `--provenance` with a logged warning rather than aborting.

This task does NOT include: post-deploy on-chain reads or live e2e against the published artifacts (those are Task 17, post-deploy-qa). This task's job is to land the artifacts; verification that they're behaving correctly is the next task's job.

  else
    has_provenance_support=false

  fi
