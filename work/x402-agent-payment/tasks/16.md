## Description

Ship the feature to the world. This is the first externally consumable release of `x402-agent-payment` and has two co-equal artifacts that must land atomically: the `PaymentSplitterFactory` + `PaymentVaultImpl` on Arc Testnet (chainId `5042002`), and the `@universal-paywall/middleware@0.1.0-alpha.0` npm package on the public registry. The middleware package contains the canonical factory address inside `NETWORKS["arc-testnet"]` / `NETWORKS["eip155:5042002"]`, so the contract deploy MUST precede the publish — there is no "publish first, patch later" path that preserves x402 routing safety (see tech-spec Risks → "Rogue clone deployment": `payTo` is derived off-chain from the canonical factory address baked into `NETWORKS`).

Concrete artifact set landing in this task:

1. **On-chain deploy via `contracts/deploy/01_deploy_factory.ts --network arcTestnet`** (script from Task 11). Outputs: `vaultImpl` address, `factory` address, deploy tx hashes, deployer EOA, constructor args (`usdc`, `platformTreasury`, `feeBps=50`). Deploy artifacts are committed under `contracts/deploy/artifacts/arcTestnet/` (or wherever Task 11 standardized) and the factory + vaultImpl addresses are mirrored into `packages/middleware/src/networks.ts` (the `NETWORKS` map flips `arc-testnet`/`eip155:5042002` entries `enabled: true` and fills `factoryAddress` + `vaultImplAddress` with the live values).
2. **Block-explorer verification of source** on `https://testnet.arcscan.app` for both `PaymentSplitterFactory.sol` and `PaymentVaultImpl.sol` using `@nomicfoundation/hardhat-verify`. Verification is required (not optional) because: (a) audit transparency for users following the 402 body — they SHOULD be able to read the source of the factory whose vault they're paying into; (b) `_disableInitializers()` / no-setters / no-`receive()` invariants from D15/D16/D17 are only externally verifiable if source is visible on-chain. The verify step uses the custom-chain config wired in Task 2's `hardhat.config.ts` (the `TODO(task-11)` placeholder for arcscan API/browser URLs must be resolved by Task 11; this task surfaces a deviation if not).
3. **Commit of `networks.ts`** with the live addresses on a dedicated release commit (subject: `release(middleware): publish 0.1.0-alpha.0 with Arc Testnet factory <addr>`). Commit also bumps `packages/middleware/package.json` version to `0.1.0-alpha.0`. This commit is what's tagged.
4. **npm publish** of `@universal-paywall/middleware@0.1.0-alpha.0` via `npm publish --access=public --tag=alpha --provenance` from `packages/middleware/`. `--access=public` is required because the package name is namespaced (`@universal-paywall/...`) and npm scoped packages default to private. `--tag=alpha` keeps the package out of `npm install @universal-paywall/middleware` (latest) while still discoverable via `npm install @universal-paywall/middleware@alpha`. `--provenance` enables SLSA build provenance, recording the workflow + commit SHA in the npm registry so consumers can audit the supply chain.

This task does NOT include: post-deploy on-chain reads or live e2e against the published artifacts (those are Task 17, post-deploy-qa). This task's job is to land the artifacts; verification that they're behaving correctly is the next task's job.

Driven by user-spec "Что делаем" (publishable npm package), tech-spec Solution (npm package distribution), tech-spec Acceptance Criteria ("Deploy script outputs factory address; verifiable on `https://testnet.arcscan.app`"), and the canonical `NETWORKS` registry pattern from `networks.ts`.

**Edge cases:**
- Arcscan verification flaky / rate-limited: retry once with backoff; if still failing, capture the error in the deploy log and abort the task (do not publish without verification — verification is a hard AC, not soft).
- Deploy succeeds but verify fails: contracts are live but unverified. Do NOT publish in this state. Either fix verification (preferred) or abort and surface to team lead — partial-publish (deploy without verify) is a deviation.
- `npm publish` fails after deploy succeeded (auth, 2FA, name conflict, network blip): contracts are live but package is not. The release commit is still meaningful but `0.1.0-alpha.0` may be burned (npm forbids republishing the same version). If `0.1.0-alpha.0` is burned, surface deviation and bump to `0.1.0-alpha.1` for the retry rather than republishing the same version.
- Provenance attestation requires GitHub OIDC; if running locally without OIDC, `--provenance` will fail. Document the publish environment in the publish log; if provenance cannot be obtained in the current env, surface to team lead before bumping to `-alpha.1` and re-publishing.
- 2FA challenge timing out mid-publish: redo the publish, but be aware npm may have partially registered the version. Check `npm view @universal-paywall/middleware versions` before retrying.
- Deployer EOA out of USDC mid-deploy: deploy script aborts cleanly; refill and re-run (deploys are not deterministic — factory address WILL change between runs because CREATE / deploy nonce differs). If the first attempt did create the factory (verify on arcscan), use that address; do not re-deploy.
- `dist/` over 30 KB: abort. Do not relax the AC silently. Investigate accidental bundling (e.g., did `viem` get inlined?) and fix at the source.
- `networks.ts` accidentally edited with a different address than what arcscan shows: cross-check both keys (alias + CAIP-2) point at the byte-equal `factoryAddress` from the deploy log AND from arcscan before committing. Mismatch is a security incident (rogue clone risk per tech-spec Risks).

**Implementation hints:**
- Run deploy from a clean checkout on the release branch — no uncommitted local changes, no swapped env vars. The deploy log should be reproducible if anyone re-reads it.
- Capture the deploy log with `tee`: `npx hardhat run ... 2>&1 | tee work/x402-agent-payment/logs/deploy/arcTestnet-$(date -u +%Y%m%dT%H%M%SZ).log`.
- For `hardhat verify`, the constructor args MUST match the deploy exactly. If `01_deploy_factory.ts` reads `PLATFORM_TREASURY_ADDRESS` from env, the verify command receives the same address as a positional arg — sanity-check by `npx hardhat console --network arcTestnet` → `await ethers.getContractAt('PaymentSplitterFactory', '<addr>').then(c => c.platformTreasury())`.
- `npm publish --dry-run` outputs the tarball file list to stderr — pipe through `tee` and grep `(grep -E '^(npm notice )?[0-9.]+(kB|B) .*' dry-run.log)` to inspect.
- Provenance: `npm publish --provenance` requires npm CLI ≥ 9.5; verify version with `npm --version` before publishing.
- `--tag=alpha` is critical. Default tag is `latest`, which would make `npm install @universal-paywall/middleware` install this alpha — not what we want for the first cut. Test locally: `npm view @universal-paywall/middleware dist-tags` after publish should show `alpha: 0.1.0-alpha.0` and either no `latest` entry or the same pre-publish `latest`.
- Tag the release commit (`git tag -a middleware-v0.1.0-alpha.0 -m "..." HEAD`) but **do not push the tag in this task** — the tag push is part of the post-deploy verification flow (Task 17) or a separate operational step. Pushing the tag triggers GitHub release automation if configured, which we want to gate on post-deploy verification.
- Do not store deployer / relayer / publisher keys in any committed file. All come from env. Re-confirm gitleaks pre-commit will reject any accidental paste.
