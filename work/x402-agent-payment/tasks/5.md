# Task 5: Contract tests

## Required Skills

Перед выполнением задачи загрузи:

## Description

Написать полный набор Hardhat + chai тестов для `PaymentSplitterFactory` и `PaymentVaultImpl` (контракты из Task 4). Тесты должны покрывать всю бизнес-логику фабрики и vault, все ветки безопасности из decisions D3/D4/D8/D10–D17, и достигать **≥95% branch coverage** через `npx hardhat coverage` на **обоих** контрактах (`PaymentSplitterFactory.sol` и `PaymentVaultImpl.sol`).

Это последняя линия защиты перед security audit (Task 13) и forked-e2e (Task 10). Любая невыявленная ветка в `register()`, `withdraw()`, `setFeeBps()` или в guard-инвариантах vault (D15/D16/D17) — это потенциальная уязвимость в money-handling коде. Тесты пишутся **после** реализации Task 4, потому что мы валидируем уже задеплоенные инварианты (не TDD на стороне контрактов: контракты были спроектированы в tech-spec, реализованы в Task 4 — здесь мы проверяем, что реализация соответствует спеку).

Использовать `MockUsdcEip3009` из Task 4 для всех сценариев с USDC-балансами. Под каждый test-suite — изолированная Hardhat fixture (`loadFixture`) с предзагруженной фабрикой, mock USDC, treasury address, owner и набором dev-EOA. **vaultImpl деплоится внутри конструктора фабрики**, его НЕ нужно отдельно деплоить в fixture и НЕ нужно передавать как 4-й аргумент конструктора (см. systemic-fix iteration-3 §1).

**Namespace разделения с Task 4:**
- `contracts/contracts/mocks/` — namespace Task 4 (`MockUsdcEip3009.sol`). Эта таска туда НЕ пишет.
- `contracts/test/mocks/` — namespace Task 5 для test-only fixtures (`MockMaliciousDeveloper.sol`, опционально `MockMaliciousTreasury.sol`). Создаём именно здесь.

**Канонический конструктор фабрики (iteration-3 §1):**
```solidity
constructor(IERC20 _usdc, address _platformTreasury, uint16 _initialFeeBps)
```
Ровно **3 аргумента**. `vaultImpl` создаётся внутри конструктора (`vaultImpl = address(new PaymentVaultImpl())`). `initialOwner` приходит из `Ownable2Step` base — `msg.sender` deploy-транзакции. `feeBps` — `uint16`, не `uint256`.

## What to do

### PaymentSplitterFactory.test.ts

Покрыть следующие группы тестов (ровно по Testing Strategy → Contract tests):

1. **`register()` — happy path и инварианты**
   - Deploys vault at predicted address (off-chain CREATE2 compute == `factory.computeVaultAddress(dev)` on-chain).
   - После register: `vaults[developer]` равен задеплоенному адресу.
   - Vault `initialize(_developer)` вызван ровно один раз и с `_developer == msg.sender` register-call'а.
   - Второй вызов `register()` от того же EOA — revert `AlreadyRegistered`.
   - Когда фабрика `paused()` — `register()` revert с OZ 5.x `EnforcedPause()`.

2. **CREATE2 cross-component invariant**
   - Off-chain хелпер на основе viem `getContractAddress({ opcode: 'CREATE2', ... })` (точно так же как middleware будет считать `payTo`) вычисляет адрес vault.
   - Этот адрес ДОЛЖЕН byte-в-byte совпадать с тем, что вернёт `factory.computeVaultAddress(developerEoa)` on-chain.
   - Проверить минимум для 3 разных developer EOA — это инвариант, на котором держится вся off-chain маршрутизация `payTo` в middleware (D3). Если они разъедутся, middleware будет отправлять платежи на адрес, который никогда не задеплоится.
   - Контр-проверка (минимум для одного EOA): тот же адрес через `ethers.getCreate2Address(factoryAddress, salt, keccak256(initCode))` — оба пути должны дать одинаковый результат. См. Hint в Details.

3. **`register()` reentrancy invariant — structural (iteration-3 §15)**
   - Динамический reentrancy-тест **исключён**: `PaymentVaultImpl.initialize()` не делает внешних вызовов на developer-EOA, callback-path отсутствует, поэтому реентри-вектор структурно невозможен.
   - Вместо этого — **structural source-pattern check**:
     - Прочитать исходник `contracts/contracts/PaymentSplitterFactory.sol` (через `fs.readFileSync`).
     - Найти тело функции `register()`.
     - Assert: строка с `vaults[msg.sender] = ` (либо `vaults[developer] = `) появляется **ДО** строки с `IPaymentVault(vault).initialize(` (либо `vault.initialize(`). Это CEI-invariant: state write до external call.
     - Assert: между объявлением `register()` и записью в `vaults[...]` нет вызовов вида `.call(`, `.delegatecall(`, `.send(`, `.transfer(`, `.initialize(` — checks-effects-interactions соблюдён, никаких external calls до state writes.
   - Документировать в test-comment: «Reentrancy в register() структурно невозможен (нет external calls до state writes); этот тест pin'ит инвариант через source-pattern check, чтобы будущий рефакторинг не добавил callback-path в register()».
   - Инвариант ТАКЖЕ зафиксирован в NatSpec у `register()` (Task 4) и в audit checklist (Task 13).

4. **`setFeeBps(uint16 bps)`**
   - Owner-only — non-owner получает revert (OZ 5.x `OwnableUnauthorizedAccount`).
   - `bps > 1000` (1001, max uint16) — revert `InvalidFeeBps`.
   - `bps == 0`, `bps == 1000`, `bps == 50` (default), `bps == 500` — успешно, состояние обновляется.
   - Событие `FeeBpsUpdated(oldBps, newBps)` эмитится с правильными значениями.

5. **`setPlatformTreasury(address)`**
   - Owner-only.
   - `address(0)` — revert `ZeroAddress`.
   - Happy path: меняется состояние, эмитится `PlatformTreasuryUpdated(oldTreasury, newTreasury)`.

6. **`Ownable2Step`**
   - `transferOwnership(newOwner)` сам по себе не передаёт владение — нужен `acceptOwnership()` от `newOwner`.
   - До `acceptOwnership` старый owner всё ещё owner; `setFeeBps` от newOwner revert'ит.
   - После `acceptOwnership` владение перешло; старый owner больше не может вызывать owner-only функции.
   - Сценарий cancel: `transferOwnership(newOwner)` → `transferOwnership(address(0))` (или `transferOwnership` на другого) отменяет pending передачу.

7. **`pause()` / `unpause()`**
   - Owner-only — non-owner revert (`OwnableUnauthorizedAccount`).
   - После `pause`: `paused() == true`, `register()` revert'ит.
   - После `unpause`: `register()` снова работает.

8. **Constructor reverts (3-arg signature, iteration-3 §1)**
   - `_usdc == address(0)` — revert `ZeroAddress`.
   - `_treasury == address(0)` — revert `ZeroAddress`.
   - `_feeBps > 1000` — revert `InvalidFeeBps`.
   - Happy: deployed с валидными аргументами `(usdc, treasury, 50)` — `usdc()`, `platformTreasury()`, `feeBps()`, `vaultImpl()`, `owner()` все читаются корректно.
   - `vaultImpl()` возвращает non-zero адрес — фабрика задеплоила impl внутри конструктора.

9. **`VaultDeployed` event + edge cases**
   - Event arg `vault` равен предсказанному CREATE2 адресу.
   - Event arg `developer` равен `msg.sender` register-call'а.
   - `vaults[unregistered_address]` возвращает `address(0)`.
   - Two distinct dev EOAs в одном Hardhat-блоке (через `network.provider.send("evm_setAutomine", [false])` + `evm_mine` после) — получают **distinct vault addresses** (CREATE2 salt различается).

### PaymentVaultImpl.test.ts

Все тесты через **clone**, развёрнутый из factory в test fixture (а не напрямую через `vaultImpl`), кроме явных D15 тестов.

1. **`initialize` single-call**
   - Первый вызов через factory.register() — success.
   - Прямой второй вызов `vault.initialize(otherDev)` после деплоя — revert `InvalidInitialization` (OZ 5.x).

2. **D15: `_disableInitializers()` на impl**
   - Получить адрес `factory.vaultImpl()` (это сам impl, не клон).
   - Прямой вызов `IVault(vaultImpl).initialize(anyDeveloper)` — revert `InvalidInitialization`.
   - Это канонический hijack-vector для cloneable контрактов; явный отдельный тест.

3. **D16: нет `receive()` / `fallback()`**
   - Отправить native value на адрес vault через `owner.sendTransaction({to: vault, value: 1n})` — revert (low-level reject, в OZ 5.x обычно generic revert).
   - На Arc USDC и есть native gas, но контракт-тесты идут на дефолтной Hardhat-цепи; мы тестируем именно EVM-инвариант "нет payable точек входа".

4. **D17: ABI-assertion — нет setters для `developer` / `factory`**
   - Прочитать compiled ABI (`artifacts.readArtifact("PaymentVaultImpl")`).
   - Assert: нет фрагмента с `name == "setDeveloper"` и аргументом `address`.
   - Assert: нет фрагмента с `name == "setFactory"` и аргументом `address`.
   - Дополнительно: нет ни одного state-changing метода кроме `initialize` и `withdraw` (для надёжности).

5. **`withdraw` — happy path и базовые ветки (developer-first transfer order)**
   - Подготовка: fund vault через `mockUsdc.mint(vault, gross)`.
   - `withdraw()` от не-developer — revert `NotDeveloper`.
   - `withdraw()` когда `usdc.balanceOf(vault) == 0` — revert `NoBalance`.
   - Happy path: vault разбивает `gross` на `net = gross - fee` (где `fee = gross * feeBps / 10000`).
     - **Порядок transfer'ов (canonical per systemic-fix iteration-2 §7):** сначала `safeTransfer(usdc, developer, net)`, затем (если `fee > 0`) `safeTransfer(usdc, platformTreasury, fee)`.
     - Balances developer и treasury растут на правильные значения; vault balance = 0.
   - Событие `Withdrawal(developer, gross, fee)` эмитится с правильными аргументами.

6. **Fee edge cases**
   - `feeBps == 0` → `fee == 0`, происходит **только один** transfer (на developer); второго (fee→treasury) НЕ происходит — проверить через mock USDC ивент-сниффер либо через assertion `treasury.balance` не изменился.
   - `feeBps == 1000` (max) → 10% от gross идёт treasury, 90% — developer.
   - Dust: `gross == 1` micro-USDC, `feeBps == 50` → `fee == 1 * 50 / 10000 == 0` (truncation) → весь 1 unit уходит developer, treasury не получает ничего.
   - `gross == 199` micro-USDC, `feeBps == 50` → `fee == 0`, всё уходит developer (boundary для truncation).
   - `gross == 200` micro-USDC, `feeBps == 50` → `fee == 1`, dev получает `199`.

7. **Fee-snapshot semantics**
   - Сценарий: deploy vault, `feeBps == 50`. Fund vault на `1000` USDC (платёж 1). Owner вызывает `factory.setFeeBps(100)`. Fund vault на ещё `1000` USDC (платёж 2). Теперь developer вызывает `withdraw()`.
   - Ожидаемое поведение: vault использует **текущее** значение `feeBps` (100) для всего gross = 2000, т.е. `fee = 20`, `dev = 1980`. Документированная семантика: fee читается at withdraw time, не snapshot'ится per payment.
   - Тест pins текущее поведение, чтобы случайное изменение в коде (например, snapshot per-payment) сразу провалило тест.

8. **Reentrancy через `MockMaliciousDeveloper` (developer-first attack vector)**
   - **Vector (per systemic-fix iteration-2 §7):** Vault шлёт USDC `developer` ПЕРВЫМ. Если `developer` — это контракт с собственной точкой входа, дёрнутой из mock USDC, он может попытаться повторно вызвать `vault.withdraw()`. Это правильный вектор для текущего transfer-order контракта.
   - Setup: deploy `contracts/test/mocks/MockMaliciousDeveloper.sol` — контракт, чей колбэк при получении USDC.transfer (через mock-USDC, у которого добавлен test-only `_hook` дёргающий получателя) пытается re-enter `vault.withdraw()`. Реализация в test/mocks/, чтобы не загрязнять production-namespace `contracts/contracts/mocks/`.
   - Зарегистрировать vault на адрес `MockMaliciousDeveloper` (через impersonation), funded vault на `1000` USDC, вызвать `vault.withdraw()` от имени malicious developer.
   - Ожидаемое: re-entry blocked OZ `ReentrancyGuard` (revert `ReentrancyGuardReentrantCall`). Внешний вызов `withdraw()` revert'ит целиком; malicious developer НЕ получает дополнительных средств. Vault balance после revert остаётся `1000` (state не изменился).
   - Альтернативный (опциональный) вектор `MockMaliciousTreasury` — также реализуется в `contracts/test/mocks/MockMaliciousTreasury.sol` (НЕ в `contracts/contracts/mocks/`, который — namespace T4). Он тестирует тот же `ReentrancyGuard` через вторую ногу transfer; но **основной** канонический тест — через `MockMaliciousDeveloper`, потому что developer получает transfer первым (transfer order developer-then-platform).

9. **Withdraw работает при `paused()` factory**
   - Setup: register dev, fund vault, owner вызывает `factory.pause()`.
   - `vault.withdraw()` от developer — success. Developers никогда не блокируются от снятия (D12).

- **Off-chain CREATE2 via viem (iteration-3 §13):**

  EIP-1167 minimal proxy creation bytecode имеет формат:
  ```
  0x3d602d80600a3d3981f3363d3d373d3d3d363d73<impl-20-bytes>5af43d82803e903d91602b57fd5bf3
  ```
  где `<impl-20-bytes>` — это адрес impl-контракта (получаем через `factory.vaultImpl()`).

  Off-chain compute через viem:

  ```ts
  import { keccak256, concat, getContractAddress, pad } from 'viem';

  // EIP-1167 minimal proxy creation code (two hardcoded halves around the impl address)
  const PROXY_PREFIX = '0x3d602d80600a3d3981f3363d3d373d3d3d363d73' as const;
  const PROXY_SUFFIX = '0x5af43d82803e903d91602b57fd5bf3' as const;

  function eip1167InitCode(impl: `0x${string}`): `0x${string}` {
    return concat([PROXY_PREFIX, impl, PROXY_SUFFIX]);
  }

  function predictVaultAddress(
    factoryAddress: `0x${string}`,
    vaultImpl: `0x${string}`,
    developerEoa: `0x${string}`,
  ): `0x${string}` {
    const salt = pad(developerEoa, { size: 32 });
    const bytecode = eip1167InitCode(vaultImpl);
    const bytecodeHash = keccak256(bytecode);
    return getContractAddress({
      opcode: 'CREATE2',
      from: factoryAddress,
      salt,
      bytecodeHash,
    });
  }
  ```

  Этот off-chain путь должен byte-в-byte совпасть с on-chain `factory.computeVaultAddress(developerEoa)`. Если расходится — либо неправильно собран `salt` (должен быть `bytes32(uint256(uint160(developerEoa)))`, т.е. left-pad address на 32 байта), либо неправильно сформирован EIP-1167 bytecode (проверить hex-префиксы).

- **Structural reentrancy invariant test (iteration-3 §15):**

  ```ts
  import { readFileSync } from 'node:fs';
  import { resolve } from 'node:path';

  it('register() source enforces CEI: state write before external call', () => {
    const sourcePath = resolve(__dirname, '../contracts/PaymentSplitterFactory.sol');
    const source = readFileSync(sourcePath, 'utf8');

    // Strip single-line comments and normalize whitespace
    const normalized = source
      .replace(/\/\/.*$/gm, '')
      .replace(/\/\*[\s\S]*?\*\//g, '')
      .replace(/\s+/g, ' ');

    // Find register() function body
    const registerMatch = normalized.match(/function\s+register\s*\([^)]*\)[^{]*\{([^}]*)\}/);
    expect(registerMatch, 'register() function not found').to.not.be.null;
    const body = registerMatch![1];

    const stateWriteIdx = body.search(/vaults\s*\[\s*(msg\.sender|developer)\s*\]\s*=/);
    const initializeIdx = body.search(/\.initialize\s*\(/);

    expect(stateWriteIdx, 'vaults[...] assignment not found').to.be.greaterThanOrEqual(0);
    expect(initializeIdx, '.initialize() call not found').to.be.greaterThanOrEqual(0);
    expect(stateWriteIdx, 'CEI violated: state write must precede initialize() call').to.be.lessThan(initializeIdx);

    // No external calls before state write
    const preStateWrite = body.slice(0, stateWriteIdx);
    expect(preStateWrite).to.not.match(/\.call\s*\(/);
    expect(preStateWrite).to.not.match(/\.delegatecall\s*\(/);
    expect(preStateWrite).to.not.match(/\.initialize\s*\(/);
  });
  ```

  Комментарий в тесте: "Reentrancy in register() is structurally impossible — there are no external calls to developer EOA before state writes. This test pins the invariant via source-pattern check so a future refactor that introduces a callback path will fail CI."
