# Task 3: Verify Arc Testnet USDC supports EIP-3009 + measure gas (spike)

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Wave 2 spike: write a standalone TypeScript script (executed with `tsx`) that uses **viem** to connect to live Arc Testnet RPC and de-risks two unknowns before we commit to Arc as the MVP chain.

**Toolchain note (iter-4 §4):** Per the Foundry migration, `contracts/` is a Foundry project (no Hardhat). T3 explicitly stays **TS-only via viem** — independent of Foundry. It reads RPC + writes a JSON artifact. No `hre`, no `hardhat run`, no `ethers`. Viem is already added to `contracts/devDependencies` by T2 (per iter-3 §13).

1. **EIP-3009 surface check.** Confirm that the canonical Arc Testnet USDC contract (`0x3600000000000000000000000000000000000000`) exposes the methods our facilitator depends on:
   - `transferWithAuthorization(address,address,uint256,uint256,uint256,bytes32,uint8,bytes32,bytes32)` — selector **`0xe3ee160e`** (NOT `0xef55bec6`, which is the unrelated `receiveWithAuthorization` selector — checking the wrong one would give a false positive)
   - `authorizationState(address,bytes32)` — required by `verify.ts` to detect on-chain replay
   - `name()`, `version()`, `decimals()` — EIP-712 domain inputs and ERC-20 view

2. **Gas economics check.** Estimate gas for a sample `transferWithAuthorization` call against the live contract, convert the estimate to USDC cost using current `gasPrice` (Arc Testnet pays gas in USDC), and surface a warning to the operator if `gasCost > 0.0005 USDC` (5% of a 0.01 USDC payment — the per-payment economics threshold defined in the tech-spec Risks table row "Per-payment settlement creates per-event gas overhead"). This is the empirical input to that Risks-table row.

3. **Emit JSON artifact for Task 6.** Write the verified `name()`, `version()`, `decimals()`, `supportsEip3009`, gas measurement, and any observational `notes` to `contracts/scripts/arc-testnet-usdc-domain.json`. Task 6 (Wave 5, `networks.ts` author) reads this artifact to populate `usdcEip712Name` and `usdcEip712Version` in the NETWORKS registry **and** surfaces every `notes[]` entry at module load (informational, not blocking — see iter-4 §5 T3). **This JSON artifact is the only handoff out of this task** — do NOT touch `packages/middleware/src/networks.ts` directly (it does not exist yet at this point in the wave sequence; Task 6 owns its creation).

The script is run manually by the operator (and by the agent during `Verify-smoke`); it is NOT part of the automated test suite. Its outputs land in the tech-spec deviation log if the live contract diverges from assumptions, and we pivot to Base Sepolia (per Risks fallback) if `supportsEip3009 === false`.

## What to do

1. **Confirm Task 2's prerequisites are present.** Read `contracts/package.json` (created in Task 2). Confirm `viem` is in `devDependencies` (added per iter-3 §13) and that `tsx` is available (root devDependency from T1 per iter-3 §9, or pulled transitively). Confirm `ARC_RPC_URL` is documented (default `https://rpc.testnet.arc.network`, chain id `5042002`). If anything is missing, surface a deviation note rather than silently patching Task 2's output.

2. **Create `contracts/scripts/verify-usdc-eip3009.ts`.** The script must:
   - Use **viem** (`createPublicClient`, `http`, `parseAbi`, `toFunctionSelector`, `defineChain`) — NOT `hre.ethers`. The script is invoked directly with `tsx` (e.g., `tsx contracts/scripts/verify-usdc-eip3009.ts`), not through Hardhat.
   - Define a minimal `defineChain({ id: 5042002, name: 'arc-testnet', rpcUrls: { default: { http: [process.env.ARC_RPC_URL ?? 'https://rpc.testnet.arc.network'] } }, nativeCurrency: { name: 'USD Coin', symbol: 'USDC', decimals: 18 } })` and construct a `PublicClient` from it.
   - Hardcode the USDC address `0x3600000000000000000000000000000000000000` as a `const USDC_ADDRESS`.
   - Construct a minimal ABI fragment via `parseAbi([...])` covering: `name()`, `version()`, `decimals()`, `transferWithAuthorization(...)`, `authorizationState(address,bytes32)`.
   - Call `name()`, `version()`, `decimals()` via `publicClient.readContract` and record results.
   - **Selector self-check:** assert `toFunctionSelector('transferWithAuthorization(address,address,uint256,uint256,uint256,bytes32,uint8,bytes32,bytes32)')` equals `0xe3ee160e`. This guards against a typo in the ABI string.
   - **On-chain method existence:** prove that both `authorizationState(address,bytes32)` and `transferWithAuthorization(...)` are present on the live contract. Choose a reliable detection strategy for the JSON-RPC at hand — the script's contract is to set `supportsEip3009 = false` only when there is unambiguous evidence the method does not exist, and `true` when the method clearly does exist. Ambiguous outcomes must surface as a `notes` entry, not a silent assumption.
   - **Gas estimation:** attempt a gas estimate via `publicClient.estimateContractGas` for a sample `transferWithAuthorization` call with dummy arguments (random nonce, plausible `validBefore`, sample `value = 10000n` = 0.01 USDC in micro-USDC, dummy signature bytes). If the node refuses to estimate gas for a reverting call, fall back to a documented constant lower-bound and append a `notes[]` entry to make the fallback visible (downstream T6 will surface it at module load).
   - **Convert gas to USDC cost:** read `gasPrice` via `publicClient.getGasPrice()` (or `publicClient.estimateFeesPerGas()` if EIP-1559 fields are returned), compute `gasCostWei = gasEstimate * gasPrice`, and convert to micro-USDC. Be explicit in code about which decimal scale each variable lives in (Arc reports gas in 18-decimal wei-scale but `decimals()` is 6 for the ERC-20 view — the dual-decimal foot-gun flagged in tech-spec line 208). Append a `notes[]` entry like `"arc-dual-decimal: native gas is 18-decimal but ERC-20 view is 6"` when the conversion is performed, so T6 records the assumption at module load. If the math feels off (cost looks 10^12× too small or too large), STOP and surface to operator — do NOT publish a wrong number.
   - **Pretty-print the result** as JSON on stdout (matches `Verify-smoke` expectation): `{name, version, decimals, supportsEip3009, sampleGasCost: "<n> micro-USDC", gasCostExceedsThreshold: boolean, notes?: string[]}`. The stdout shape MUST match the artifact shape exactly (same keys, same `notes?` semantics). `notes` is omitted entirely when empty (do not emit `"notes": []`).
   - **Exit non-zero if `supportsEip3009 === false` or `decimals !== 6`** — these are hard blockers per Risks rows.
   - **Print a clearly-formatted warning (but exit 0)** if `gasCostExceedsThreshold === true` — that signals the per-payment economics need a post-MVP batched-settlement follow-up but does NOT block the wave.

3. **Write `contracts/scripts/arc-testnet-usdc-domain.json`** with the **same shape as the stdout JSON** (including the optional `notes?: string[]` field — omit the key entirely when there are no notes, do not write `"notes": []`). This is the sole handoff to Task 6:
   - Task 6 reads the file to populate `NETWORKS['arc-testnet'].usdcEip712Name` and `NETWORKS['arc-testnet'].usdcEip712Version` (and the canonical `eip155:5042002` mirror).
   - Task 6 **also** reads `notes[]` at module load and surfaces every entry via `console.warn` or the configured logger — purely informational, **never blocking** module initialization. This is the iter-4 §5 T3 contract that this task fulfills.
   - Do NOT modify `packages/middleware/src/networks.ts` — it does not exist at this wave and Task 6 owns its creation. The script header must call out this dataflow so Task 6 picks up the artifact and the `notes` semantics.

4. **Document spike outcome in `work/x402-agent-payment/decisions.md`** post-completion: the four chain-read values (name/version/decimals/supportsEip3009), the measured gas cost, whether it exceeds the 5% threshold, the literal `notes[]` array emitted (so reviewers can audit T6's module-load surfacing), and any dual-decimal surprises.

**JSON artifact shape — canonical (iter-4 §5 T3):**

```jsonc
// contracts/scripts/arc-testnet-usdc-domain.json
{
  "name": "USD Coin",
  "version": "2",
  "decimals": 6,
  "supportsEip3009": true,
  "sampleGasCost": "<micro-USDC>",
  "gasCostExceedsThreshold": false,
  "notes": ["arc-dual-decimal: native gas is 18-decimal but ERC-20 view is 6"]
}
```

- `notes` is `string[] | undefined`. Omit the key when empty — do NOT write `"notes": []`.
- Stdout JSON MUST use the identical field set (same `notes?` semantics).
- T6 (Wave 5) consumes the artifact and surfaces every `notes[]` entry at module load via warn-level log — informational, NOT blocking. This task's contract is to make sure every observation worth surfacing makes it into `notes[]`.

**Edge cases:**
- RPC unreachable / 5xx / timeout: script should retry up to 3 times with backoff, then exit non-zero with a clear "RPC unavailable" message (not a stack trace dump — operator-facing UX).
- `gasPrice` unavailable from `publicClient.getGasPrice()` / `estimateFeesPerGas()`: in viem, these can throw on chains with non-standard fee semantics, and some Arc dev configurations also return `0n`. Catch the throw and guard against `0n`. Do NOT silently report `0 micro-USDC` and do NOT crash on a thrown error. Surface `"gas pricing unavailable, threshold check skipped"` via the `notes[]` field and set `gasCostExceedsThreshold = false` with the explanatory note attached. T6 will surface this note at module load.
- Live USDC contract reverts `name()` / `version()` (rare for a USDC-compliant contract, but theoretically possible): treat as hard failure and exit non-zero — the EIP-712 domain construction depends on these.
- Dual-decimal interaction: Arc treats USDC as native gas. `gasPrice` may be reported in 18-decimal wei-equivalent while `decimals()` reports 6 for ERC-20 view. Be explicit in the cost conversion code about which decimal scale each variable is in, and append the canonical dual-decimal `notes[]` entry whenever the conversion is performed. If the math feels off, STOP and surface to operator — do NOT publish a wrong number.
- Selector self-check: if the computed selector doesn't equal `0xe3ee160e`, the ABI signature string in the script is wrong — fail loudly with a developer-facing error, not a chain check.
- `notes` semantics drift between stdout and artifact: if you emit a note in stdout, the same note MUST appear in the artifact (and vice versa). Compute the array once, serialize twice.

**Implementation hints:**
- Use a typed `interface UsdcEip3009Probe { name: string; version: string; decimals: number; supportsEip3009: boolean; sampleGasCost: string; gasCostExceedsThreshold: boolean; notes?: string[]; }` for the result object — don't return `any`.
- Build `notes: string[]` as a local mutable array, push entries as observations occur, then in the final serialization step compute `const out = { ...probe, ...(notes.length > 0 ? { notes } : {}) }` so the key is omitted when empty.
- The JSON artifact must be written using `fs.writeFileSync` with `JSON.stringify(out, null, 2)` so a human can diff it. Use the same `out` object for `console.log(JSON.stringify(out))` (single-line stdout for the smoke check) — guarantees stdout/artifact parity.
- Per `patterns.md`, run the workspace lint command before declaring done.
- Per `patterns.md` security gate: gitleaks must not flag the spike script. The script must not write the relayer key or any 0x-hex 64-char string to stdout / file unless it's the expected USDC address.
- The script header should include a comment block: source RPC, target USDC address, intended consumer of output (Task 6 `networks.ts` via `arc-testnet-usdc-domain.json`), explicit note that **T6 reads `notes[]` at module load and surfaces them as informational warnings (not blocking)**, and a pointer to this task file.
- Wave sequencing note: this task ships in Wave 2; `packages/middleware/src/networks.ts` is created in Task 6 (Wave 5). The JSON artifact is the only handoff — do NOT touch `networks.ts` from this task.
