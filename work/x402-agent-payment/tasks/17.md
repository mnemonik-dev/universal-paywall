# Task 17: Post-deploy verification

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:post-deploy-qa` — [skills/post-deploy-qa/SKILL.md](~/.claude/skills/post-deploy-qa/SKILL.md)

## Description

После того как Task 16 задеплоил `PaymentSplitterFactory` + `PaymentVaultImpl` на Arc Testnet и опубликовал `@universal-paywall/middleware@0.1.0-alpha.0` в npm, эта задача делает живую верификацию того, что задеплоенный артефакт действительно работает в продакшен-окружении (Arc Testnet) и что опубликованный npm-пакет корректно ставится и отдаёт x402-совместимый 402.

Это **finalize-стадия** фичи: до этой задачи acceptance criteria из user-spec и tech-spec проверены только локально / на форке / в CI; здесь — против настоящего deployed factory и настоящего npm tarball. Результат фиксируется в `work/x402-agent-payment/post-deploy-report.md` и становится основой для финального коммита фичи.

Три независимых живых проверки:

1. **On-chain reads против deployed factory** на Arc Testnet (chainId 5042002). Проверяем что параметры контракта совпадают с deploy-конфигом из Task 16 — никаких "deployed но с неправильным fee" сюрпризов.
2. **Live e2e** против настоящего factory: реальный EIP-3009 signer, реальный USDC контракт Arc Testnet, реальная settlement-транзакция через relayer. Гейтится переменной окружения `ARC_TESTNET_E2E=1`, реализована в Task 11.
3. **HTTP smoke опубликованного npm-пакета**: ставим `@universal-paywall/middleware@0.1.0-alpha.0` в чистую директорию (не из workspace, не из локального tarball), поднимаем минимальный Node http server с `withPaywall()`, делаем curl без X-PAYMENT и проверяем что body соответствует x402 v1 wire format (ajv-валидация против схемы).

Никакого кода не пишем — только верифицируем deployed артефакты. Если что-то фейлится — задача проваливается, фича не закрывается, fix идёт через возврат к Task 16 (или к соответствующей задаче в коде).

## What to do

1. Подготовить рабочую директорию для верификации (вне workspace репозитория, чтобы не словить случайный `npm link`-style эффект). Например `/tmp/upw-postdeploy-$(date +%s)` или аналог.

2. Считать deployed factory address и expected параметры:
   - `PAYMENT_SPLITTER_FACTORY_ADDRESS` — из `packages/middleware/src/networks.ts` (Task 16 запатчил его туда) либо из `contracts/deploy/` output / deploy log.
   - `PLATFORM_TREASURY_ADDRESS` expected — из `.env` / deploy-конфига Task 16.
   - Expected owner (multisig или deployer EOA) — из deploy-конфига Task 16.
   - Expected `feeBps == 50` (0.5% — зафиксировано в tech-spec / contracts).

3. Шаг 1 — On-chain reads. Написать одноразовый viem-скрипт (программный, не MCP Blockscout), который через Arc Testnet RPC читает factory:
   - `factory.feeBps()` → ожидание `50`
   - `factory.platformTreasury()` → ожидание совпадения с `PLATFORM_TREASURY_ADDRESS` из deploy-конфига
   - `factory.owner()` → ожидание совпадения с expected multisig / deployer
   - `factory.paused()` → ожидание `false`
   Каждое чтение — отдельный assert. Любое расхождение = FAIL, записываем actual vs expected в репорт.

4. Шаг 2 — Live e2e. Запустить из корня репозитория:
   ```
   ARC_TESTNET_E2E=1 npm run test:e2e
   ```
   (тест существует с Task 11). Test использует real EIP-3009 signer против deployed factory + vault на Arc Testnet via real RPC. Условия успеха: exit code 0, все asserts проходят, on-chain settlement реально произошла (можно дополнительно подтвердить через Blockscout).

5. Шаг 3 — HTTP smoke npm-пакета:
   - В чистой scratch-директории (см. шаг 1) сделать `npm init -y`.
   - `npm install @universal-paywall/middleware@0.1.0-alpha.0` (тянем из публичного npm registry, не из local).
   - Создать `server.js` который импортирует `withPaywall` из установленного пакета и поднимает Node http server на свободном порту с минимальной paywall-обёрткой (`price`, `network: "arc-testnet"`, `developerEoa` — любой валидный test EOA).
   - Запустить server в фоне.
   - `curl -sS -o body.json -w '%{http_code}' http://localhost:<port>/protected` — ожидание HTTP `402`, body — валидный x402 v1 JSON.
   - Ajv-валидация body против x402 v1 schema: `x402Version`, `accepts[]` с обязательными полями (`scheme`, `network`, `maxAmountRequired`, `resource`, `description`, `mimeType`, `payTo`, `asset`, `extra.name`, `extra.version`, `extra.assetTransferMethod`). Любое поле missing/wrong type = FAIL.
   - Остановить server, удалить scratch-директорию после фиксации результата.

6. Собрать результаты всех трёх шагов в `work/x402-agent-payment/post-deploy-report.md`:
   - Header: дата, git commit hash, factory address, npm версия middleware.
   - Раздел "On-chain reads": expected vs actual для каждой переменной.
   - Раздел "Live e2e": команда, exit code, ссылка на settlement tx hash на Arc Testnet explorer (если есть).
   - Раздел "HTTP smoke": curl-команда, статус, прикреплённый 402 body (полный JSON), ajv-результат.
   - Итоговый verdict: PASS / FAIL. Если FAIL — список расхождений.

7. Закоммитить `post-deploy-report.md` в репозиторий. Задача считается выполненной только если verdict == PASS.

## Verification Steps

### Smoke

Три проверки выполняет агент в процессе задачи. Все три должны вернуть success — это и есть acceptance этой задачи.

1. **On-chain reads (bash + viem):**
   ```bash
   node scripts/postdeploy/read-factory.mjs   # одноразовый viem-скрипт, см. What to do шаг 3
   ```
   Ожидание: stdout печатает все 4 поля с `OK` для каждого; exit code 0.

2. **Live e2e (bash):**
   ```bash
   ARC_TESTNET_E2E=1 npm run test:e2e
   ```
   Ожидание: exit code 0, все тесты PASS.

3. **HTTP smoke опубликованного пакета (bash + curl):**
   ```bash
   SCRATCH=$(mktemp -d) && cd "$SCRATCH" \
     && npm init -y \
     && npm install @universal-paywall/middleware@0.1.0-alpha.0 \
     && node server.js &        # см. What to do шаг 5
   sleep 1
   curl -sS -o body.json -w '%{http_code}\n' http://localhost:<port>/protected
   node validate-x402.mjs body.json   # ajv-валидация
   ```
   Ожидание: HTTP 402, ajv валидация проходит, body содержит x402 v1 поля.

Финальный артефакт: `work/x402-agent-payment/post-deploy-report.md` коммитится в репозиторий с verdict PASS.

## Details

**Files:**
- `work/x402-agent-payment/post-deploy-report.md` — создаём (это и есть основной deliverable задачи): заголовок, три раздела результатов, итоговый verdict.
- `packages/middleware/src/networks.ts` — только читаем (для получения deployed factory address и Arc Testnet config из Task 16).
- `contracts/deploy/` output — только читаем (для expected `platformTreasury`, `owner`).
- Одноразовые скрипты (`read-factory.mjs`, минимальный `server.js`, `validate-x402.mjs`) пишутся в scratch-директорию вне репозитория и удаляются по завершении — в репозиторий не коммитятся.

**Dependencies:**
- Task 16 (Deploy + npm publish) должен быть DONE — иначе ни factory address, ни npm-версии не существуют.
- Task 11 (test suite) — `npm run test:e2e` с `ARC_TESTNET_E2E` гейтом написан там.
- Внешние: Arc Testnet RPC доступен; публичный npm registry доступен; relayer EOA имеет USDC на Arc Testnet (для оплаты gas в live e2e — см. tech-spec, risk "Relayer wallet exhausts USDC").

**Edge cases:**
- Arc Testnet RPC временно лежит → retry через fallback RPC (`https://5042002.rpc.thirdweb.com`, см. `deployment.md`). Если оба fail — FAIL задачи с пометкой "infrastructure", не "deploy regression".
- `npm install @universal-paywall/middleware@0.1.0-alpha.0` тянет latest вместо alpha → явно указывать точную версию, не диапазон.
- Live e2e mid-test упирается в relayer USDC balance → классифицируется как FAIL по `settlement_failed.reason = "relayer_no_balance"` (см. tech-spec risk), фиксируем в репорте, fix вне этой задачи.
- 402 body отдаёт correct поля но `payTo` — нулевой адрес (factory деплой/конфиг рассинхрон) → FAIL по on-chain step 1 раньше, чем дойдём до HTTP smoke.
- Расхождение `factory.owner()` с ожиданием — самый опасный случай (компрометация ключа / wrong deploy). Останавливаемся, не закрываем фичу, эскалируем.

**Implementation hints:**
- Использовать **программный viem** (`createPublicClient`, `readContract`) — не MCP Blockscout (как явно указано в tech-spec).
- ABI factory можно вытащить из `contracts/artifacts/` (build artifact Task 16) или захардкодить минимальный fragment с 4 нужными view-функциями.
- Для ajv-валидации x402 wire format взять схему из tech-spec / шарить с тестами Task 7 (там есть Zod-схема для PaymentRequirements; для post-deploy достаточно ajv-конверсии base shape).
- Все три шага идемпотентны и read-only по отношению к state (e2e шлёт реальную транзакцию, но это контролируемый платёж с relayer-а — это by design).
- Сохранять raw stdout каждого шага в репорт — для аудита через 6 месяцев.
