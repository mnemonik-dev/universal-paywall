# Task 6: Types, NETWORKS, x402 codec, errors, relayer-key, replay-store

## Description

Implement all "pure" (no external I/O, no on-chain calls, no network) modules of `@universal-paywall/middleware` in one task. These modules are leaf nodes in the middleware dependency graph and have no inter-dependencies beyond `types.ts`, which is why they're bundled into a single Wave 4 task that runs in parallel with the contracts wave. After this task lands, Wave 5 (`verify.ts` + `settle.ts`) can compose on top of stable types, a strict x402 codec, the typed error builder, the redacted relayer-key wrapper, and the synchronous NonceStore.

Concrete modules in scope:
- `packages/middleware/src/types.ts` — the TypeScript surface from tech-spec "Data Models → Middleware types (TypeScript, ESM)": `NetworkConfig`, `OpaqueRelayerKey` (interface — implementation in `relayer-key.ts`), `PaywallConfig`, `PaymentRequirements`, `ExactEvmPayload`, `PaymentPayload`. ESM-only, `export` per item.
- `packages/middleware/src/networks.ts` — `NETWORKS` registry. Per D1, keys MUST be present BOTH as alias (`'arc-testnet'`) AND canonical CAIP-2 (`'eip155:5042002'`) pointing at the same `NetworkConfig` object (object identity, not a copy) so lookup by either form returns the same row. Per tech-spec line 22, `arc-mainnet` is a placeholder gated by `enabled: false`. `usdcEip712Name`/`usdcEip712Version` are stubbed (`"USD Coin"` / `"2"`) and the real values are patched by Task 3 after the on-chain spike; this task ships the structure, not the verified values.
- `packages/middleware/src/x402.ts` — three pure functions per tech-spec D1 + code-research §"Verified wire format":
  - `build402Body(req: PaymentRequirements, error?: string)` → `{ x402Version: 1, accepts: [req], ...(error ? {error} : {}) }`. JSON-serialization is the caller's job; this function returns the plain object.
  - `decodeXPayment(header: string)` → `PaymentPayload`. Enforces `Buffer.byteLength(header, 'utf8') <= 4096` (D1 + Solution-7a "header_too_large"); base64-decodes; `JSON.parse`s; validates shape strictly: top-level keys must be exactly `{x402Version: 1, scheme: 'exact', network: string, payload: ExactEvmPayload}`, and `payload` must contain exactly `{signature, authorization: {from, to, value, validAfter, validBefore, nonce}}` and no extra fields (per D1 "strictly `{signature, authorization}` — no custom fields"). Throws typed errors that `errors.ts` consumes to map to HTTP 400 reasons (`header_too_large`, `malformed_payment_header`).
  - `encodeXPaymentResponse({success, transaction, network, payer})` → base64(JSON(...)). Used by `core.ts` on settle success.
- `packages/middleware/src/errors.ts` — typed builder per failure reason. Two HTTP status classes per tech-spec Solution §"How it works" step 7 and the SecurityLogger event catalog (D18):
  - **HTTP 400** (request can't be parsed): `header_too_large`, `malformed_payment_header`. Body is x402-v1-shaped `{x402Version:1, accepts:[...], error}` so off-the-shelf clients see a structured response.
  - **HTTP 402** (payment required / declined): `signature_invalid`, `nonce_already_used`, `network_mismatch`, `chain_id_mismatch`, `paused`, `vault_not_deployed`, `authorization_not_yet_valid`, `authorization_expired`, `value_too_low`, `settlement_failed` (with optional sub-reason from D5 settlement taxonomy: `rpc_timeout`, `rpc_5xx`, `gas_estimate_revert`, `mine_timeout`, `receipt_reverted`, `relayer_no_balance`, `authorization_already_used_onchain`).
  - One canonical helper: `buildErrorResponse(reason, ctx?) → {status, headers, body}` where `body` matches the x402 v1 schema. The `accepts` field carries the same `PaymentRequirements` the original 402 would have, so retry clients can re-sign.
- `packages/middleware/src/relayer-key.ts` — implements `OpaqueRelayerKey` per D13. Defense-in-depth redaction surface:
  - The class stores the key in a non-enumerable instance field (or a private `WeakMap<this, string>` slot — pick whichever genuinely prevents enumeration; `#privateField` is acceptable since it's not enumerated by `Object.keys` or `JSON.stringify`).
  - `toJSON()` returns the literal string `'<redacted:relayer-key>'`.
  - `toString()` returns `'<redacted:relayer-key>'`.
  - `[Symbol.for('nodejs.util.inspect.custom')]` (NOT a top-level `import('util')` — use the global `Symbol.for` key to keep the module Node/Bun/Deno-portable) returns `'<redacted:relayer-key>'`.
  - A module-private `Symbol('OPAQUE_RELAYER_KEY_EXTRACT')` — exported only via a `getRelayerKeySecret(key: OpaqueRelayerKey)` function in this file — is the ONLY extraction path. `settle.ts` (Task 7) imports `getRelayerKeySecret`; no other module should.
  - A runtime brand: an internal `Symbol.for('@universal-paywall/middleware/OpaqueRelayerKey')` exposed via a `static is(x): x is OpaqueRelayerKey` predicate; `scrubSecrets` uses this brand check to redact instances that slipped into a payload.
  - `scrubSecrets(input: unknown): unknown` — recursive scrubber for use by D18 SecurityLogger and the error-formatting boundary. Walks objects/arrays; for any string value, replaces matches of ALL the following regex patterns with `<redacted:relayer-key>`:
    - `/0x[0-9a-fA-F]{64}/g` — 32-byte hex (raw key with `0x`).
    - `/\b[0-9a-fA-F]{64}\b/gi` — 32-byte hex without `0x` prefix, word-boundary anchored (env-var shape).
    - `/0x[0-9a-fA-F]{130}/g` — 65-byte ECDSA signature with `0x`.
    - Any value where `OpaqueRelayerKey.is(value)` is true → replaced entirely with the literal string `<redacted:relayer-key>`.
  - Handles cycles (visited `WeakSet`) so `scrubSecrets({a: x}); x.self = x` does not stack-overflow.
- `packages/middleware/src/replay-store.ts` — `NonceStore` per D5:
  - Storage: `Map<from, Map<nonce, validBefore>>`. `from` is the lower-cased `0x…` address; `nonce` is the `0x…` 32-byte hex. Normalize both to lowercase on every public-API boundary.
  - `has({from, nonce, now}): boolean` and `insert({from, nonce, validBefore})` execute as a single synchronous block when called via the convenience `checkAndInsert({from, nonce, validBefore, now}): {accepted: boolean}` — caller (`verify.ts`) uses `checkAndInsert` so there is no `await` between `has` and `insert`. The TOCTOU contract is enforced by API shape, not by convention.
  - TTL eviction is lazy: every `has` (and therefore every `checkAndInsert`) call first walks the outer `Map.get(from)` and deletes any inner entries whose `validBefore <= now` (string compare on the EIP-3009 string-seconds field is wrong — parse via `BigInt(validBefore) * 1000n` for ms or `Number(validBefore)` for seconds — pick one and document at the top of the file; consistent with verify.ts's `validBefore > now + 5_000 ms` check, use ms internally).
  - Hard cap 100k entries (summed across all inner maps). On overflow, evict the oldest-`validBefore` FIFO (cheapest correct approach: when count crosses 100k, scan once and drop the entry with the smallest `validBefore`; if that's still above cap, repeat). Document the O(n) eviction is acceptable because 100k is the absolute ceiling, not a hot path.
  - Multi-instance scope is explicitly NOT supported: add a top-of-file source comment per D5 / Risks row saying "single-process scope; multi-instance support is post-MVP — see decisions.md D5".

This task ships with comprehensive unit tests for all six modules under `packages/middleware/src/__tests__/`. Test file names match the smoke command in tech-spec line 593: `x402.test.ts`, `errors.test.ts`, `networks.test.ts`, `relayer-key.test.ts`, `replay-store.test.ts` (plus an implicit `types.test.ts` is unnecessary — types are compile-time only and exercised by the other tests).

- Create `packages/middleware/src/x402.ts`:
  - `build402Body(req, error?)`: pure object construction per D1. No JSON.stringify.
  - `decodeXPayment(header)`:
    1. `Buffer.byteLength(header, 'utf8') > 4096` → throw `MalformedPaymentHeaderError('header_too_large', {size})`.
    2. `Buffer.from(header, 'base64').toString('utf8')` → may produce garbage if not valid base64; that's fine because step 3 catches it.
    3. `JSON.parse(decoded)` → catch and re-throw as `MalformedPaymentHeaderError('malformed_payment_header', {phase: 'json'})`.
    4. Shape validation: assert top-level keys are exactly `['x402Version','scheme','network','payload']` (or a permitted subset that includes all four). Reject extra keys. Same strict assertion on `payload` (`['signature','authorization']`) and on `payload.authorization` (`['from','to','value','validAfter','validBefore','nonce']`). On any shape failure → `MalformedPaymentHeaderError('malformed_payment_header', {phase: 'shape'})`.
    5. Field-type validation: `x402Version === 1`, `scheme === 'exact'`, `network` is string, `signature` matches `/^0x[0-9a-fA-F]{130}$/`, each address matches `/^0x[0-9a-fA-F]{40}$/`, `value`/`validAfter`/`validBefore` are decimal-string non-negative integers (`/^\d+$/`), `nonce` matches `/^0x[0-9a-fA-F]{64}$/`. Failure → `MalformedPaymentHeaderError('malformed_payment_header', {phase: 'shape'})`.
    6. Return the parsed `PaymentPayload`.
  - `encodeXPaymentResponse({success, transaction, network, payer})`: `Buffer.from(JSON.stringify({…})).toString('base64')`.
- Create `packages/middleware/src/errors.ts`:
  - Export an `ErrorReason` union (the 400 set + the 402 set above).
  - Export a `MalformedPaymentHeaderError` class extending `Error` with `reason: 'header_too_large' | 'malformed_payment_header'` and optional `detail` (matches D18 SecurityLogger payloads for `header_too_large` and `malformed_header`).
  - Export `buildErrorResponse(reason, ctx?: {accepts?: PaymentRequirements[], settlementReason?: string, detail?: unknown}): {status: 400 | 402, headers: Record<string,string>, body: object}`. The body is x402-v1-shaped: `{x402Version: 1, accepts: ctx?.accepts ?? [], error: reason, ...(ctx?.settlementReason ? {settlementReason: ctx.settlementReason} : {})}`. Headers: `{'content-type': 'application/json; charset=utf-8'}`.
  - Map: 400 set → `status: 400`; everything else → `status: 402`.
- Create `packages/middleware/src/relayer-key.ts`:
  - `class OpaqueRelayerKey implements types.OpaqueRelayerKey { #key: string; constructor(key: string){…} }`. Validate constructor input is a non-empty string starting with `0x` (warn but don't throw on missing `0x` — env var habits). Reject empty string with a TypeError.
  - Override `toJSON`, `toString`, `[Symbol.for('nodejs.util.inspect.custom')]` to return the redacted literal.
  - Add a static `is(x): x is OpaqueRelayerKey` using an internal `Symbol.for('@universal-paywall/middleware/OpaqueRelayerKey')` brand stamped on the prototype.
  - Export a function `getRelayerKeySecret(key: OpaqueRelayerKey): string` that reads the `#key` field — this function is the ONLY public path to the secret. Document at the call site that only `settle.ts` should import it.
  - Export `scrubSecrets(input: unknown): unknown` per the spec above.
- Create `packages/middleware/src/replay-store.ts`:
  - `class NonceStore { … }`. Internal `Map<string, Map<string, number>>` where the inner value is `validBefore` in **ms** (document the unit choice at the top of the file).
  - Constructor accepts `{maxEntries?: number}` with default 100_000.
  - `checkAndInsert({from, nonce, validBefore, now}): {accepted: boolean, reason?: 'nonce_already_used' | 'authorization_expired'}` — synchronous, no await. Steps: normalize `from`/`nonce` to lowercase; evict expired entries from `this.store.get(from)` based on `now`; if `validBefore <= now`, return `{accepted: false, reason: 'authorization_expired'}` (this is a safety-net — `verify.ts` SHOULD have caught this with the 5 s margin first); if `this.store.get(from)?.has(nonce)`, return `{accepted: false, reason: 'nonce_already_used'}`; else insert and return `{accepted: true}`. If `this.count >= this.maxEntries`, evict the FIFO-oldest-by-`validBefore` entry before inserting (loop until count < cap).
  - Expose `has({from, nonce, now}): boolean` and `insert({from, nonce, validBefore}): void` as primitives for tests, but the production code path is `checkAndInsert`.
  - Expose a `size(): number` for tests.
  - Top-of-file comment per D5 / Risks: "Single-process scope. Multi-instance support (Redis-backed) is post-MVP — see decisions.md D5."

The redaction guarantee is preserved (the secret didn't travel with the clone). Test asserts the clone does NOT contain the secret string.

- `scrubSecrets` with the 64-hex pattern matching legitimate non-secret data (e.g., a transaction hash). This is an accepted false-positive cost: per D13, the patterns are scoped to the security-logger boundary and error-handler boundary, where false-positives are preferable to leaks. Document this trade-off in the source comment.
- The `[Symbol.for('nodejs.util.inspect.custom')]` key (NOT `Symbol(...)`) is critical — `util.inspect` in Node specifically looks up the symbol via `Symbol.for`. Using a local `Symbol(...)` instance silently fails. Test catches the regression.

**Implementation hints:**
- `Buffer.byteLength(s, 'utf8')` is the correct size check for the 4 KB cap, NOT `s.length` (latter counts UTF-16 code units in the JS string, not octets on the wire).
- For strict-key validation, compare `Object.keys(obj).sort().join(',')` against the expected list joined; this catches both extra keys and missing keys in one pass.
- `Symbol.for('nodejs.util.inspect.custom')` is the documented, stable interop key (since Node 10). `util.inspect.custom` from `import { inspect } from 'node:util'` returns the same symbol; either works, but `Symbol.for` keeps the module Bun/Deno-portable.
- For non-enumerable private state, prefer `#privateField` (ergonomic, real privacy) — `Object.keys` and `JSON.stringify` skip class private fields by design. A `WeakMap<this, string>` keyed by instance is the alternative if `#field` syntax is somehow blocked by tsup config.
- For `scrubSecrets` cycle handling, use `const seen = new WeakSet()` populated on object entry; skip already-seen references and return a placeholder (`'<circular>'`) or just leave the reference in place (it's an object — won't match a regex).
- `pino` serializes via `JSON.stringify` with optional `stdSerializers`; since `toJSON` returns `'<redacted:relayer-key>'`, the log line will contain the literal string. `winston` with `format.json()` likewise uses `JSON.stringify`. Both tests just need to inspect the rendered output for absence of the raw key and presence of the redacted literal.
- For 100k cap eviction, an O(n) scan-on-overflow is acceptable because (a) overflow is rare, (b) a heap would optimize the wrong metric (insert speed) at the cost of more memory and a more complex implementation. Document the trade-off.
- For TTL eviction, evict only on `has`/`checkAndInsert` for the specific `from` (not a global walk) — keeps the hot path O(k) per developer rather than O(n) global.
- ESM-only: all imports use the `.js` extension in source even though the file is `.ts` (TypeScript NodeNext convention).
