## Description

Implement all "pure" (no external I/O, no on-chain calls, no network) modules of `@universal-paywall/middleware` in one task. These modules are leaf nodes in the middleware dependency graph and have no inter-dependencies beyond `types.ts`, which is why they're bundled into a single Wave 5 task that runs after T1 (scaffolding) and T3 (USDC spike + gas measurement). T3's JSON artefact at `contracts/scripts/arc-testnet-usdc-domain.json` is consumed by this task to populate the real `usdcEip712Name` / `usdcEip712Version` for the Arc Testnet network row. After this task lands, Wave 6 (`verify.ts` + `settle.ts`) can compose on top of stable types, a strict x402 codec, the typed error builder, the redacted relayer-key wrapper, and the synchronous NonceStore.

Concrete modules in scope:
- `packages/middleware/src/types.ts` — the TypeScript surface from tech-spec "Data Models → Middleware types (TypeScript, ESM)": `NetworkConfig`, `OpaqueRelayerKey` (interface — implementation in `relayer-key.ts`), `PaywallConfig`, `PaymentRequirements`, `ExactEvmPayload`, `PaymentPayload`. ESM-only, `export` per item.
- `packages/middleware/src/networks.ts` — `NETWORKS` registry. Per D1, keys MUST be present BOTH as alias (`'arc-testnet'`) AND canonical CAIP-2 (`'eip155:5042002'`) pointing at the same `NetworkConfig` object (object identity, not a copy) so lookup by either form returns the same row. Per tech-spec line 22, `arc-mainnet` is a placeholder gated by `enabled: false` (`chainId: 0`, `id: 'eip155:0'` — explicitly NOT `eip155:42161` which would alias Arbitrum One). `usdcEip712Name` / `usdcEip712Version` are populated from T3's JSON artefact (`contracts/scripts/arc-testnet-usdc-domain.json`). If that file is missing at module load — surface a "T3 not run" blocker error (NOT a silent stub) so consumers fail fast rather than ship with wrong EIP-712 domain values. `factoryAddress` / `vaultImplAddress` ship as `0x000…` placeholders WITH sentinel comments (per systemic-fix §13) so T11's deploy script can deterministically patch them.
- `packages/middleware/src/x402.ts` — four pure functions per tech-spec D1 + code-research §"Verified wire format":
  - `build402Body(req: PaymentRequirements, error?: string)` → `{ x402Version: 1, accepts: [req], ...(error ? {error} : {}) }`. JSON-serialization is the caller's job; this function returns the plain object. Internally calls `parseUsdPrice` to compute `maxAmountRequired` if needed (caller may also pre-compute).
  - `parseUsdPrice(input: string): bigint` (per systemic-fix §6) — converts a developer-supplied price string like `'0.01'` (USD) into a USDC base-unit `bigint` (6 decimals → `10000n` for `'0.01'`). Accepts `^\d+(\.\d{1,6})?$`. Throws `TypeError` with reason `invalid_price` on shape mismatch. Used by `build402Body` to populate `maxAmountRequired`. T9 tests it from here.
  - `decodeXPayment(header: string)` → `PaymentPayload`. Enforces `Buffer.byteLength(header, 'utf8') <= 4096` (D1 + Solution-7a `header_too_large`); base64-decodes; `JSON.parse`s; validates shape strictly: top-level keys must be exactly `{x402Version: 1, scheme: 'exact', network: string, payload: ExactEvmPayload}`, and `payload` must contain exactly `{signature, authorization: {from, to, value, validAfter, validBefore, nonce}}` and no extra fields (per D1 "strictly `{signature, authorization}` — no custom fields"). Throws typed errors that `errors.ts` consumes to map to HTTP 400 reasons (`header_too_large`, `malformed_payment_header`).
  - `encodeXPaymentResponse({success, transaction, network, payer})` → base64(JSON(...)). Used by `core.ts` on settle success.
- `packages/middleware/src/errors.ts` — typed builder per failure reason. Two HTTP status classes per tech-spec Solution §"How it works" step 7 and the SecurityLogger event catalog (D18). Uses the CANONICAL error reason strings from systemic-fix §4:
  - **HTTP 400** (request can't be parsed): `header_too_large`, `malformed_payment_header`. Body is x402-v1-shaped `{x402Version:1, accepts:[...], error}` so off-the-shelf clients see a structured response.
  - **HTTP 402** (payment required / declined): `payment_required`, `invalid_signature`, `insufficient_amount` (with extra fields `required`, `received`), `authorization_expired`, `authorization_not_yet_valid`, `nonce_already_used`, `network_mismatch`, `paused`, `vault_not_deployed`, `to_mismatch` (NOT the earlier-draft `recipient_mismatch`), `settlement_failed` (with sub-reason from D5 settlement taxonomy: `rpc_timeout`, `rpc_5xx`, `gas_estimate_revert`, `mine_timeout`, `receipt_reverted`, `relayer_no_balance`, `authorization_already_used_onchain`).
  - One canonical helper: `buildErrorResponse(reason, ctx?) → {status, headers, body}` where `body` matches the x402 v1 schema. The `accepts` field carries the same `PaymentRequirements` the original 402 would have, so retry clients can re-sign.
- `packages/middleware/src/relayer-key.ts` — implements `OpaqueRelayerKey` per D13. Defense-in-depth redaction surface:
  - The class stores the key in a non-enumerable instance field (`#privateField` — not enumerated by `Object.keys` or `JSON.stringify`).
  - `toJSON()` returns the literal string `'<redacted:relayer-key>'`.
  - `toString()` returns `'<redacted:relayer-key>'`.
  - `[Symbol.for('nodejs.util.inspect.custom')]` (NOT a top-level `import('util')` — use the global `Symbol.for` key to keep the module Node/Bun/Deno-portable) returns `'<redacted:relayer-key>'`.
  - A module-private `Symbol('OPAQUE_RELAYER_KEY_EXTRACT')` — exported only via a `getRelayerKeySecret(key: OpaqueRelayerKey)` function in this file — is the ONLY extraction path. `settle.ts` (Task 7) imports `getRelayerKeySecret`; no other module should.
  - A runtime brand: an internal `Symbol.for('@universal-paywall/middleware/OpaqueRelayerKey')` exposed via a `static is(x): x is OpaqueRelayerKey` predicate; `scrubSecrets` uses this brand check to redact instances that slipped into a payload.
  - `scrubSecrets(input: unknown): unknown` — recursive scrubber for use by D18 SecurityLogger and the error-formatting boundary. Walks objects/arrays; for any string value, replaces matches of ALL the following regex patterns with `<redacted:relayer-key>`:
    - `/0x[0-9a-fA-F]{64}/g` — 32-byte hex (raw key with `0x`).
    - `/\b[0-9a-fA-F]{64}\b/gi` — 32-byte hex without `0x` prefix, word-boundary anchored (env-var shape).
    - `/0x[0-9a-fA-F]{130}/g` — 65-byte ECDSA signature with `0x`.
    - Any value where `OpaqueRelayerKey.is(value)` is true → replaced entirely with the literal string `<redacted:relayer-key>`.
  - Handles cycles (visited `WeakSet`) so `scrubSecrets({a: x}); x.self = x` does not stack-overflow.
- `packages/middleware/src/replay-store.ts` — `NonceStore` per D5:
  - Storage: `Map<from, Map<nonce, validBefore>>`. `from` is the lower-cased `0x…` address; `nonce` is the `0x…` 32-byte hex. Normalize both to lowercase on every public-API boundary.
  - `has({from, nonce, now}): boolean` and `insert({from, nonce, validBefore})` execute as a single synchronous block when called via the convenience `checkAndInsert({from, nonce, validBefore, now}): {accepted: boolean}` — caller (`verify.ts`) uses `checkAndInsert` so there is no `await` between `has` and `insert`. The TOCTOU contract is enforced by API shape, not by convention.
  - TTL eviction is lazy: every `has` (and therefore every `checkAndInsert`) call first walks the outer `Map.get(from)` and deletes any inner entries whose `validBefore <= now`. Internal unit is **ms** (documented at the top of the file) consistent with verify.ts's `validBefore > now + 5_000 ms` check.
  - Hard cap 100k entries (summed across all inner maps). On overflow, evict the oldest-`validBefore` FIFO (scan once and drop the entry with the smallest `validBefore`; if that's still above cap, repeat). Document the O(n) eviction is acceptable because 100k is the absolute ceiling, not a hot path.
  - Multi-instance scope is explicitly NOT supported: add a top-of-file source comment per D5 / Risks row saying "single-process scope; multi-instance support is post-MVP — see decisions.md D5".

This task ships with comprehensive unit tests for all six modules under `packages/middleware/src/__tests__/` (canonical test directory per systemic-fix §2). Test file names match the smoke command: `x402.test.ts`, `errors.test.ts`, `networks.test.ts`, `relayer-key.test.ts`, `replay-store.test.ts`.

**Implementation hints:**
- `Buffer.byteLength(s, 'utf8')` is the correct size check for the 4 KB cap, NOT `s.length` (latter counts UTF-16 code units in the JS string, not octets on the wire).
- For strict-key validation, compare `Object.keys(obj).sort().join(',')` against the expected list joined; this catches both extra keys and missing keys in one pass.
- `Symbol.for('nodejs.util.inspect.custom')` is the documented, stable interop key (since Node 10). `util.inspect.custom` from `import { inspect } from 'node:util'` returns the same symbol; either works, but `Symbol.for` keeps the module Bun/Deno-portable.
- For non-enumerable private state, prefer `#privateField` (ergonomic, real privacy) — `Object.keys` and `JSON.stringify` skip class private fields by design.
- For `scrubSecrets` cycle handling, use `const seen = new WeakSet()` populated on object entry; skip already-seen references.
- `pino` serializes via `JSON.stringify` with optional `stdSerializers`; since `toJSON` returns `'<redacted:relayer-key>'`, the log line will contain the literal string. `winston` with `format.json()` likewise uses `JSON.stringify`. Both tests just need to inspect the rendered output for absence of the raw key and presence of the redacted literal.
- For 100k cap eviction, an O(n) scan-on-overflow is acceptable because (a) overflow is rare, (b) a heap would optimize the wrong metric (insert speed) at the cost of more memory and a more complex implementation. Document the trade-off.
- For TTL eviction, evict only on `has`/`checkAndInsert` for the specific `from` (not a global walk) — keeps the hot path O(k) per developer rather than O(n) global.
- ESM-only: all imports use the `.js` extension in source even though the file is `.ts` (TypeScript NodeNext convention).
- For the T3 artefact read in `networks.ts`: use `fs.readFileSync` at module-load (sync — this is bootstrap, not a request hot path). Resolve the path relative to the repo root or via a known-good relative path from the module location; document the resolution strategy.
- For sentinel comments in `networks.ts`, use the EXACT form `/* deploy-script:factoryAddress */` and `/* deploy-script:vaultImplAddress */` — T11's `sed` will key on these strings.
