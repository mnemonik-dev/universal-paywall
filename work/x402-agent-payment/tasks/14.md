## Required Skills

Before starting, load:
- `/skill:test-master` — [skills/test-master/SKILL.md](~/.claude/skills/test-master/SKILL.md)

## Description

Holistic full-feature test quality audit across the entire x402-agent-payment feature. This is part of the Audit Wave (Wave 10), running in parallel with Code Audit (Task 12) and Security Audit (Task 13). The auditor IS the review — there are no downstream reviewers for this task.

The audit verifies:
1. **Coverage targets are met** — middleware ≥85% line coverage, contracts ≥95% branch coverage.
2. **Presence of every test the tech-spec mandates** — every bullet in the Testing Strategy section is covered by an actual, executable test.
3. **Test quality** — assertions are meaningful (not just truthy), tests are isolated, mocks are realistic, no sleep-based flakiness, no over-mocking that hides real bugs.

Output: `work/x402-agent-payment/audit-tests.md` — structured report listing what was checked, what passed, what is missing or weak, and concrete recommendations. The report serves as the test-quality gate before Pre-deploy QA (Task 15).

## What to do

1. T14 owns the coverage gate verification. Verify the coverage reports produced by upstream tasks meet the thresholds; flag any gap with file/branch detail.
   - Middleware coverage report is produced by Task 9 (`npm test --workspace=@universal-paywall/middleware -- --coverage`). Re-run if needed; confirm line coverage ≥85% on `packages/middleware/src/`.
   - Contracts coverage report is produced by Task 5 (`cd contracts && npx hardhat coverage`). Re-run if needed; confirm branch coverage ≥95% on `contracts/contracts/`.
   - If either threshold is missed (overall OR per-file/branch for any critical module), list the uncovered files/branches in the report and explicitly flag the gap in the verdict.

2. For each tech-spec Testing Strategy item, locate the test that implements it. Cross-check by test name and assertion content (not just file presence). Specifically verify presence and quality of:

   **Middleware unit tests (`packages/middleware/src/__tests__/`):**
   - ajv schema validation of 402 body against vendored x402 v1 schema fixture (`packages/middleware/src/__tests__/fixtures/x402-v1.schema.json`).
   - EIP-712 tamper tests — separate failing test for tampered `chainId`, `verifyingContract`, `domain.name`, `domain.version` (4 distinct tests).
   - Network id normalization (CAIP-2 ↔ alias) round-trips — both directions, all enabled entries in `networks.ts`, including unknown id → typed error.
   - `verify.ts` boundary tests — `to != computedVaultAddress`, `value < required`, `validBefore <= now + 4s` fail / `validBefore == now + 6s` pass, `validAfter > now` returns `authorization_not_yet_valid`, network mismatch.
   - `settle.ts` failure taxonomy — all 7 reasons: `rpc_timeout`, `rpc_5xx`, `gas_estimate_revert`, `mine_timeout`, `receipt_reverted`, `relayer_no_balance`, `authorization_already_used_onchain`. Each mocked + asserted on resulting 402 reason.
   - Replay-store retention on settlement failure — `receipt_reverted` → nonce NOT deleted → retry returns `nonce_already_used`.
   - `relayer-key.ts` redaction — `JSON.stringify`, `toString`, `util.inspect` (custom symbol), `structuredClone`, `pino` serialization, `winston` serialization, 0x[a-f0-9]{64} pattern in error stacks, non-enumerable field.
   - D14 startup chainId pin — `NetworkMismatchError` on mismatched RPC.
   - D18 SecurityLogger — all 10 event names emit on trigger; payloads pass redactor; default no-op logger silent.
   - Factory-state cache — 5s TTL; fresh registration invisible until expiry; RPC error returns last good value if non-stale; otherwise propagates as `settlement_failed.reason = "rpc_5xx"`.
   - Adapter unit tests — `withPaywall` (Node http) propagates handler exceptions, sets response headers, flushes before user handler. `fastifyPaywall` integrates into Fastify `preHandler` lifecycle, sends 402 when X-PAYMENT absent, preserves reply chaining on 200.
   - `errors.ts` — each reason produces canonical body matching schema.
   - Price parsing edge cases — accepted and rejected inputs as listed in spec.

   **Contract tests (`contracts/test/`):**
   - `PaymentSplitterFactory.test.ts` — `register` happy path / `AlreadyRegistered` / `EnforcedPause`; `setFeeBps` / `setPlatformTreasury` / `Ownable2Step` / `pause`/`unpause`; constructor revert paths.
   - **CREATE2 cross-component invariant** — off-chain middleware helper (`Clones.predictDeterministicAddress` mirror) matches on-chain `computeVaultAddress` byte-equal for ≥3 distinct developer EOAs.
   - **`register` reentrancy invariant** — malicious developer EOA whose `initialize` callback re-enters `factory.register()` is blocked.
   - `VaultDeployed` event — arg matches predicted address; unregistered → `0x0`; two developers in same block → two distinct addresses.
   - `PaymentVaultImpl.test.ts` — single-call `initialize`; `_disableInitializers()` impl-hijack test (D15) — direct `initialize` on `vaultImpl` (not a clone) reverts.
   - **No `receive()` / `fallback()`** (D16) — sending native asset to vault reverts.
   - **ABI assertions** (D17) — `setDeveloper(address)` and `setFactory(address)` selectors do NOT exist in ABI.
   - `withdraw` — developer-only, `NoBalance`, happy path split, `Withdrawal` event.
   - Fee math edge cases — `feeBps = 0` → no second transfer; `feeBps = 1000` → 10% fee; `gross = 1` micro-USDC → `fee = 0` (truncation), full unit to developer.
   - **Fee-snapshot semantics** — `setFeeBps` between two payments and one withdraw → withdraw uses fee at withdraw time.
   - Reentrancy — malicious treasury re-entering `withdraw` blocked.
   - Withdraw works when factory is `paused()`.

   **Forked integration (`contracts/test/integration/forked-e2e.test.ts`):**
   - Full happy path against BOTH adapters (Node http AND Fastify) in same process.
   - **Cross-adapter replay protection** — pay via Node http, retry SAME X-PAYMENT on Fastify endpoint in same process → 402 `nonce_already_used` (proves process-singleton NonceStore).
   - Assertions: 200 received, vault USDC balance increased by `value`, NonceStore rejects same-nonce retry, on-chain `usdc.authorizationState(from, nonce)` is true after settle.
   - Rejection branches: `vault_not_deployed`, `paused`.

   **Register CLI test (`packages/middleware/src/__tests__/register-cli.test.ts`):**
   - Runs `register.ts` against forked node + factory; tx hits `factory.register()`.
   - **Stdout/stderr scrub** — CLI never prints developer key.
   - Incorrect `REGISTER_KEY` format → non-zero exit with typed error not exposing input.

3. Audit test quality across all suites. For each suite, evaluate:
   - **Meaningful assertions** — tests assert concrete expected values, not just `toBeTruthy()` / `not.toThrow()`. Flag any test whose assertions only check that something happened, not what happened.
   - **Test isolation** — each test sets up its own state; no order dependencies; shared fixtures reset between tests.
   - **Mock realism** — mocks return shapes that match real RPC/contract responses; not just stubbed to make tests pass. Flag mocks that diverge from production behavior.
   - **No sleep-based flakiness** — no raw `setTimeout` / `sleep` for synchronization. Cache TTL tests must use deterministic time control (vi.useFakeTimers or equivalent), not real sleeps.
   - **Over-mocking** — flag tests that mock the unit under test or mock away the assertion's subject.

4. Write the audit report to `work/x402-agent-payment/audit-tests.md` using the structure under "Audit report structure" below.

**Edge cases:**
- Coverage tool reports per-file numbers — if the overall number meets the threshold but a critical file (e.g. `settle.ts`, `verify.ts`) is under-covered, flag it. Aggregate numbers can hide weak spots.
- A test may exist by name but assert nothing meaningful (e.g. `expect(result).toBeDefined()`). Read the body of each mandatory test, not just the file listing.
- A test may pass for the wrong reason (over-mocking — the mock is the answer). Spot-check mocks against the real source module behavior.
- Hardhat branch coverage on Solidity counts each `require` / `if` branch — missing a single revert branch can fail the 95% bar. Identify exact branches if shortfall.
- The forked-e2e test runs in one process by design (cross-adapter replay test depends on this). If the test file launches a subprocess instead, the cross-adapter invariant is not actually proven — flag it.

**Implementation hints:**
- For coverage reports: vitest produces `coverage/coverage-summary.json` and `coverage/index.html`. Hardhat (solidity-coverage) produces `coverage.json` and `coverage/index.html`.
- Use `jq` against coverage JSON for precise per-file numbers.
- For the test quality audit, prefer reading test files directly over running them — the audit is about content, not green/red.
- If a tech-spec requirement is present but split awkwardly across files, that is acceptable (mark PRESENT with a note). The audit cares about whether the property is verified, not stylistic placement.
