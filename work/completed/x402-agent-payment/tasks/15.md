# Task 15: Pre-deploy QA (requires user approval of tech-spec)

## Required Skills

Before starting, load:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](~/.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Final gate before deploy and npm publish (Task 16). This task validates that the entire x402-agent-payment feature is production-ready by running the full automated test suite and methodically walking every acceptance criterion in `user-spec.md` and `tech-spec.md`. The output is a single checklist report (`qa-report.md`) that either greenlights deploy or blocks it with a concrete list of unmet criteria.

Pre-conditions:
- **`tech-spec.md` frontmatter must be `status: approved`.** If it is `status: draft`, abort immediately and surface the block to the user — deploy cannot proceed without explicit user approval of the tech-spec.
- All prior waves (Tasks 1–14) must be marked `status: done`.

This is a verification-only task — it does NOT modify code or tests. If any AC fails, it stops, reports, and hands back to the relevant implementation task.

## What to do

1. **Approval gate:** Read `work/x402-agent-payment/tech-spec.md` frontmatter. If `status != approved`, write a one-line blocker to `qa-report.md` (`BLOCKED: tech-spec status is "<value>", awaiting user approval`) and stop. Do not run tests.

2. **Run the three canonical test suites** from repo root and collect exit codes + summaries. These three commands cover the entire automated surface — middleware vitest (incl. forked-e2e), contracts (forge), and live Arc Testnet e2e:
   - `npm test --workspace=@universal-paywall/middleware` — middleware vitest suite. Must exit 0. Capture coverage and verify ≥85% line coverage on `src/`. This includes the forked-e2e integration test at `packages/middleware/src/__tests__/integration/forked-e2e.test.ts` (spawns anvil, deploys via viem — see iter-4 addendum §4 T10). Confirm the forked-e2e file actually ran (grep vitest output for its suite name) — do not accept silent skipping.
   - `cd contracts && forge test && forge coverage --report summary` — Foundry contract tests (unit + fuzz + invariant under `contracts/test/`) AND branch-coverage report. Must exit 0. Verify ≥95% branch coverage on both `PaymentSplitterFactory` and `PaymentVaultImpl` (excluding `test/`, `script/`, `lib/`). Fuzz + invariant tests must be present in the run output (T5 deliverables).
   - `ARC_TESTNET_E2E=1 npm run test:e2e --workspace=@universal-paywall/middleware` — live Arc Testnet e2e (requires funded relayer key + test USDC in env per `deployment.md`: `ARC_RPC_URL`, `PAYWALL_RELAYER_KEY`). Must exit 0; live tx hash must appear in the log and resolve on `https://testnet.arcscan.app`.

   Additionally, run these supporting checks:
   - `npm install && npm run build --workspace=@universal-paywall/middleware` — confirm ESM-only build succeeds; inspect output to confirm no CJS artifacts and bundle <30 KB minified+gzip (excluding `viem`).
   - `cd contracts && forge build` — confirm clean compile under pragma `^0.8.20`, OZ 5.x resolves via `remappings.txt` (foundry).
   - `gitleaks detect --no-banner` (or repo pre-commit equivalent) — confirm no secret-shaped patterns in tree.

3. **Walk every acceptance criterion** from both specs and mark pass/fail with evidence (test name, file:line, log excerpt, or on-chain tx hash). Group the report into the same sections as user-spec to make gaps obvious. Total: 41 user-spec ACs + 10 tech-spec ACs = 51 ACs.
   - **user-spec → Middleware** (lines 148–176): **18 ACs** covering `withPaywall` export, `fastifyPaywall` export, 402 body shape (ajv-valid against vendored x402 v1 schema at `packages/middleware/src/__tests__/fixtures/x402-v1.schema.json`), valid `X-PAYMENT` flow (EIP-712 ecrecover, `to == computeVaultAddress`, value/time bounds, NonceStore, network match, paused/vault-not-deployed gates, `transferWithAuthorization` settle, receipt wait, 200 + `X-PAYMENT-RESPONSE` header) — this is one composite AC with sub-bullets, walk every sub-bullet, then each error code as its own AC: `invalid_signature`, `insufficient_amount`, `authorization_expired`, `authorization_not_yet_valid`, `nonce_already_used`, `settlement_failed` with full reason taxonomy, `network_mismatch`, `header_too_large` (HTTP 400), `malformed_payment_header` (HTTP 400), `NETWORKS` dual-keyed map, chain-id mismatch throw, `OpaqueRelayerKey` redaction (non-enumerable, no leak in `JSON.stringify` / error stacks), npm package shape (ESM-only), price-to-amount integer math.
   - **user-spec → Contracts: PaymentSplitterFactory** (lines 178–196): **11 ACs** — constructor validation, `Ownable2Step`, `Pausable`, `vaultImpl` deployed in constructor, `register()` semantics (`whenNotPaused`, no double-register, `Clones.cloneDeterministic` with salt = developer EOA, `initialize`, `vaults` mapping, `VaultDeployed` event), `computeVaultAddress` matches `predictDeterministicAddress`, `setFeeBps` (owner, cap 1000, event), `setPlatformTreasury` (owner, non-zero, event), `pause`/`unpause`, all view getters, arcscan verification.
   - **user-spec → Contracts: PaymentVaultImpl** (lines 198–205): **6 ACs** — (1) `Initializable` + storage-based `ReentrancyGuard` (not transient), (2) `initialize(address _developer)` semantics, (3) `withdraw()` math + transfer order (developer FIRST, then platform per systemic-fix §7) + `SafeERC20` + event, (4) withdraw works while `factory.paused() == true`, (5) no setters for `developer`/`factory`, (6) no `receive()` payable.
   - **user-spec → Деплой и тестирование** (lines 207–214): **6 ACs** — `npm test --workspace=@universal-paywall/middleware` ≥85% line, `cd contracts && forge test && forge coverage --report summary` ≥95% branch, `forked-e2e.test.ts` runs in CI without env flag (under `packages/middleware/src/__tests__/integration/` — middleware-side per iter-4 addendum §4 T10), `arc-testnet-e2e.test.ts` (at `packages/middleware/src/__tests__/integration/`) gated on `ARC_TESTNET_E2E=1`, deploy script (`forge script script/Deploy.s.sol` + `contracts/scripts/post-deploy.ts`) runs end-to-end, README onboarding flow.
   - **tech-spec → Acceptance Criteria (technical complement)** (lines 524–537): **10 ACs** — build/compile success (`forge build` + middleware `npm run build`), coverage thresholds (line ≥85% middleware, branch ≥95% contracts via `forge coverage`), forked + live e2e pass, deploy script (forge script Deploy.s.sol + TS post-deploy) + arcscan verification, no committed secrets / gitleaks blocks, bundle size <30 KB excluding viem, `engines.node: ">=20"`, ESM-only exports map.

4. **Cross-check artifacts** produced by the audit tasks: `audit-code.md` (Task 12), `audit-security.md` (Task 13), `audit-tests.md` (Task 14). Every blocker-level finding in those reports must be resolved (linked commit or fix-task). If any blocker is open, mark the corresponding AC as failed.

5. **Write `work/x402-agent-payment/qa-report.md`** with:
   - Header: feature name, commit SHA, tech-spec status, run date, runner identity.
   - Summary block: total ACs / passed / failed / blocked, decision (`READY FOR DEPLOY` / `BLOCKED`).
   - Suite results table — one row per command: command, exit code, duration, coverage %, log path. At minimum the three canonical suites + the supporting checks listed in step 2.
   - Per-section AC checklist (the 5 groups above: Middleware 18, Factory 11, Vault 6, Деплой 6, tech-spec 10 = 51 total) with evidence for each item.
   - Open blockers list (if any) with concrete next step per item.
   - Sign-off line — only present if all 51 ACs pass.

6. **If any AC fails:** do NOT pass the gate. Report stays as `BLOCKED`, Task 16 must not start. Hand back to the agent owning the failing area.
