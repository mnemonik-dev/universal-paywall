# Task 12: Code Audit

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-reviewing` — [skills/code-reviewing/SKILL.md](~/.claude/skills/code-reviewing/SKILL.md)

## Description

Holistic full-feature code quality audit across every source file produced by Waves 1–6 of the `x402-agent-payment` feature. This is an Audit Wave task: the auditor IS the review — no further reviewers run after this task. The goal is to surface code-quality issues that per-task reviews could miss because each per-task review only sees a slice of the system. This audit looks at the feature as a whole: cross-module consistency, adherence to the Architecture decisions (D1–D18), and overall code health.

Scope of files audited:
- `packages/middleware/src/` — all middleware modules (core, adapters, verify, settle, x402, networks, errors, relayer-key, replay-store, types) and their `__tests__` siblings.
- `contracts/contracts/` — Solidity sources (`PaymentSplitterFactory.sol`, `PaymentVaultImpl.sol`, interfaces, mocks).
- `contracts/deploy/` — deploy scripts (`01_deploy_factory.ts`).
- `contracts/scripts/` — Hardhat scripts (`verify-usdc-eip3009.ts`). NOTE: this is a Wave 1 spike, not a production module. Audit at low severity only — no Blocker/Major findings unless it is actively wired into the production deploy path.
- `scripts/` — repo-root CLI scripts (`register.ts`).

Deliverable: a single audit report at `work/x402-agent-payment/audit-code.md` with categorized findings, severities, and concrete fix recommendations. No source code is modified by this task — analysis only. If the report flags issues, the feature-execution lead will spawn a fixer task in a follow-up step.

## What to do

1. Load all in-scope source files into context. Sanity-check the file inventory against tech-spec Architecture "What we're building/modifying".
2. Run the 11 code-review dimensions from the `code-reviewing` skill across the full feature surface. For each finding, record: file + line range, severity (blocker / major / minor / nit), the issue, and a concrete fix recommendation.
3. Targeted cross-cutting checks specific to this feature:
   - **Shared-resource compliance with Architecture "Shared resources" table** — confirm `PublicClient`, `WalletClient`, `NonceStore`, factory-state cache, `NETWORKS`, and `OpaqueRelayerKey` are owned/consumed by the modules listed, with the documented instance counts. Flag any leakage of shared state or duplicate instances.
   - **Adapter consistency between `adapters/node-http.ts` and `adapters/fastify.ts`** — same `core.paywall(req, opts)` entrypoint, same error mapping, same header propagation (X-PAYMENT-RESPONSE), same handler-exception semantics. Any behavior drift between the two is a finding.
   - **Naming consistency** — function, type, file, and error-reason names follow one style across the package (camelCase for TS, PascalCase for types, kebab-case for files; the seven settlement reasons from D5 spelled identically everywhere they are referenced).
   - **Error handling** — every async path has a defined failure mode; no swallowed errors except the explicit `try/catch` around `SecurityLogger` calls (D18); no `console.error` left in production paths; no raw `0x...` hex (private key or signature shape) printable through any error path (D13).
   - **Complexity** — function length, nesting depth, and module cohesion. Flag any module/function that violates project patterns.md guidelines.
   - **Architectural consistency** — D1 (strict x402 v1 wire format), D6 (framework-agnostic core + thin adapters — no framework-specific code leaking into `core.ts`), D7 (viem everywhere, no `ethers` imports), D8 (Solidity `^0.8.20`, OZ 5.x storage-based `ReentrancyGuard`), D14 (chainId pin at first use), D15 (`_disableInitializers()` present, no `selfdestruct` / `delegatecall` in vault impl), D16 (no `receive()` / `fallback()` on vault), D17 (no setters for `developer` / `factory`), D18 (SecurityLogger surface wired in every emit site listed in the event catalog).
   - **ESM-only invariants** — no CJS `require(...)` in `packages/middleware/src/`; no `module.exports`; `package.json` `"type": "module"` not undermined by a stray `.cjs`.
   - **Custom errors over `require(...)` strings** — Solidity sources use the named custom errors from Data Models (`NotDeveloper`, `AlreadyRegistered`, `InvalidFeeBps`, `ZeroAddress`, `NoBalance`); flag any leftover `require(cond, "string")`.
   - **NatSpec invariants on `PaymentVaultImpl`** — the `@custom:security-invariant` lines from D15 / D17 are present verbatim.
4. Group findings by severity. Write `work/x402-agent-payment/audit-code.md` using this structure:
   - **Summary** — 2–3 sentences: overall health, top concerns, recommended next step (fix vs. ship).
   - **Findings by severity** — Blocker, Major, Minor, Nit. Each finding: file:line, dimension (from the 11 review dimensions), description, fix recommendation.
   - **Architecture-decision compliance matrix** — table mapping D1–D18 to "Compliant / Violation / N/A" with a short note.
   - **Cross-cutting observations** — adapter consistency, naming consistency, error-handling consistency, shared-resource compliance — each as its own subsection.
   - **Verdict** — `clean` (no Blocker/Major), `needs-fix` (any Blocker/Major), or `advisory` (Minor/Nit only).
5. Commit the report as part of the audit task. Do NOT modify any source file.

## Details

**Files (read for analysis — current state at end of Wave 6):**
- `packages/middleware/src/index.ts` — public exports surface; confirm it re-exports `withPaywall`, `fastifyPaywall`, `NETWORKS`, and the public types from `types.ts`.
- `packages/middleware/src/core.ts` — `paywall(req, opts)` orchestrator; should be framework-agnostic, lazy-init `PublicClient` / `WalletClient` / `NonceStore` per the Shared resources table.
- `packages/middleware/src/adapters/node-http.ts` — `withPaywall(handler, opts)`; thin wrapper around `core.paywall`.
- `packages/middleware/src/adapters/fastify.ts` — `fastifyPaywall(opts)` Fastify plugin; same semantics via `preHandler`.
- `packages/middleware/src/verify.ts` — EIP-712 ecrecover via `recoverTypedDataAddress`; Solution step-7c checks; classifies error reasons.
- `packages/middleware/src/settle.ts` — `USDC.transferWithAuthorization` via `WalletClient.writeContract` with explicit `from`; receipt-await with 30s timeout; 7-reason failure taxonomy from D5.
- `packages/middleware/src/x402.ts` — `build402Body`, `encodeXPaymentResponse`, `decodeXPayment`, 4 KB header cap.
- `packages/middleware/src/networks.ts` — `NETWORKS` keyed by alias (`arc-testnet`) and CAIP-2 (`eip155:5042002`); USDC EIP-712 name/version patched by Task 3.
- `packages/middleware/src/errors.ts` — `buildErrorResponse(reason)` producing typed 402 / 400 bodies.
- `packages/middleware/src/relayer-key.ts` — `OpaqueRelayerKey` wrapper with non-enumerable field, `toJSON` / `toString` / `[util.inspect.custom]` redactions, symbol-based extract.
- `packages/middleware/src/replay-store.ts` — synchronous `has` + `insert`, lazy TTL eviction, 100k cap.
- `packages/middleware/src/types.ts` — `PaymentRequirements`, `PaymentPayload`, `ExactEvmPayload`, `PaywallConfig`, `NetworkConfig`, `OpaqueRelayerKey`.
- `contracts/contracts/PaymentSplitterFactory.sol` — `Ownable2Step` + `Pausable`; `register()` deploys `Clones.cloneDeterministic` proxy; `setFeeBps` 0–1000 bps; `setPlatformTreasury`.
- `contracts/contracts/PaymentVaultImpl.sol` — `Initializable` + `ReentrancyGuard`; constructor calls `_disableInitializers()`; `withdraw()` reads `usdc.balanceOf(this)` and splits to developer + treasury; no `receive` / `fallback` / setters.
- `contracts/contracts/interfaces/IERC3009.sol` — selectors for `transferWithAuthorization` and `authorizationState`.
- `contracts/contracts/mocks/MockUsdcEip3009.sol` — test-only USDC mock.
- `contracts/deploy/01_deploy_factory.ts` — deploys impl + factory; verifies on arcscan; patches `networks.ts`.
- `contracts/scripts/verify-usdc-eip3009.ts` — Wave 1 spike.
- `scripts/register.ts` — developer EOA CLI calling `factory.register()`.

**Dependencies:**
- Depends on Tasks 9, 10, 11 (Wave 6 completed) — audit needs the final state of the code, including tests and tooling.
- Runs in parallel with Task 13 (Security Audit) and Task 14 (Test Audit) inside the Audit Wave.

**Edge cases / scope discipline:**
- T12, T13, T14 run independently in parallel (Wave 10). Do NOT attempt to coordinate with or cross-reference the other audits during execution — each audit is independent, and the orchestrator merges/de-duplicates findings after the wave completes.
- Security findings (reentrancy, access control, EIP-712 chain replay, CREATE2 collision, OpaqueRelayerKey leak shapes) are the primary domain of the Security Audit. Do not chase them as the headline issue here; if you stumble on a clear security defect while reading code for quality, log it as a Minor "noted for security audit" observation in cross-cutting observations and move on.
- Test-quality findings (coverage gaps, missing tamper tests, weak assertions) are the primary domain of the Test Audit. Code structure inside `__tests__` directories is in scope only when it points at a defect in the source under test.
- Do not propose architectural rewrites — if an Architecture decision (D1–D18) is the cause of friction, log it as an observation, not a finding.
- `contracts/scripts/verify-usdc-eip3009.ts` is a Wave 1 spike, not production code. Audit at low severity only — surface Minor/Nit findings if any, do NOT raise Blocker/Major against it.

**Implementation hints:**
- Use the 11 review dimensions from the `code-reviewing` skill as the rubric, not free-form opinion.
- Severity calibration: Blocker = correctness/consistency violation that breaks an acceptance criterion; Major = clear defect or material drift from a Decision; Minor = readability, naming, or local cohesion; Nit = style.
- The report is read by the lead agent — keep findings actionable. Every finding should answer "where, what, why, fix".
- The compliance matrix is the fastest way for the lead to see whether the feature is consistent with the agreed Decisions; do not skip it.
