# Task 17: Post-deploy verification

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:post-deploy-qa` — [skills/post-deploy-qa/SKILL.md](~/.claude/skills/post-deploy-qa/SKILL.md)

## Description

После того как Task 16 задеплоил `PaymentSplitterFactory` + `PaymentVaultImpl` на Arc Testnet и опубликовал `@universal-paywall/middleware@0.1.0-alpha.0` в npm, эта задача делает живую верификацию того, что задеплоенный артефакт действительно работает в продакшен-окружении (Arc Testnet) и что опубликованный npm-пакет корректно ставится и отдаёт x402-совместимый 402.

Это **finalize-стадия** фичи: до этой задачи acceptance criteria из user-spec и tech-spec проверены только локально / на форке / в CI; здесь — против настоящего deployed factory и настоящего npm tarball. Результат фиксируется в `work/x402-agent-payment/post-deploy-report.md` и становится основой для финального коммита фичи.

Три независимых живых проверки:

1. **On-chain reads против deployed factory** на Arc Testnet (chainId 5042002). Проверяем что параметры контракта совпадают с deploy-конфигом из Task 16 — никаких "deployed но с неправильным fee" сюрпризов.
2. **Live e2e** против настоящего factory: реальный EIP-3009 signer, реальный USDC контракт Arc Testnet, реальная settlement-транзакция через relayer. Гейтится переменной окружения `ARC_TESTNET_E2E=1`, реализована в Task 10. npm-script `test:e2e` (добавлен в `packages/middleware/package.json` Task 1) запускает vitest над `src/__tests__/integration/`.
3. **HTTP smoke опубликованного npm-пакета**: ставим `@universal-paywall/middleware@0.1.0-alpha.0` в чистую директорию (не из workspace, не из локального tarball), поднимаем минимальный Node http server с `withPaywall()`, делаем curl без X-PAYMENT и проверяем что body соответствует x402 v1 wire format (ajv-валидация против схемы из T9-фикстуры).

Никакого кода не пишем — только верифицируем deployed артефакты. Если что-то фейлится — задача проваливается, фича не закрывается, fix идёт через возврат к Task 16 (или к соответствующей задаче в коде).

## What to do

1. Подготовить рабочую директорию для верификации вне workspace репозитория (чтобы не словить случайный `npm link`-style эффект и не наследовать `node_modules`). Например `SCRATCH=$(mktemp -d)`.

   - В этой scratch-директории сделать `npm init -y`, затем **явно записать `package.json` с `"type": "module"`** — это критично, чтобы статические ESM-импорты (`import { ... } from '@universal-paywall/middleware'`, `import Ajv from 'ajv'`) работали в `server.js` и `validate-x402.mjs`. Без `type: module` Node воспринимает `.js` как CommonJS и static ESM-импорты падают `SyntaxError: Cannot use import statement outside a module`.
   - Пример инициализации:
     ```bash
     SCRATCH=$(mktemp -d -t upw-postdeploy)
     cd "$SCRATCH"
     npm init -y >/dev/null
     node -e 'const fs=require("fs");const p=JSON.parse(fs.readFileSync("package.json"));p.type="module";fs.writeFileSync("package.json",JSON.stringify(p,null,2));'
     ```

2. Считать deployed factory address и expected параметры:
   - `PAYMENT_SPLITTER_FACTORY_ADDRESS` — из `packages/middleware/src/networks.ts` (Task 16 запатчил его туда через sentinel `/* deploy-script:factoryAddress */`) либо из deploy-лога Task 16 в `contracts/deploy/`.
   - `PLATFORM_TREASURY_ADDRESS` expected — из `.env` / deploy-конфига Task 16.
   - **Expected `factory.owner()` — берём из коммитнутого deploy-лога Task 16** (см. post-deploy запись в `work/x402-agent-payment/decisions.md`, добавленную после Task 16; там зафиксированы deployed factory address, owner address и tx-hash деплоя). Это canonical источник, **не** `.env` исполнителя и не локальная переменная.
   - Expected `feeBps == 50` (uint16, 0.5% — зафиксировано в tech-spec Data Models / contracts).
   - `ARC_RPC_URL` — канонический env-var (default `https://rpc.testnet.arc.network`, fallback `https://5042002.rpc.thirdweb.com` из `deployment.md`).

3. Шаг 1 — On-chain reads. **Не создавать отдельный файл-скрипт.** Использовать программный viem inline через `node -e "..."` one-liner (см. systemic-fix §10). Скрипт через Arc Testnet RPC читает factory:
   - `factory.feeBps()` → ожидание `50` (тип `uint16`, согласно tech-spec Data Models)
   - `factory.platformTreasury()` → ожидание совпадения с `PLATFORM_TREASURY_ADDRESS` из deploy-конфига
   - `factory.owner()` → ожидание совпадения с owner-адресом из коммитнутой post-deploy записи в `work/x402-agent-payment/decisions.md` (запись добавлена после Task 16)
   - `factory.paused()` → ожидание `false`

   Каждое чтение — отдельный assert. Любое расхождение = FAIL, записываем actual vs expected в репорт.

   Реализационная заметка: viem уже стоит в scratch-директории (он tranзитивная зависимость `@universal-paywall/middleware`, ставится автоматически шагом 5). Поэтому on-chain reads выполняются ПОСЛЕ `npm install`, либо `viem` ставится явно одной строкой: `npm install viem`. ABI minimum fragment с четырьмя view-функциями (`feeBps`, `platformTreasury`, `owner`, `paused`) указывается inline в команде. **ABI types должны точно соответствовать Solidity-сигнатурам из tech-spec Data Models**: `feeBps()` возвращает `uint16`, не `uint256` — расхождение типа в ABI приведёт к неверному декодированию слотов и фантомному PASS / FAIL. Пример минимального вызова:

   ```bash
   REPO_ROOT=$(git rev-parse --show-toplevel)
   FACTORY=0x...   # подставить deployed address
   PLATFORM_TREASURY_EXPECTED=0x...
   OWNER_EXPECTED=0x...   # из post-deploy записи в decisions.md
   node -e "
   import('viem').then(async ({createPublicClient,http,parseAbi})=>{
     const c=createPublicClient({transport:http(process.env.ARC_RPC_URL)});
     const abi=parseAbi([
       'function feeBps() view returns (uint16)',
       'function platformTreasury() view returns (address)',
       'function owner() view returns (address)',
       'function paused() view returns (bool)',
     ]);
     const f=process.env.FACTORY;
     const r={
       feeBps:await c.readContract({address:f,abi,functionName:'feeBps'}),
       platformTreasury:await c.readContract({address:f,abi,functionName:'platformTreasury'}),
       owner:await c.readContract({address:f,abi,functionName:'owner'}),
       paused:await c.readContract({address:f,abi,functionName:'paused'}),
     };
     console.log(JSON.stringify(r));
   })"
   ```

   Парсим JSON, сравниваем с expected, печатаем `OK` / `FAIL: actual=... expected=...` по каждому полю.

   Если же по каким-то причинам one-liner не вмещается, разрешено создать **временный** файл (например `$SCRATCH/read-factory.mjs`) средствами `node -e`/heredoc, выполнить его и **немедленно удалить** (`rm -f $SCRATCH/read-factory.mjs`). В репозиторий ничего из этого не коммитится. Никакого файла в `scripts/postdeploy/` не существует и создавать его не нужно.

Test использует real EIP-3009 signer против deployed factory + vault на Arc Testnet via real RPC. 5. Шаг 3 — HTTP smoke npm-пакета (продолжение в scratch-директории, инициализированной в шаге 1):
   - `npm install @universal-paywall/middleware@0.1.0-alpha.0` (тянем из публичного npm registry, не из local). Версия указывается **точно**, не диапазон.
   - Создать `server.js` (ESM, т.к. `type: module`) который импортирует `withPaywall` из установленного пакета и поднимает Node http server на свободном порту (выбираем порт, например через `0` и читаем `.address().port`, либо фиксируем `PORT=18402` если свободен). Минимальная paywall-обёртка: `price: '0.01'`, `network: 'arc-testnet'`, `developerEoa` — любой валидный test EOA (например vanity-адрес из доки).
   - Добавить health-endpoint (`/health` → 200) — нужен для readiness loop (см. ниже).
   - Запустить server в фоне: `node server.js &` и сохранить PID.
   - **Readiness loop вместо фиксированного sleep** — ждём пока сервер начнёт принимать соединения. Никаких `sleep 1`:
     ```bash
     REPO_ROOT=$(git rev-parse --show-toplevel)
     PORT=18402
     for i in {1..20}; do
       curl -sf "http://localhost:$PORT/health" >/dev/null && break
       sleep 0.2
     done
     ```
   - Сделать protected запрос: `curl -sS -o body.json -w '%{http_code}\n' "http://localhost:$PORT/protected"` — ожидание HTTP `402`, body — валидный x402 v1 JSON.
   - Ajv-валидация body против x402 v1 schema. Источник схемы — **vendored fixture, написанная Task 9**: `packages/middleware/src/__tests__/fixtures/x402-v1.schema.json` (см. systemic-fix §2 и §1; не Task 7). На время smoke копируем её в scratch: `cp "$REPO_ROOT/packages/middleware/src/__tests__/fixtures/x402-v1.schema.json" "$SCRATCH/x402-v1.schema.json"` (переменная `REPO_ROOT` определена выше через `git rev-parse --show-toplevel`). Validation-скрипт можно вызывать inline через `node -e`, либо записать `validate-x402.mjs` в scratch и удалить после:
     ```bash
     npm install ajv ajv-formats
     node -e "
     import('ajv').then(async ({default:Ajv})=>{
       const af=(await import('ajv-formats')).default;
       const fs=await import('node:fs');
       const schema=JSON.parse(fs.readFileSync('x402-v1.schema.json','utf8'));
       const body=JSON.parse(fs.readFileSync('body.json','utf8'));
       const ajv=new Ajv({allErrors:true,strict:false}); af(ajv);
       const ok=ajv.compile(schema)(body);
       if(!ok){console.error(JSON.stringify(ajv.errors,null,2));process.exit(1)}
       console.log('OK');
     })"
     ```
     Проверяемые поля (как фиксирует tech-spec): `x402Version`, `accepts[]` с обязательными `scheme`, `network`, `maxAmountRequired`, `resource`, `description`, `mimeType`, `payTo`, `asset`, `extra.name`, `extra.version`, `extra.assetTransferMethod`. Любое поле missing / wrong type = FAIL.
   - Остановить server (`kill $PID`), удалить scratch-директорию (`rm -rf $SCRATCH`) **после фиксации результата в отчёт**.

6. Собрать результаты всех трёх шагов в `work/x402-agent-payment/post-deploy-report.md`:

   - Раздел "Live e2e": команда (точная), exit code, ссылка на settlement tx hash на Arc Testnet explorer (Blockscout) если есть.

   - Итоговый verdict: PASS / FAIL. Если FAIL — список расхождений.

7. Закоммитить `post-deploy-report.md` в репозиторий. Задача считается выполненной только если verdict == PASS.

- Внешние: Arc Testnet RPC доступен (`ARC_RPC_URL`, default `https://rpc.testnet.arc.network`, fallback `https://5042002.rpc.thirdweb.com`); публичный npm registry доступен; relayer EOA (`PAYWALL_RELAYER_KEY`) имеет USDC на Arc Testnet (для оплаты gas в live e2e — см. tech-spec, risk "Relayer wallet exhausts USDC").

**Edge cases:**
- Arc Testnet RPC временно лежит → retry через fallback RPC (`https://5042002.rpc.thirdweb.com`, см. `deployment.md`). Если оба fail — FAIL задачи с пометкой "infrastructure", не "deploy regression".
- `npm install @universal-paywall/middleware@0.1.0-alpha.0` тянет latest вместо alpha → явно указывать точную версию (без `^`, `~`, диапазонов).
- Live e2e mid-test упирается в relayer USDC balance → классифицируется как FAIL по `settlement_failed.reason = "relayer_no_balance"` (canonical error reason из systemic-fix §4), фиксируем в репорте, fix вне этой задачи.
- 402 body отдаёт correct поля но `payTo` — нулевой адрес (factory деплой/конфиг рассинхрон) → FAIL по on-chain step 1 раньше, чем дойдём до HTTP smoke.
- Расхождение `factory.owner()` с ожиданием — самый опасный случай (компрометация ключа / wrong deploy). Останавливаемся, не закрываем фичу, эскалируем.
- Scratch `package.json` без `"type": "module"` → `server.js` падает `SyntaxError: Cannot use import statement outside a module`. Шаг 1 явно патчит `type` сразу после `npm init -y`.
- Server не успел подняться к моменту первого curl → readiness loop (`for i in {1..20}; sleep 0.2`) даёт до 4 сек на boot. Если за 20 итераций `/health` не отвечает — FAIL и repport "server failed to start".

**Implementation hints:**
- Использовать **программный viem** (`createPublicClient`, `readContract`) — не MCP Blockscout (как явно указано в tech-spec и в инструкциях). Все on-chain reads делаются inline через `node -e` (или через временный файл, который удаляется сразу после выполнения).
- ABI factory — минимальный fragment с четырьмя view-функциями (`feeBps`, `platformTreasury`, `owner`, `paused`), inline через `parseAbi([...])`. Не тащить весь artifact.
- Для ajv-валидации x402 wire format использовать **vendored fixture от Task 9**: `packages/middleware/src/__tests__/fixtures/x402-v1.schema.json` (systemic-fix §1: T9 owns the x402 schema fixture). Не пытаться сгенерировать schema из Zod-схем T7 — fixture это и есть canonical wire-format truth.
- Server в HTTP smoke — минимальная обёртка вокруг `withPaywall`: добавить `/health` endpoint для readiness loop. Без `/health` обходимся `curl -sf http://localhost:$PORT/` и игнорируем 402 как успех (200 от health чище).
- Все три шага идемпотентны и read-only по отношению к state (e2e шлёт реальную транзакцию, но это контролируемый платёж с relayer-а — это by design).
- Сохранять raw stdout каждого шага в репорт — для аудита через 6 месяцев.
- Канонические env-vars из systemic-fix §3: `ARC_RPC_URL`, `PAYWALL_RELAYER_KEY`, `PLATFORM_TREASURY_ADDRESS`, `PAYMENT_SPLITTER_FACTORY_ADDRESS`. Никаких `ARC_TESTNET_RPC_URL` / `ARC_TESTNET_PRIVATE_KEY`.
