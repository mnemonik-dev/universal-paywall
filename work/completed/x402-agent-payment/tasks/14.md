## Description

Holistic full-feature test quality audit across the entire x402-agent-payment feature. This is part of the Audit Wave (Wave 10), running in parallel with Code Audit (Task 12) and Security Audit (Task 13). The auditor IS the review — there are no downstream reviewers for this task.

The audit verifies:
1. **Coverage targets are met** — middleware ≥85% line coverage, contracts ≥95% branch coverage (Foundry LCOV report on `contracts/src/`, excluding `test/`, `script/`, `lib/`).
2. **Presence of every test the tech-spec mandates** — every bullet in the Testing Strategy section is covered by an actual, executable test, including forge fuzz and invariant tests.
3. **Test quality** — assertions are meaningful (not just truthy), tests are isolated, mocks are realistic, no sleep-based flakiness, no over-mocking that hides real bugs.

Output: `work/x402-agent-payment/audit-tests.md` — structured report listing what was checked, what passed, what is missing or weak, and concrete recommendations. The report serves as the test-quality gate before Pre-deploy QA (Task 15).

**Edge cases:**
- Coverage tool reports per-file numbers — if the overall number meets the threshold but a critical file (e.g. `settle.ts`, `verify.ts`) is under-covered, flag it. Aggregate numbers can hide weak spots.
- A test may exist by name but assert nothing meaningful (e.g. `expect(result).toBeDefined()`). Read the body of each mandatory test, not just the file listing.
- A test may pass for the wrong reason (over-mocking — the mock is the answer). Spot-check mocks against the real source module behavior.
- Forge LCOV branch coverage on Solidity counts each `require` / `if` branch — missing a single revert branch can fail the 95% bar. Identify exact branches if shortfall.
- The forked-e2e test runs in one process by design (cross-adapter replay test depends on this). If the test file launches a subprocess instead, the cross-adapter invariant is not actually proven — flag it.
- `forge coverage` output includes test contracts by default. Compute the threshold against `contracts/src/` ONLY. Do not let high coverage on `test/` artifacts inflate the production-code number.
- `VaultInvariants.t.sol` may compile but contain only one invariant function; the count check is on `function invariant_*` declarations inside the file — verify by reading the file, not by relying on compile success.

**Implementation hints:**
- For middleware coverage reports: vitest produces `coverage/coverage-summary.json` and `coverage/index.html`.
- For contract coverage: `forge coverage --report lcov` produces `lcov.info` at `contracts/lcov.info`. Use `lcov --summary lcov.info` for totals, or `grep -E '^(SF|BRDA|BRF|BRH):' lcov.info` to extract branch data per source file. Filter to `SF:contracts/src/` blocks for the threshold.
- Quick per-file branch ratio (when `lcov` CLI absent): `awk` over `lcov.info` blocks summing `BRH:` (hit) over `BRF:` (found) for files whose `SF:` line matches `contracts/src/`.
- For counting fuzz/invariant tests: `grep -E '^\s*function (testFuzz|invariant)_' contracts/test/**/*.sol` and read each declaration to confirm it is a real test (not a helper).
- For the test quality audit, prefer reading test files directly over running them — the audit is about content, not green/red.
- If a tech-spec requirement is present but split awkwardly across files, that is acceptable (mark PRESENT with a note). The audit cares about whether the property is verified, not stylistic placement.
