# Task 7: Verify + Settle

## Required Skills

Перед выполнением задачи загрузи:

## Description

Implement the two facilitator-side modules that form the core of the x402 payment path: off-chain signature verification (`verify.ts`) and on-chain settlement (`settle.ts`). These modules turn a parsed `PaymentPayload` into either an authoritative "ok, settled on-chain" result or a precisely-classified failure that the core orchestrator can translate into the right 402 reason for the agent.

`verify.ts` is the security gate. It runs `viem.recoverTypedDataAddress` against the EIP-712 domain pulled from the `NETWORKS` registry (so the chainId and verifyingContract are bound to the configured network — defense against EIP-712 chain-replay per D14 + Risk row "EIP-712 chain-replay attack") and then walks every check from Solution step 7c: recovered signer equals `authorization.from`, `to` equals the expected vault address, `value >= maxAmountRequired` (BigInt compare), `validBefore > nowMs + 5_000ms` after converting the EIP-3009 `validBefore` (unix seconds) to milliseconds via `Number(validBefore) * 1000` (5s safety margin against block-time skew), `validAfter * 1000 <= nowMs`, the payload network normalizes to the same canonical form as the configured network, and finally the synchronous `NonceStore.has(...) → false` followed by `insert(...)` with no `await` between them (TOCTOU-safe per D5). Any failed check produces a typed reason string that maps directly to the canonical 402 reason catalog from systemic-fixes §4 (e.g. `invalid_signature`, `to_mismatch`, `insufficient_amount`).

`settle.ts` is the on-chain executor and is the **sole owner** of two cross-cutting concerns per systemic-fixes §5: (1) it constructs and caches the viem `WalletClient` because, per D13, it is the only module allowed to extract the raw key from the `OpaqueRelayerKey` (core.ts holds the wrapper opaque and passes it through unchanged); (2) it performs the D14 startup chainId pin because it issues the request's first `writeContract` call. `settle.ts` reuses the `PublicClient` lazy-created by `core.ts` (passed in via `opts.publicClient`) for receipt-await and balance reads. It calls `USDC.transferWithAuthorization(from, to, value, validAfter, validBefore, nonce, v, r, s)` directly on the configured USDC contract (no wrapper contract — D2). It awaits the receipt with a 30s timeout, and every conceivable failure path collapses into exactly one of the seven settlement reasons enumerated in D5: `rpc_timeout`, `rpc_5xx`, `gas_estimate_revert`, `mine_timeout`, `receipt_reverted`, `relayer_no_balance`, `authorization_already_used_onchain`. Of these, `relayer_no_balance` has two detection paths per the Risks table: a proactive `USDC.balanceOf(relayer) < 1_000_000n` (1 USDC) check before `writeContract`, and a reactive re-classification when `writeContract` fails on insufficient gas. Both surface as the `relayer_no_balance` reason in the return value — `settle.ts` does NOT import or call `SecurityLogger`; D18 `securityEvent` emission is wired in `core.ts` (Task 8) per systemic-fixes §2, which is the single owner of all security event emissions. This module only returns the typed reason.

Replay-store retention on settle failure: when `settle.ts` returns a non-ok result, the `NonceStore` entry inserted by `verify.ts` must NOT be deleted (Risks row "Settlement failure mid-flight" — the agent must mint a fresh nonce to retry). This is enforced structurally: `verify.ts` is the only module that touches the store; `settle.ts` never imports or calls any NonceStore API.

## What to do

1. Implement `packages/middleware/src/verify.ts` exporting `verifyEip3009Authorization(payload, opts) → Promise<{ ok: true, recoveredFrom } | { ok: false, reason }>`. The function takes exactly **2 arguments** per systemic-fixes-3 §3: `(payload, opts)`. `opts` contains `expectedVaultAddress`, `expectedNetwork`, `maxAmountRequired`, `publicClient`, `nonceStore`, and `nowMs` (injectable for tests; defaults to `Date.now()`). `nonceStore` is part of `opts`, NOT a third positional argument.
   - Build the EIP-712 domain from `NETWORKS[opts.expectedNetwork]` (name=`usdcEip712Name`, version=`usdcEip712Version`, chainId, verifyingContract=`usdcAddress`).
   - Run `viem.recoverTypedDataAddress` against the `TransferWithAuthorization` typed-data structure with the supplied `payload.payload.signature` and `payload.payload.authorization` message.
   - Run all Solution-7c checks in order, using **milliseconds throughout** per systemic-fixes-3 §8. The EIP-3009 `validBefore` and `validAfter` fields are unix seconds; convert to ms before comparing against `opts.nowMs`:
     ```ts
     const nowMs = opts.nowMs ?? Date.now();
     const validBeforeMs = Number(authorization.validBefore) * 1000;
     const validAfterMs = Number(authorization.validAfter) * 1000;
     const SAFETY_MARGIN_MS = 5_000;
     ```
     Checks: recovered == from; `to == opts.expectedVaultAddress`; `BigInt(value) >= opts.maxAmountRequired`; `validBeforeMs > nowMs + SAFETY_MARGIN_MS` (else `authorization_expired`); `validAfterMs <= nowMs` (else `authorization_not_yet_valid`); normalized `payload.network === opts.expectedNetwork` (use `NETWORKS` helpers to compare alias and CAIP-2 forms).
   - Synchronously call `opts.nonceStore.has({ from, nonce })` — if true, return `{ ok: false, reason: 'nonce_already_used' }`. Otherwise immediately call `opts.nonceStore.insert({ from, nonce, validBefore })` (no `await` in between). Return `{ ok: true, recoveredFrom }`.
   - Map each individual check failure to the canonical reason string from systemic-fixes §4: `invalid_signature` (recovered != from OR any domain tamper — chainId, verifyingContract, name, version), `to_mismatch` (NOT `recipient_mismatch`), `insufficient_amount` (NOT `value_too_low`), `authorization_expired`, `authorization_not_yet_valid`, `network_mismatch`, `nonce_already_used`. These are the wire-format codes consumed by `errors.ts` to build the 402 body.
   - **No `SecurityLogger` import.** Per systemic-fixes-3 §2, `verify.ts` does not call `logger.securityEvent(...)`. It returns a classified result; `core.ts` (Task 8) is the policy enforcement point that maps return values to security events.

2. Implement `packages/middleware/src/settle.ts` exporting `settleOnChain(payload, recoveredFrom, opts) → Promise<SettleResult>` where (per systemic-fixes-3 §6):
   ```ts
   type SettleResult =
     | { ok: true; txHash: `0x${string}`; payer: `0x${string}` }
     | { ok: false; reason: SettleReason; details?: { gasEstimate?: bigint; balance?: bigint } };

   type SettleReason =
     | 'rpc_timeout' | 'rpc_5xx' | 'gas_estimate_revert'
     | 'mine_timeout' | 'receipt_reverted'
     | 'relayer_no_balance' | 'authorization_already_used_onchain';
   ```
   - **WalletClient ownership (per systemic-fixes §5):** `settle.ts` is the sole owner of the viem `WalletClient`. Maintain a module-level `Map<network, { walletClient, chainIdPinned: boolean }>` cache. On first use of a network: extract the raw key from `opts.relayerKey` (an `OpaqueRelayerKey`) via the private symbol exported by `relayer-key.ts`, build a viem account, construct `walletClient = createWalletClient({ account, chain, transport: http(rpcUrl) })`, and cache. The raw key never escapes this module — no parameters, no return values, no error fields contain it.
   - **PublicClient consumption:** `opts.publicClient` is provided by `core.ts` (which owns the per-network `PublicClient` cache per systemic-fixes §5). Use it for `getChainId()`, `readContract` (USDC balance), and `waitForTransactionReceipt`. Do NOT create a `PublicClient` here.
   - **ChainId pin (per systemic-fixes §5, D14):** because `settle.ts` issues the first `writeContract` per request, it owns the chainId pin check. On first use of a network (`chainIdPinned === false`), call `opts.publicClient.getChainId()` and assert it equals `NETWORKS[opts.network].chainId`. Mismatch throws `NetworkMismatchError` with `expectedChainId` and `observedChainId` fields. On success, set `chainIdPinned = true` so subsequent requests on the same network skip the check.
   - **Proactive `relayer_no_balance` check (per systemic-fixes-3 §7):** read `USDC.balanceOf(relayerAddress)` (use the same USDC contract address as the transfer target). The threshold is the **constant** `MIN_RELAYER_USDC_BALANCE = 1_000_000n` (= 1 USDC at 6 decimals) — NOT `gasEstimate * 2`. If `relayerBalance < MIN_RELAYER_USDC_BALANCE`, return `{ ok: false, reason: 'relayer_no_balance', details: { balance: relayerBalance } }` without touching the chain. Pin this constant at module scope so tests reference the same value.
   - Call `walletClient.writeContract` with the USDC contract address and `transferWithAuthorization(from, to, value, validAfter, validBefore, nonce, v, r, s)` (split the 65-byte signature into v/r/s using viem's `parseSignature` or equivalent — normalize v to 27/28 per EIP-3009).
   - Call `opts.publicClient.waitForTransactionReceipt({ hash, timeout: 30_000 })`.
   - On `receipt.status === 'success'`, return `{ ok: true, txHash: hash, payer: recoveredFrom }`.
   - **Single classifier function** mapping each error to exactly one of the 7 reasons:
     - viem `TimeoutError` (or `RpcRequestError` wrapping a fetch network timeout) raised by the RPC transport during `writeContract` or `balanceOf` → `rpc_timeout`
     - viem `HttpRequestError` with `status >= 500` → `rpc_5xx`
     - viem `ContractFunctionExecutionError` raised during gas estimation (pre-broadcast) → `gas_estimate_revert`
     - viem `WaitForTransactionReceiptTimeoutError` thrown from `waitForTransactionReceipt` after the 30s timeout → `mine_timeout` (note: this is the receipt-mining timeout class, NOT the generic transport `TimeoutError`)
     - receipt returned but `receipt.status === 'reverted'` → `receipt_reverted`
     - **`authorization_already_used_onchain` detection via STRING revert matching only** (per systemic-fixes-3 §6): Circle's FiatTokenV2 reverts with a `require()` string `"FiatTokenV2: authorization is used or canceled"`. The classifier extracts the decoded revert reason from the viem error (e.g. `error.shortMessage`, `error.metaMessages`, or `error.cause.reason`) and performs a **case-insensitive substring match** on the literal `"authorization is used"`. There is NO 4-byte selector (the previously documented `0xee0e9b58` was hallucinated — Circle uses `require()`, not a custom error). If the revert reason cannot be decoded from the error, fall back to `receipt_reverted` — do not guess.
     - reactive insufficient-gas / insufficient-balance error during `writeContract` (viem `InsufficientFundsError` or revert message matching that pattern) → `relayer_no_balance`
   - Return `{ ok: false, reason, details? }` for any classified failure (details is optional and may carry `{ gasEstimate?, balance? }`). On settle failure the replay-store entry stays intact (this is enforced by the architectural choice that `verify.ts` already inserted and `settle.ts` does not call any nonce-store API — document this contract in a code comment).
   - **No `SecurityLogger` import.** Per systemic-fixes-3 §2, `settle.ts` does not call `logger.securityEvent(...)`. It returns a classified `SettleResult`; `core.ts` (Task 8) is the single owner of D18 event emission and maps reason strings to the appropriate event names per the D18 table.

3. Add `NetworkMismatchError` class (or reuse the one from `errors.ts` if Task 6 added it; otherwise define it locally and re-export from `errors.ts` once consolidated). The error must carry `expectedChainId` and `observedChainId` fields so the core orchestrator can emit the D18 `chain_id_mismatch` security event.

4. The per-network cache for `settle.ts` is `Map<network, { walletClient, chainIdPinned: boolean }>` — `PublicClient` is NOT in this cache (it's owned by core.ts). Eviction is unnecessary for MVP (network set is bounded by `NETWORKS`).

5. Export `verifyEip3009Authorization` from `verify.ts` and `settleOnChain` from `settle.ts` so they can be unit-tested in isolation (Task 9) and orchestrated by `core.ts` (Task 8). The settle.ts `opts` type must include `publicClient: PublicClient` and `relayerKey: OpaqueRelayerKey` (passed through opaquely from core).

**Edge cases:**
- Clock skew between agent and facilitator: handled by the `SAFETY_MARGIN_MS = 5_000` margin on `validBeforeMs` (per systemic-fixes-3 §8) and the strict `<=` on `validAfterMs`. All comparisons are in **milliseconds** after converting the EIP-3009 unix-seconds field.
- The relayer wallet's USDC balance dropping below `MIN_RELAYER_USDC_BALANCE` between the proactive check and `writeContract`: reactive classifier catches this and returns `relayer_no_balance`.
- viem's exception hierarchy is wide; ensure the classifier checks the most specific class first. In particular: `WaitForTransactionReceiptTimeoutError` (mine_timeout) is checked before the generic `TimeoutError` (rpc_timeout) — they are distinct viem classes and must NOT collapse into one branch.
- Undecoded revert: if viem cannot decode the revert reason from a `ContractFunctionExecutionError` (no `shortMessage`/`metaMessages`/`cause.reason` string), classifier MUST fall back to `receipt_reverted` rather than guessing `authorization_already_used_onchain`. Do not invent revert reasons.
- `parseSignature` of a 65-byte signature: confirm the `v` value normalization (some libraries return 0/1, EIP-3009 expects 27/28). Test with both formats from a known signer.
- Race between two requests with the same `(from, nonce)`: NonceStore guarantees synchronous has+insert in `verify.ts`, so only the first wins; second sees `nonce_already_used` from the store before any on-chain tx is attempted.
- Network ID input may arrive as alias `arc-testnet` or CAIP-2 `eip155:5042002`; always normalize through `networks.ts` helpers before lookup or comparison.
